# Java の歴史 -- 誕生から Java 25 までの 30 年

## 概要

このドキュメントは、Java という言語が「なぜ生まれ」「どのように進化し」「今どこにいるのか」を、1991 年の Green Project から 2025 年の Java 25 に至るまでの 30 年超の歴史として記述する。技術仕様の羅列ではなく、人物・企業・時代背景を交えた「読み物」として構成している。

> **対象読者:** Java を学んでいる開発者、OpenJDK の内部構造に関心がある方、「Java はなぜこうなっているのか」の文脈を知りたい方。

## 1. 前史: Java が生まれる前の世界

### 1980〜90 年代のコンピュータ業界

Java が誕生する背景を理解するには、1990 年前後のソフトウェア業界がどういう状況だったかを知る必要がある。

**C/C++ が支配する世界:**
1980 年代後半〜1990 年代前半、システムプログラミングの主力は C と C++ だった。この二つの言語は圧倒的なパフォーマンスとハードウェア直接制御能力を持つ一方、深刻な課題を抱えていた。

```
┌─────────────────────────────────────────────────────────┐
│              C/C++ の光と影 (1990年前後)                  │
├───────────────────────┬─────────────────────────────────┤
│       光 (強み)        │         影 (課題)                │
├───────────────────────┼─────────────────────────────────┤
│ ハードウェア直接制御   │ 手動メモリ管理 → メモリリーク    │
│ 高いパフォーマンス     │ ポインタ演算 → バッファオーバーフロー│
│ OS/ドライバ開発に必須  │ プラットフォームごとに再コンパイル│
│ 豊富なライブラリ資産   │ undefined behavior の地雷原     │
│ 業界標準の地位         │ ヘッダファイル地獄              │
└───────────────────────┴─────────────────────────────────┘
```

特に「メモリ管理」は深刻だった。`malloc`/`free` の対応漏れによるメモリリーク、解放済みメモリへのアクセス（dangling pointer）、バッファオーバーラン ── これらは当時のソフトウェアバグの大部分を占めていた。

**プラットフォームの分断:**
1990 年前後は、ハードウェアとOS の多様性が今以上に激しかった。

```
┌─────────────────── 1990年前後の主要プラットフォーム ───────────────────┐
│                                                                       │
│  ワークステーション        PC                  組み込み/家電           │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────────┐        │
│  │ Sun SPARC    │    │ IBM PC/AT    │    │ セットトップボックス│        │
│  │ HP PA-RISC   │    │ (Intel x86)  │    │ PDA              │        │
│  │ SGI MIPS     │    │ MS-DOS       │    │ 家庭用ゲーム機    │        │
│  │ DEC Alpha    │    │ Windows 3.1  │    │ 携帯電話 (初期)   │        │
│  │ IBM RS/6000  │    │              │    │                  │        │
│  └──────────────┘    └──────────────┘    └──────────────────┘        │
│         ↑                   ↑                      ↑                 │
│    各社独自の UNIX       DOS/Windows          独自 RTOS              │
│    (Solaris, HP-UX,     のみ                 (VxWorks 等)           │
│     IRIX, AIX...)                                                    │
│                                                                       │
│  → 一つのプログラムを動かすだけでも、対象ごとにコードを書き直す必要   │
└───────────────────────────────────────────────────────────────────────┘
```

ソフトウェアを一度書いたらどこでも動かせる ── これは当時の開発者にとって夢物語だった。

**インターネットの黎明期:**
1990 年に Tim Berners-Lee が World Wide Web を発明。1993 年に Mosaic ブラウザが登場すると、ウェブは爆発的に広がり始めた。しかし当時のウェブは静的な HTML を表示するだけで、ブラウザ上でプログラムを実行する手段はなかった。ここに巨大なビジネスチャンスが眠っていた。

## 2. Sun Microsystems の創業と文化

### 4人の創業者 (1982年)

Java を理解するには、その母体となった Sun Microsystems という会社を理解する必要がある。

1982 年 2 月、4 人の創業者が Sun Microsystems, Inc. を設立した。

```
┌─────────────── Sun Microsystems 創業 (1982年2月) ────────────────┐
│                                                                   │
│  ┌─────────────────┐    ┌─────────────────┐                      │
│  │ Vinod Khosla    │    │ Andy             │                      │
│  │ ビノド・コースラ │    │ Bechtolsheim    │                      │
│  │                 │    │ アンディ・        │                      │
│  │ スタンフォード   │    │ ベクトルシャイム  │                      │
│  │ MBA 卒          │    │                  │                      │
│  │ → ビジネス面を   │    │ スタンフォード    │                      │
│  │   統括          │    │ 院生             │                      │
│  │                 │    │ → SUN ワーク     │                      │
│  │                 │    │   ステーションの  │                      │
│  │                 │    │   設計者          │                      │
│  └─────────────────┘    └─────────────────┘                      │
│                                                                   │
│  ┌─────────────────┐    ┌─────────────────┐                      │
│  │ Scott McNealy   │    │ Bill Joy         │                      │
│  │ スコット・       │    │ ビル・ジョイ     │                      │
│  │ マクネリ        │    │                  │                      │
│  │                 │    │ UC バークレー    │                      │
│  │ スタンフォード   │    │ 出身             │                      │
│  │ MBA 卒          │    │ → BSD UNIX の    │                      │
│  │ → CEO として     │    │   開発者         │                      │
│  │   Sun を牽引    │    │ → vi エディタの  │                      │
│  │                 │    │   作者           │                      │
│  │                 │    │ → TCP/IP 実装に  │                      │
│  │                 │    │   貢献           │                      │
│  └─────────────────┘    └─────────────────┘                      │
│                                                                   │
│  "SUN" = Stanford University Network                              │
│   Andy の設計した SUN ワークステーションがそのまま社名に            │
└───────────────────────────────────────────────────────────────────┘
```

> **トリビア:** 社名 "SUN" は Stanford University Network の頭字語で、Andy Bechtolsheim がスタンフォード大学で設計したワークステーションに由来する。

### "The Network is the Computer"

Sun のビジョンを一言で表すスローガンが "The Network is the Computer"（ネットワークこそがコンピュータである）だった。1980 年代にこのビジョンを掲げていたのは先見の明と言える。コンピュータは孤立して動くものではなく、ネットワークに接続されて初めて本当の価値を発揮する ── この思想は、後に Java が「ネットワーク指向言語」として設計される土壌となった。

### 技術的成果

Sun はワークステーション企業として数々の技術革新を生み出した。

| 技術 | 説明 |
|---|---|
| **SPARC** | Sun が開発したオープンな RISC プロセッサアーキテクチャ。ワークステーション市場を席巻 |
| **SunOS / Solaris** | BSD 系から SVR4 系に移行した UNIX OS。高い信頼性と性能で知られた |
| **NFS** (Network File System) | Sun が開発し公開仕様として公開。ネットワーク越しのファイル共有を実現 |
| **RPC** (Remote Procedure Call) | NFS の基盤技術。後の分散コンピューティングの原型 |
| **NeWS** | ネットワーク拡張ウィンドウシステム（X Window System との競争に敗北） |

### Sun の企業文化

Sun は徹底した**技術者主導**の企業文化を持っていた。CEO のスコット・マクネリは技術者に大きな裁量を与え、「面白い技術プロジェクトには予算をつける」という風土があった。この文化が Green Project の誕生を可能にした。

> **マクネリ語録:** "Microsoft is a monopoly. We're a company that happens to make software."（Microsoft は独占企業だ。我々はたまたまソフトウェアを作っている会社だ）―― Sun は常に Microsoft を仮想敵として意識していた。

## 3. Green Project と Oak の誕生 (1991-1994)

### Patrick Naughton の退職メール (1990年12月)

Java の歴史は、一通の退職メールから始まった。

1990 年 12 月、Sun の若手エンジニア **Patrick Naughton**（パトリック・ノートン）は、会社に不満を感じて退職のメールを書いた。彼の不満は「Sun のソフトウェア開発ツールは質が低く、自社の NeWS や NeXT を見ても明らかなように、ユーザーインターフェースの未来を捉えられていない」というものだった。

このメールが上層部の目に留まった。CEO のスコット・マクネリは彼を引き留め、「Sun を辞める代わりに、好きなようにやってみろ。何を変えるべきか提案しろ」と持ちかけた。

Naughton は、**James Gosling**（ジェームズ・ゴスリング）と **Mike Sheridan**（マイク・シェリダン）を巻き込み、1991 年に **Green Project** を立ち上げた。

```
┌──────── Green Project の始まり (1991年) ────────┐
│                                                   │
│    Patrick Naughton                               │
│    「Sun のソフトは最悪だ、辞める！」              │
│            │                                      │
│            ▼                                      │
│    Scott McNealy (CEO)                            │
│    「辞めるな。好きにやれ」                        │
│            │                                      │
│            ▼                                      │
│    ┌─────────────────────────────┐                │
│    │      Green Project 始動     │                │
│    │                             │                │
│    │  James Gosling  (言語設計)  │                │
│    │  Patrick Naughton (UI/デモ) │                │
│    │  Mike Sheridan (ビジネス)   │                │
│    │  + 数名のエンジニア         │                │
│    └─────────────────────────────┘                │
│                                                   │
│    ミッション: "次世代の家電製品のための           │
│                コンピューティング基盤を作れ"       │
└───────────────────────────────────────────────────┘
```

### 「家電のための言語」という当初のビジョン

Green Project のミッションは**次世代のデジタル家電**だった。テレビ、電子レンジ、セットトップボックス ── 1990 年代初頭、「家電がデジタル化されネットワークにつながる」というビジョンは先進的だった（今で言う IoT の先駆け）。

しかし問題があった。家電製品のプロセッサは多種多様で（SPARC, MIPS, x86, ARM, 独自チップ...）、C/C++ でプログラムを書くと、ハードウェアごとにコードを書き直す必要がある。しかも家電製品は「ユーザーが電源を入れたら動く」信頼性が求められ、C/C++ のメモリ破壊バグは許容できない。

James Gosling は最初 C++ を改良して使おうとしたが、やがて「C++ を直すより、一から言語を作り直した方が早い」と判断した。

### James Gosling の言語設計

Gosling が設計した新言語 **Oak** は、C++ の問題を明確に意識して設計された。

```
┌─────────── Gosling が C++ から排除したもの ──────────────┐
│                                                          │
│  C++ の機能              →  Oak での対応                  │
│  ─────────────────────      ─────────────────────────    │
│  手動メモリ管理            →  ガベージコレクション (GC)    │
│  (malloc/free)                自動でメモリを回収          │
│                                                          │
│  ポインタ演算              →  参照のみ (ポインタ廃止)     │
│                                メモリ安全性を保証         │
│                                                          │
│  多重継承                  →  単一継承 + インターフェース │
│                                ダイヤモンド継承問題を回避 │
│                                                          │
│  演算子オーバーロード      →  廃止                        │
│                                コードの意味が明確に       │
│                                                          │
│  プリプロセッサ (#define)  →  廃止                        │
│                                マクロ地獄からの解放       │
│                                                          │
│  ヘッダファイル            →  廃止 (パッケージで管理)     │
│                                宣言と定義の分離が不要に   │
│                                                          │
│  プラットフォーム依存      →  バイトコード + VM            │
│  のネイティブコード           どこでも同じバイトコードが  │
│                                動作する                   │
│                                                          │
│  goto 文                  →  廃止 (ラベル付き break で代替)│
└──────────────────────────────────────────────────────────┘
```

> **設計思想:** Gosling はこう語っている ―― "I wanted a language where ordinary programmers could produce reliable code."（普通のプログラマが信頼性の高いコードを書ける言語が欲しかった）。C++ は「優秀なプログラマが注意深く書けば安全」だが、Java は「普通のプログラマが書いても安全」を目指した。

特にガベージコレクション（GC）の導入は画期的だった。当時、GC を搭載した言語は Lisp や Smalltalk など学術寄りの言語に限られ、「実用言語に GC は遅くて使えない」というのが通説だった。Gosling はこの通説に挑戦した。

> GC の詳細については [GC ディープダイブ](memory/gc-deep-dive.md) を参照。

### *7 (Star Seven) デバイス

Green Project の成果物が **\*7（Star Seven）** と呼ばれるハンドヘルドデバイスだった。タッチスクリーンを搭載し、アニメーションの「Duke」キャラクターが画面上でユーザーを案内する。家庭内のデジタル機器を操作するリモコン的なデバイスで、Oak 言語で書かれたプログラムが動いていた。

```
┌──────────────── *7 (Star Seven) デバイス ─────────────────┐
│                                                           │
│    ┌─────────────────────┐                                │
│    │  ╔═══════════════╗  │  ← タッチスクリーン           │
│    │  ║               ║  │                                │
│    │  ║    (Duke)      ║  │  ← アニメーションキャラ       │
│    │  ║   ╱    ╲       ║  │     (後に Java のマスコットに) │
│    │  ║  ╱      ╲      ║  │                                │
│    │  ║               ║  │                                │
│    │  ╚═══════════════╝  │                                │
│    │  [TV] [VCR] [HVAC]  │  ← 家電機器の操作ボタン       │
│    └─────────────────────┘                                │
│                                                           │
│    CPU: 独自プロセッサ (SPARC ではない)                    │
│    OS:  Green OS (独自開発)                                │
│    言語: Oak (後の Java)                                    │
│    VM:  バイトコード仮想マシン                              │
│                                                           │
│    → 社内デモでは好評だったが、製品化には至らず            │
└───────────────────────────────────────────────────────────┘
```

> **トリビア:** Duke は \*7 のために Joe Palrang がデザインしたキャラクターで、今日に至るまで Java の公式マスコットとして使われている。三角形の鼻と大きく振る手が特徴。

### Oak の名前の由来

"Oak" という名前は、**Gosling のオフィスの窓から見えたオークの木**に由来する。特に深い意味はなく、「目の前にあったから」という実にエンジニア的な命名理由だった。

### 家電市場での挫折

Green Project は FirstPerson, Inc. という子会社に発展し、デジタル家電市場への参入を試みた。最大のチャンスは **Time Warner のインタラクティブ TV セットトップボックス**の入札だった。しかし、この入札で Silicon Graphics (SGI) に敗北。

家電メーカーも「なぜ家電にそんな複雑なソフトウェアが必要なのか？」と冷淡だった。1994 年初頭、FirstPerson は解散に追い込まれた。

```
┌─────── Green Project / FirstPerson の軌跡 ───────┐
│                                                   │
│  1991     Green Project 始動                      │
│    │                                              │
│    ▼                                              │
│  1992     Oak 言語 / *7 デバイス 完成             │
│    │                                              │
│    ▼                                              │
│  1993     FirstPerson, Inc. 設立                  │
│    │      (子会社化して事業化を目指す)             │
│    │                                              │
│    ├──→  Time Warner 入札 → SGI に敗北 ✗          │
│    │                                              │
│    ├──→  3DO との提携交渉 → 不成立 ✗              │
│    │                                              │
│    ▼                                              │
│  1994初   FirstPerson 解散                        │
│    │      「家電市場はまだ早すぎた」               │
│    │                                              │
│    ▼                                              │
│  ここで Oak は死んでもおかしくなかった...          │
│  しかし、インターネットが運命を変える              │
└───────────────────────────────────────────────────┘
```

Oak (後の Java) は、ここで消えていてもおかしくなかった。しかし歴史は思わぬ方向に転がる。

## 4. Oak から Java へ -- インターネットとの出会い (1994-1995)

### Bill Joy の慧眼

1993〜1994 年、World Wide Web が爆発的に広がり始めた。NCSA Mosaic ブラウザの登場で、インターネットはもはや学術研究者だけのものではなくなった。

Sun の共同創業者 **Bill Joy**（ビル・ジョイ）は、この変化を見て重要な洞察を得た。

> "Oak が持つ特性 ── プラットフォーム非依存、セキュリティ意識、ネットワーク対応 ── はまさにインターネットのために設計されたようなものだ。"

家電市場で失敗した Oak の特性が、奇しくもインターネット時代に完璧にフィットしたのだ。

```
┌───────── Oak の特性とインターネットの要求の合致 ──────────┐
│                                                           │
│  Oak の設計目標             インターネットの要件           │
│  (家電向けに設計)           (1994年に顕在化)              │
│                                                           │
│  プラットフォーム非依存 ←→ 多様なクライアント環境        │
│  (どの CPU でも動く)       (Windows, Mac, UNIX...)        │
│                                                           │
│  コンパクトな実行環境   ←→ 低帯域ネットワークで配信      │
│  (家電の限られたリソース)  (バイトコードは小さい)         │
│                                                           │
│  セキュリティ重視       ←→ 信頼できないコードの実行       │
│  (家電は堅牢でなければ)    (ブラウザで他人のコードを実行) │
│                                                           │
│  ネットワーク指向       ←→ ネットワークこそが本質         │
│  (家電をネットに接続)      (Web はネットワーク上で動く)   │
│                                                           │
│  → 完璧な一致。家電用に設計した言語が、                   │
│    インターネット用言語として最適だった                    │
└───────────────────────────────────────────────────────────┘
```

### WebRunner (HotJava) ブラウザの開発

1994 年後半、Green Project の生き残りメンバーは方向転換し、Oak で動く**ウェブブラウザ**の開発に取りかかった。Patrick Naughton が中心となって開発したこのブラウザは、最初 **WebRunner** と呼ばれ、後に **HotJava** と改名された。

HotJava の革新性は、ウェブページの中に Oak（Java）プログラムを埋め込んで実行できることだった。これが後に **Java アプレット**と呼ばれる技術になる。静的な HTML しか表示できなかったウェブに、動的なインタラクションをもたらすという革命的なコンセプトだった。

### Oak から Java への改名

ここで一つ問題が発覚した。**"Oak" という名前は既に商標登録されていた**（Oak Technology というビデオカードメーカーが持っていた）。改名が必要になった。

命名会議のエピソードは諸説あるが、よく知られている話では ―― チームメンバーが Peet's Coffee & Tea（カリフォルニアのコーヒーチェーン）に集まってブレインストーミングを行い、候補リストの中から **"Java"** が選ばれた。Java はインドネシアのジャワ島に由来し、良質なコーヒー豆の産地として知られる。

> **トリビア:** 他の候補には "Silk"、"DNA"、"Lyric"、"Pepper" などがあったとされる。最終的に Java が選ばれた理由は「エネルギーを感じさせる、短くて覚えやすい、コーヒーは開発者のエネルギー源」だから、とも言われている。Java のソースファイルの先頭バイトがコーヒーカップのマジックナンバー `0xCAFEBABE` であるのも、このコーヒーとの関連から。

### 1995年5月23日 -- SunWorld での発表

1995 年 5 月 23 日、サンフランシスコで開催された SunWorld カンファレンスで、Sun は **Java 言語**と **HotJava ブラウザ**を正式に発表した。さらに重要なことに、**Netscape Communications**（当時のブラウザ市場のリーダー）が **Netscape Navigator 2.0 に Java を搭載する**と発表した。

これは決定的だった。Netscape Navigator は当時最も使われていたウェブブラウザであり、これに Java が搭載されるということは、世界中のインターネットユーザーのコンピュータで Java が動くということを意味した。

### Java アプレットが世界を席巻

Java アプレットによって、ウェブページに動くグラフ、インタラクティブなゲーム、リアルタイムの株価表示といった動的コンテンツを埋め込めるようになった。今では JavaScript や HTML5 Canvas で当たり前にできることだが、1995 年当時は魔法のように感じられた。

アプレットの仕組みは Java のアーキテクチャを象徴していた。

### Java 1.0 (1996年1月23日)

最初の正式リリース。わずか 212 クラスでスタートした（参考: Java 25 は数千クラス）。主な用途は Java アプレットで、ブラウザ上で動くインタラクティブなコンテンツが世界中のウェブサイトに溢れた。

> **トリビア:** Java 1.0 の JVM は純粋なインタプリタで、C/C++ の 20〜100 倍遅かった。「Java は遅い」という評判はこの時代に生まれ、長く尾を引いた。

### Java 1.1 (1997年2月19日)

内部クラス（Inner Classes）が追加され、イベントハンドラの記述が容易に。JDBC によりデータベース接続が標準化された。RMI（Remote Method Invocation）は Sun のネットワーク思想を反映した機能で、異なるマシン上のオブジェクトをあたかもローカルのように呼び出せる仕組みだった。

### Java 1.2 / "Java 2" (1998年12月8日)

Java の歴史で最初の大きな転換点。Sun はこのバージョンから "Java 2 Platform" にリブランドし、プラットフォームを3つのエディションに分割した。

```
┌──────── Java 2 Platform の 3 エディション体制 ────────┐
│                                                        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │
│  │    J2SE      │  │    J2EE      │  │    J2ME      │ │
│  │  Standard    │  │ Enterprise   │  │   Micro      │ │
│  │  Edition     │  │  Edition     │  │  Edition     │ │
│  │              │  │              │  │              │ │
│  │ デスクトップ │  │ サーバーサイド│  │ 携帯電話    │ │
│  │ アプリケーション│  │ エンタープライズ│  │ 組み込み    │ │
│  │              │  │ Servlet, JSP │  │ PDA         │ │
│  │              │  │ EJB          │  │              │ │
│  └──────────────┘  └──────────────┘  └──────────────┘ │
│                                                        │
│  → この 3 エディション体制は Java 5 まで続いた         │
│    (後に Java SE, Java EE, Java ME に改名)             │
└────────────────────────────────────────────────────────┘
```

**Collections Framework** は今日の Java プログラミングの基盤となるクラス群で、`ArrayList`, `HashMap`, `HashSet` といったおなじみのクラスが登場。`Vector` を使わなくて済むようになった（が、`Vector` は残り続けた）。

### Java 1.3 (2000年5月8日)

**HotSpot VM** がデフォルトの JVM として搭載された。これは Java の歴史における重要なマイルストーンだ。

HotSpot は「ホットスポット（頻繁に実行される箇所）」を検出し、その部分だけをネイティブコードにコンパイルする **JIT (Just-In-Time) コンパイラ**を搭載していた。これにより、Java のパフォーマンスは劇的に向上した。

> **名前の由来:** "HotSpot" は、プログラムの実行時に「ホットスポット（熱い場所）＝頻繁に実行されるコードパス」を検出して最適化するという VM の動作原理に由来する。

> HotSpot VM の詳細については [JVM 概要](jvm-overview.md) を参照。

### Java 1.4 (2002年2月6日)

JCP (Java Community Process) の下で初めてリリースされたバージョン。JCP は Java の仕様策定をオープンなプロセスで行う仕組みで、Java 1.4 は JSR 59 として標準化された。

NIO (New I/O) は従来のストリームベース I/O に加えて、チャネルとバッファを使ったノンブロッキング I/O を提供した。高性能ネットワークサーバーの構築が可能になった。

### Java 5.0 (2004年9月30日)

**Java の歴史上、最大の言語変革。** それまでの Java とは別の言語と言えるほどの大幅な変更が入った。

バージョン番号も「1.5」から「5.0」に変更された（マーケティング上の理由：大きな変更にふさわしいメジャーバージョン番号にしたかった）。

```

│                                                         │

│  │  Before:  for (int i = 0; i < a.length; i++) │       │

│                                                         │
│  java.util.concurrent (並行処理ライブラリ)              │
│  ┌──────────────────────────────────────────────┐       │
│  │  Doug Lea の設計                              │       │
│  │  ExecutorService, ConcurrentHashMap,          │       │
│  │  ReentrantLock, CountDownLatch, ...           │       │
```

> **トリビア:** Java の Generics は型消去（Type Erasure）方式を採用した。コンパイル時に型情報をチェックし、実行時には型パラメータ情報を削除する。これは既存のバイトコードとの互換性を保つための妥協だったが、`new T[]` ができない等の制限を生んだ。この制限を解消するのが、現在進行中の Project Valhalla だ。

## 6. Java 6-7: Sun の落日と Oracle の台頭 (2006-2011)

### Java 6 (2006年12月11日)

言語レベルの大きな変更はなく、パフォーマンスと管理性の改善が中心のリリース。Scripting API (JSR 223) により、JVM 上で JavaScript (Rhino) 等のスクリプト言語を動かせるようになった。JVM の監視・管理ツール（JMX, VisualVM の前身）が充実した。

### OpenJDK の誕生 (2006-2007年)

2006 年 11 月、Sun は Java をオープンソース化すると発表した。ライセンスは **GPLv2 + Classpath Exception**（GNUの条項にClasspath例外を加え、Javaアプリの開発者がGPLの制約を受けないようにしたもの）。

2007 年 5 月、**OpenJDK** プロジェクトが正式に始動。これにより、Java の参照実装のソースコードが誰でも閲覧・改変・再配布できるようになった。

> **歴史的文脈:** Sun は長らく「Java のソースは公開するがライセンスは独自」という立場を取っていた。完全なオープンソース化は、Linux コミュニティからの圧力と、Sun 自身のオープンソースへの傾倒（MySQL の買収など）が重なった結果だった。

> OpenJDK から派生したディストリビューションについては [JDK ディストリビューション概要](jdk-distributions-overview.md) を参照。

### Sun Microsystems の経営危機と Oracle による買収 (2009-2010年)

**Oracle が Java を手に入れた意味:**
Oracle は世界最大のエンタープライズソフトウェア企業で、データベース製品が主力だった。Java（特に Java EE）はエンタープライズ市場の基盤技術であり、Oracle はその所有権を獲得した。これは Java の未来にとって両刃の剣だった ―― Oracle は資金力と技術力で Java の開発を継続できるが、Sun のようなオープンな文化とは異なる企業文化を持っていた。

### James Gosling の退社

2010 年 4 月、Java の父 James Gosling は Oracle を退社した。彼はブログでこう書いた。

> "Just about anything I could say that would be accurate and honest would do more harm than good."
> （正直に言えることは何であれ、益よりも害をなすだろう）

多くを語らなかったが、Oracle の企業文化と合わなかったことは明白だった。Java コミュニティにとって象徴的な出来事だった。

### Oracle vs Google 訴訟

2010 年 8 月、Oracle は **Google を著作権侵害で訴えた**。争点は、Google が Android OS で Java API（Application Programming Interface）を使用したことだった。

```

│            │                                               │
```

この訴訟は Java コミュニティだけでなく、ソフトウェア業界全体に大きな影響を与えた。「API のインターフェース定義に著作権が及ぶのか」という根本的な問いは、相互運用性の根幹に関わる問題だった。

### Java 7 (2011年7月28日)

Java 7 は「産みの苦しみ」のリリースだった。当初の計画（Project Coin、Lambda 式、Jigsaw モジュールシステム等を全部載せる）は野心的すぎて、Oracle への移行の混乱も重なり、大幅に機能を削減した **"Plan B"** で出荷された。

```
┌───────── Java 7 で入ったもの / 入らなかったもの ─────────┐
│                                                           │
│  ★ 入ったもの (Project Coin: 小さな言語改善)              │
│  ┌─────────────────────────────────────────────────┐      │
│  │  try-with-resources   (AutoCloseable の自動クローズ)│   │
│  │  diamond operator     (new ArrayList<>())         │     │
│  │  switch での String   (case "hello":)             │     │
```

## 7. Java 8: モダン Java の始まり (2014)

**Java の歴史上、Java 5.0 に匹敵する ―― いや、それ以上の大転換。**

Java 8 は「関数型プログラミング」の要素を本格的に取り入れた初のバージョンで、Java のコーディングスタイルを根本的に変えた。Java 8 が「モダン Java」の始まりとされる所以はここにある。

### なぜ Java 8 は「長期標準」になったか

Java 8 は LTS (Long-Term Support) として長期サポートが提供され、しかもその機能追加が劇的だったため、多くの企業が Java 8 で止まった。2024 年時点でも一部の企業が Java 8 を使い続けているという現実がある。

理由は複数ある。
- Java 8 で十分に「モダン」で、多くのユースケースをカバーできる
- Java 9 の Module System (JPMS) が大きな破壊的変更を含み、移行コストが高い
- 企業のリスク回避志向（「動いているものは触るな」）
- 2018 年の Oracle JDK ライセンス変更による混乱

### Oracle JDK のライセンス変更問題

2018 年、Oracle は JDK のライセンスポリシーを変更し、**Oracle JDK を本番環境で使用するには有償サブスクリプションが必要**とした（開発目的は無料）。これはJavaコミュニティに大きな動揺をもたらした。

しかし、OpenJDK 自体は引き続き無償で利用可能であり、この変更を契機に**代替ディストリビューション**が急速に台頭した。

> ディストリビューションの詳細な比較については [JDK ディストリビューション概要](jdk-distributions-overview.md) を参照。

### Java 10 (2018年3月20日)

新リリースモデルでの最初のバージョン。目玉は **var（ローカル変数型推論、LVTI）**。

```java
// Before
HashMap<String, List<Integer>> map = new HashMap<String, List<Integer>>();

// After (Java 10+)
var map = new HashMap<String, List<Integer>>();
```

> **設計の背景:** Brian Goetz は `var` の導入にあたり、「型推論は読者の負担を減らすための道具であり、書き手の怠惰のための道具ではない」と強調した。右辺から型が明白な場合にのみ使うべき、というガイドラインを示した。

### Java 11 (2018年9月25日)

**初の LTS リリース。** 多くの企業が Java 8 からの移行先として Java 11 を選んだ。

主な機能:
- **HTTP Client API** (java.net.http): HTTP/2 対応の標準 HTTP クライアント。
- **String 新メソッド**: `isBlank()`, `lines()`, `strip()`, `repeat()` 等。
- **単一ファイルソースプログラム**: `java Hello.java` で直接実行可能に（コンパイル不要）。
- **Flight Recorder**: JVM の診断情報を低オーバーヘッドで記録する機能がオープンソース化。

## 11. キーパーソンたち

Java の歴史は、優れたエンジニアたちの情熱と決断の歴史でもある。

### James Gosling -- Java の父

カナダ出身のソフトウェアエンジニア。カーネギーメロン大学で博士号を取得。Sun Microsystems で Emacs のクローン "Gosling Emacs" を開発した後、Green Project で Oak（Java）を設計。Java の文法、バイトコード仕様、最初のコンパイラと VM を自ら実装した。

2010 年の Oracle 退社後は、Liquid Robotics（海洋ロボット）、Amazon Web Services を経て、現在は半引退状態。

> "I've always been interested in tools. Java was a tool that grew into a phenomenon."
> （私はいつもツールに興味があった。Java は現象に成長したツールだった）

### Patrick Naughton -- Green Project の発起人

退職メールが Green Project のきっかけとなった。\*7 デバイスの UI や HotJava ブラウザを開発。Java の初期のインターネット戦略を推進した中心人物。

### Bill Joy -- Sun 共同創業者、インターネット Java へのピボット

UC バークレーで BSD UNIX と TCP/IP 実装に貢献し、vi エディタを作成した伝説的プログラマ。Sun の共同創業者であり Chief Scientist。家電市場で行き詰まった Oak をインターネット向けにピボットすべきだと進言した慧眼の持ち主。Java の初期仕様にも貢献した。

> **逸話:** Bill Joy は「世界で最も優秀なプログラマの一人」として知られ、BSD UNIX のネットワークスタック実装をほぼ一人で行ったとされる。

### Scott McNealy -- Sun CEO

Sun Microsystems の共同創業者であり長年の CEO。技術者に大きな裁量を与える経営スタイルで、Green Project のような探索的プロジェクトを許容する文化を作った。"The Network is the Computer" のビジョンを掲げ、Microsoft と正面から対決した。

### Guy Steele -- Java 言語仕様

Lisp の権威であり、プログラミング言語設計の大家。Java の言語仕様書 (JLS: Java Language Specification) の共著者。Gosling の設計した言語を厳密な形式仕様として文書化した。Scheme (Lisp の方言) の共同設計者でもある。

### Mark Reinhold -- Java Platform Chief Architect (現在)

Oracle の Java Platform Group Chief Architect。NIO、Generics の導入、Module System (Project Jigsaw)、6ヶ月リリースサイクルへの移行など、Java の重要な技術的決定に関わってきた。現在の Java プラットフォームの方向性を決める最も影響力のある人物の一人。

### Brian Goetz -- Java Language Architect (現在)

Oracle の Java Language Architect。『Java Concurrency in Practice』の著者として並行プログラミングの権威。Java 8 の Lambda 式/Stream API、Records、Sealed Classes、Pattern Matching など、モダン Java の言語機能の設計をリード。Project Amber（言語進化）の推進者。

> **設計哲学:** "The primary goal of Project Amber is to reduce the ceremony — the accidental complexity — of Java programming."
> （Project Amber の主要な目標は、Java プログラミングの儀式 ―― 偶有的な複雑さ ―― を削減することだ）

### Ron Pressler -- Project Loom リード

Oracle のエンジニアで、**Project Loom** (Virtual Threads, Structured Concurrency, Scoped Values) のリード。Java の並行処理モデルを根本から再設計し、"thread-per-request" スタイルを 100 万スレッド規模でスケールさせることを実現した。

> Virtual Threads の内部実装と Continuation メカニズムについては [Virtual Threads ディープダイブ](runtime/virtual-threads-deep-dive.md) を参照。

### John Rose -- Project Valhalla, Panama リード

Oracle のエンジニアで、JVM の低レベル機構の設計者。`invokedynamic` (JSR 292) の設計者であり、Java 7 で導入されたこの仕組みは Java 8 の Lambda 式の基盤となった。Project Valhalla（Value Types）と Project Panama（FFM API / Vector API）をリード。

## 13. Java が与えた影響

### C# への直接的影響

2000 年、Microsoft は .NET Framework と **C#** を発表した。C# は Java の直接的影響を受けて設計された言語で、Anders Hejlsberg（Turbo Pascal / Delphi の設計者）がリード。

背景には、Microsoft と Sun の間で争われた "Microsoft Java" 問題がある。Microsoft は独自の Java 実装 (MSJVM) に非標準の拡張を加え、Sun が訴訟を起こした。この和解の結果、Microsoft は独自の Java を作る道を断たれ、代わりに C# と .NET を一から構築した。

C# は Java の良い部分を取り入れつつ、Java が避けた機能（演算子オーバーロード、プロパティ、値型等）を積極的に導入。両言語はその後も互いに影響を与え合いながら進化を続けている。

### JVM 言語の誕生

Java Virtual Machine（JVM）はJava言語と独立して進化し、多数の言語のプラットフォームとなった。

```
┌───────── JVM 上で動く言語たち ─────────────────────────────────┐
│                                                                 │
│  Kotlin (2011〜, JetBrains)                                     │
│    → Android の公式言語 (2019〜)                                │
│    → Java との完全な相互運用性                                  │
│    → Null 安全性、拡張関数、コルーチン                          │
│                                                                 │
│  Scala (2004〜, Martin Odersky)                                 │
│    → 関数型 + オブジェクト指向のハイブリッド                    │
│    → Spark, Akka, Play Framework                                │
│    → Java の Generics 設計に不満を持った Odersky が設計          │
│                                                                 │
│  Groovy (2003〜, James Strachan)                                │
│    → 動的型付けの JVM 言語                                     │
│    → Gradle ビルドツール, Jenkins Pipeline                      │
│                                                                 │
│  Clojure (2007〜, Rich Hickey)                                  │
│    → JVM 上の Lisp 方言                                        │
│    → 不変データ構造、STM (Software Transactional Memory)        │
│                                                                 │
│  → JVM という「実行基盤」が多様な言語のエコシステムを支えた    │
└─────────────────────────────────────────────────────────────────┘
```

### Android 開発での役割

Google は 2008 年にリリースした Android OS の開発言語として Java を採用した（前述の Oracle vs Google 訴訟の発端）。これにより Java は世界最大のモバイルプラットフォームの基盤言語となり、何百万もの開発者が Java を学ぶきっかけとなった。

2019 年に Kotlin が Android の推奨言語になった後も、Android の基盤 API は Java で書かれており、Java の影響は依然として大きい。

### エンタープライズでの支配的地位

Java は誕生当初からエンタープライズ市場での存在感が大きかったが、特に Java EE（現 Jakarta EE）と Spring Framework の登場により、その地位は決定的になった。

- **Spring Framework (2003〜)**: Rod Johnson が設計。EJB の複雑さへのアンチテーゼとして、DI (Dependency Injection) と AOP (Aspect-Oriented Programming) を中心に据えた軽量フレームワーク。現在も企業向け Java アプリケーションの事実上の標準。
- **Spring Boot (2014〜)**: Spring をさらに簡素化し、"Convention over Configuration" を徹底。マイクロサービスアーキテクチャの普及に大きく貢献。

### 「ガベージコレクション付き言語」のメインストリーム化

Java 以前、ガベージコレクションは Lisp や Smalltalk など学術寄りの言語の特徴であり、「実用言語には遅すぎる」とされていた。Java の成功は、GC がメインストリームの実用言語で十分に機能することを証明した。

この影響は計り知れない。Java 以降に登場した主要言語のほぼ全て ―― C#, Python (CPython の参照カウント + cyclic GC), Go, Kotlin, Swift (ARC), Rust (所有権モデルという別アプローチだが手動 free は不要) ―― が何らかの形で自動メモリ管理を採用している。

> Java の GC アルゴリズムの詳細については [GC ディープダイブ](memory/gc-deep-dive.md) を参照。
