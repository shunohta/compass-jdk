## 1. バイトコードの基礎 -- JVM が実行する「仮想的な機械語」

**対象層:** Java 層 / HotSpot C++ 層

実行エンジンを理解するには、まずその入力である **バイトコード** を理解する必要がある。

### 1.1 バイトコードとは何か

バイトコードは、`javac` が Java ソースコードをコンパイルして生成する **中間表現** である。CPU が直接実行する機械語（x86, ARM 等）とは異なり、JVM という仮想マシン上で実行される。

```
 ┌─────────────────────────────────────────────────────────────┐
 │                    バイトコードの位置づけ                     │
 │                                                             │
 │  Java ソースコード          ← 人間が書く                    │
 │       │                                                     │
 │       ▼ javac                                               │
 │  バイトコード (.class)      ← JVM が実行する（1命令 = 1バイト）│
 │       │                                                     │
 │       ├──→ インタプリタ     ← 1命令ずつ解釈して実行          │
 │       │                                                     │
 │       └──→ JIT コンパイラ   ← 機械語に変換して高速実行       │
 │                 │                                           │
 │                 ▼                                           │
 │            機械語           ← CPU が直接実行する             │
 └─────────────────────────────────────────────────────────────┘
```

「バイトコード」という名前は、各命令のオペコード（操作コード）が **1バイト (0x00 〜 0xFF)** で表現されることに由来する。1バイトで最大256種類の命令を表現でき、JVM 仕様では約200種類のバイトコード命令が定義されている。

### 1.2 スタックマシン -- JVM の計算モデル

**これは実行エンジンを理解する上で最も基本的な概念である。**

JVM はスタックマシンである。すべての計算は **オペランドスタック** を使って行われる。レジスタマシン（x86 CPU 等）では演算結果をレジスタ（`eax`, `rbx` 等）に格納するが、JVM ではスタックに積む（push）、スタックから取り出す（pop）を繰り返して計算を進める。

```
 Java コード: int c = a + b;  (a=3, b=5 の場合)

 ┌─ オペランドスタックの変化 ──────────────────────────────────┐
 │                                                              │
 │  命令           スタック (← 上がトップ)     説明              │
 │  ─────────────  ──────────────────────      ──────────────── │
 │  iload_1        [3]                         a(=3)をスタックへ │
 │  iload_2        [3, 5]                      b(=5)をスタックへ │
 │  iadd           [8]                         3+5=8 を計算      │
 │  istore_3       []                          結果をローカル変数 c へ│
 │                                                              │
 └──────────────────────────────────────────────────────────────┘
```

**なぜスタックマシンなのか:**

| 特徴 | スタックマシン (JVM) | レジスタマシン (CPU) |
|------|---------------------|---------------------|
| 命令のサイズ | 小さい（レジスタ番号が不要） | 大きい（レジスタ番号を指定） |
| 移植性 | 高い（レジスタ数に依存しない） | 低い（CPU 固有のレジスタ） |
| 実装の容易さ | 容易 | 複雑 |
| 実行速度 | 遅い（メモリアクセス多い） | 速い（レジスタは高速） |

JVM がスタックマシンを選んだ最大の理由は **移植性**。どの CPU にも同じバイトコードで動作する。実行速度の差は JIT コンパイラが機械語に変換することで解消する。

### 1.3 バイトコード命令の分類

約200種類のバイトコード命令は、大きく以下のカテゴリに分類される。

```
 ┌─────────────────────────────────────────────────────────────┐
 │              バイトコード命令のカテゴリ                       │
 │                                                             │
 │  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐  │
 │  │ ロード/ストア │  │ 算術演算     │  │ 型変換           │  │
 │  │ iload, istore│  │ iadd, isub   │  │ i2l, i2f, i2d    │  │
 │  │ aload, astore│  │ imul, idiv   │  │ l2i, f2i, d2i    │  │
 │  │ fload, dload │  │ fadd, dmul   │  │                  │  │
 │  └──────────────┘  └──────────────┘  └──────────────────┘  │
 │                                                             │
 │  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐  │
 │  │ オブジェクト │  │ スタック操作 │  │ 制御フロー       │  │
 │  │ new          │  │ dup, pop     │  │ if_icmpeq        │  │
 │  │ getfield     │  │ swap         │  │ goto             │  │
 │  │ putfield     │  │              │  │ tableswitch      │  │
 │  │ checkcast    │  │              │  │ lookupswitch     │  │
 │  └──────────────┘  └──────────────┘  └──────────────────┘  │
 │                                                             │
 │  ┌──────────────────────────────────────────────────────┐  │
 │  │ メソッド呼び出し (invoke 系)                          │  │
 │  │                                                      │  │
 │  │ invokevirtual   通常のメソッド呼び出し（仮想ディスパッチ）│
 │  │ invokeinterface インタフェースメソッドの呼び出し       │  │
 │  │ invokespecial   コンストラクタ, private, super メソッド│  │
 │  │ invokestatic    static メソッドの呼び出し             │  │
 │  │ invokedynamic   ラムダ式, 文字列結合 等               │  │
 │  └──────────────────────────────────────────────────────┘  │
 └─────────────────────────────────────────────────────────────┘
```

**invoke 命令の違い（重要）:**

| 命令 | ディスパッチ | 対象 | 例 |
|------|------------|------|-----|
| `invokevirtual` | 動的（vtable） | 通常のインスタンスメソッド | `obj.toString()` |
| `invokeinterface` | 動的（itable） | インタフェースメソッド | `list.size()` |
| `invokespecial` | 静的 | コンストラクタ, private, super | `super.clone()` |
| `invokestatic` | 静的 | static メソッド | `Math.max(a, b)` |
| `invokedynamic` | ブートストラップ | ラムダ式, 文字列結合等 | `() -> "hello"` |

`invokevirtual` と `invokeinterface` はオブジェクトの実行時の型に基づいてメソッドを選択する（動的ディスパッチ）。JIT コンパイラはこの動的ディスパッチを **投機的に静的ディスパッチに最適化** することで、大幅な性能向上を実現する（[5.5節](#55-投機的最適化とガード) で詳述）。

### 1.4 javap で覗くバイトコードの世界

`javap` は .class ファイルのバイトコードを人間が読める形式で表示するツールである。

```bash
# Java ソースをコンパイルしてバイトコードを表示
javac Hello.java
javap -c Hello.class
```

**例: 簡単な計算のバイトコード**

```java
// Java ソースコード
int add(int a, int b) {
    return a + b;
}
```

```
// javap -c で表示されるバイトコード
int add(int, int);
  Code:
     0: iload_1        // ローカル変数 1 (a) をスタックに積む
     1: iload_2        // ローカル変数 2 (b) をスタックに積む
     2: iadd           // スタックから2つ取り出し、足して結果を積む
     3: ireturn        // スタックトップの int を返す
```

**例: オブジェクト生成のバイトコード**

```java
// Java ソースコード
String s = new String("hello");
```

```
// バイトコード
  0: new           #2    // class java/lang/String  ← ヒープに領域確保
  3: dup                 // 参照をスタック上で複製（コンストラクタ用）
  4: ldc           #3    // String "hello" を定数プールから取得
  6: invokespecial #4    // String.<init>(String) を呼び出し
  9: astore_1            // 結果をローカル変数 1 に格納
```

`new` 命令はメモリ確保のみを行い、初期化しない。`dup` で参照を複製してから `invokespecial` でコンストラクタを呼ぶ。コンストラクタは戻り値を返さないため、`dup` しておかないと参照が失われる。

- 関連ソースパス: `src/hotspot/share/interpreter/bytecodes.hpp`（バイトコード定義）

## 2. インタプリタ -- バイトコードを1命令ずつ実行する

**対象層:** HotSpot C++ 層（`src/hotspot/share/interpreter/`）

### 2.1 インタプリタの動作原理 -- ディスパッチループ

インタプリタの核心は **ディスパッチループ** と呼ばれる無限ループである。バイトコードを1命令ずつ読み取り、対応する処理を実行し、次の命令に進む。

```
 ┌─ インタプリタのディスパッチループ ────────────────────────┐
 │                                                           │
 │  while (true) {                                           │
 │      byte opcode = *bcp;    // 現在のバイトコードを読む    │
 │                              // (bcp = Bytecode Pointer)  │
 │                                                           │
 │      switch (opcode) {       // オペコードに応じて分岐     │
 │          case iload_0:                                    │
 │              push(locals[0]); break;                      │
 │          case iadd:                                       │
 │              int b = pop(); int a = pop();                │
 │              push(a + b); break;                          │
 │          case invokevirtual:                              │
 │              // vtable から実メソッドを解決して呼び出す    │
 │              ...                                          │
 │          // ... 約200種類の case                           │
 │      }                                                    │
 │                                                           │
 │      bcp++;  // 次の命令に進む（命令によって進む量が異なる）│
 │  }                                                        │
 │                                                           │
 └───────────────────────────────────────────────────────────┘
```

これが概念的な動作だが、実際の HotSpot ではもっと高速な仕組みが使われている。

### 2.2 Template Interpreter -- HotSpot の高速インタプリタ

HotSpot VM のインタプリタは **Template Interpreter** と呼ばれ、単純な switch 文ではなく、**各バイトコード命令に対応する機械語のテンプレートを JVM 起動時に生成** する。

```
 ┌─────────────────────────────────────────────────────────────┐
 │           単純なインタプリタ vs Template Interpreter          │
 │                                                             │
 │  【単純なインタプリタ（C で実装）】                           │
 │  ┌────────────┐   ┌────────┐   ┌────────────┐              │
 │  │ opcode 読取│──▶│ switch │──▶│  C 関数    │              │
 │  │            │   │  分岐  │   │  を呼出し  │              │
 │  └────────────┘   └────────┘   └────────────┘              │
 │  → switch のオーバーヘッド（分岐予測失敗、関数呼び出し）     │
 │                                                             │
 │  【Template Interpreter（機械語テンプレート）】               │
 │  ┌────────────┐   ┌──────────────────────┐                  │
 │  │ opcode 読取│──▶│ 対応する機械語を      │                  │
 │  │            │   │ 直接実行              │                  │
 │  └────────────┘   └──────────────────────┘                  │
 │  → 各命令が機械語の小さなコード片に直接マッピング            │
 │  → switch なし、関数呼び出しなし → 高速                     │
 └─────────────────────────────────────────────────────────────┘
```

**テンプレート生成の流れ:**

```
 JVM 起動時
     │
     ▼
 TemplateInterpreterGenerator::generate_all()
     │
     ├─ iload_0 用のテンプレート（機械語）を生成
     ├─ iadd 用のテンプレート（機械語）を生成
     ├─ invokevirtual 用のテンプレート（機械語）を生成
     :   （約200命令分すべて生成）
     │
     ▼
 テンプレートテーブルに登録
 （各オペコード → テンプレートのアドレス のマッピング）
     │
     ▼
 実行時: opcode をインデックスとしてテーブルから
         テンプレートのアドレスを引き、直接ジャンプ
```

**テンプレートの例（x86-64 での iload_0）:**

```asm
; iload_0: ローカル変数 0 をオペランドスタックに積む
; (概略的な疑似コード)
mov    eax, [r14]        ; r14 = ローカル変数配列の先頭, eax にロード
push   eax               ; オペランドスタックに push
movzbl ebx, [r13 + 1]    ; 次のバイトコードを先読み (r13 = bcp)
inc    r13                ; bcp を次の命令に進める
jmp    [dispatch_table + rbx * 8]  ; 次の命令のテンプレートに直接ジャンプ
```

最後の `jmp` が重要で、次のバイトコードのテンプレートに**直接ジャンプ**する。これにより、ループの先頭に戻って opcode を再度読む必要がなくなる。この技法を **threaded dispatch** と呼ぶ。

- 関連ソースパス: `src/hotspot/share/interpreter/templateInterpreter.cpp`
- 関連ソースパス: `src/hotspot/share/interpreter/templateInterpreterGenerator.cpp`
- 関連ソースパス: `src/hotspot/cpu/x86/templateTable_x86.cpp`（x86 固有のテンプレート）

### 2.3 インタプリタが果たす3つの役割

インタプリタは単なる「遅い実行モード」ではなく、JVM 全体の動作に不可欠な3つの役割を果たしている。

```
 ┌─────────────────────────────────────────────────────────────┐
 │              インタプリタの3つの役割                          │
 │                                                             │
 │  ① 起動時の即時実行                                        │
 │     JIT コンパイルには時間がかかる（数ms〜数百ms）。          │
 │     インタプリタがあるからこそ、コンパイルを待たずに          │
 │     すぐプログラムを実行開始できる。                         │
 │                                                             │
 │  ② プロファイリング情報の収集                               │
 │     メソッドの呼び出し回数、分岐の傾向、型情報を収集し、     │
 │     JIT コンパイラに「どこをどう最適化すべきか」を伝える。   │
 │                                                             │
 │  ③ 脱最適化のフォールバック                                 │
 │     JIT が投機的な最適化に失敗した場合、そのメソッドを        │
 │     インタプリタモードに「巻き戻す」。インタプリタは          │
 │     常に正しく動作する安全な実行環境として機能する。         │
 │                                                             │
 └─────────────────────────────────────────────────────────────┘
```

特に**③の脱最適化のフォールバック**は重要。JIT コンパイラが「この型は常に String だ」と仮定して最適化したのに、突然 Integer が来た場合、JIT コードは安全に実行できない。このとき JVM はインタプリタに戻って安全に実行を続行する（[第6章](#6-脱最適化-deoptimization----jit-のやり直し) で詳述）。

### 2.4 プロファイリング -- JIT コンパイルへの架け橋

**対象層:** HotSpot C++ 層（`src/hotspot/share/oops/methodData.hpp`）

インタプリタは実行しながら、以下のプロファイリング情報を `MethodData` オブジェクトに記録する。

```
 ┌─ MethodData（メソッドごとのプロファイル）──────────────────┐
 │                                                            │
 │  メソッド呼び出しカウンタ:  12,345 回                       │
 │  バックエッジカウンタ:      98,765 回                       │
 │  (ループの繰り返し回数の近似値)                              │
 │                                                            │
 │  分岐プロファイル:                                          │
 │    if (x > 0) の分岐: true 側 95%, false 側 5%             │
 │                                                            │
 │  型プロファイル:                                            │
 │    obj.method() の obj の実行時型:                          │
 │      String   90%                                          │
 │      Integer  10%                                          │
 │                                                            │
 │  呼び出しサイトプロファイル:                                 │
 │    invokevirtual #15 → ターゲットメソッドの統計             │
 │                                                            │
 └────────────────────────────────────────────────────────────┘
```

**JIT はこの情報を活用する:**
- 「この分岐は95%が true 側」→ true 側を前提にした機械語を生成（分岐予測最適化）
- 「obj は90%が String」→ String 前提でインライン化（投機的最適化 + ガード）
- 「このメソッドは10,000回以上呼ばれた」→ コンパイル対象に選出

- 関連ソースパス: `src/hotspot/share/oops/methodData.hpp`
- 関連ソースパス: `src/hotspot/share/oops/methodData.cpp`

## 3. JIT コンパイラ -- ホットスポットを機械語に変換する

**対象層:** HotSpot C++ 層

### 3.1 なぜ JIT が必要か -- インタプリタの限界

インタプリタの性能上の問題は、**同じコードを何度実行しても毎回解釈し直す** ことにある。

```
 ┌─────────────────────────────────────────────────────────────┐
 │  インタプリタ vs JIT コンパイル済みコードの実行速度           │
 │                                                             │
 │  単純なループ (for (int i = 0; i < 1_000_000; i++))         │
 │                                                             │
 │  インタプリタ:                                               │
 │    各反復で iload → iconst_1 → iadd → istore を           │
 │    テンプレートから解釈 × 100万回                            │
 │    → 命令ディスパッチのオーバーヘッドが支配的               │
 │                                                             │
 │  JIT コンパイル後:                                           │
 │    ループ全体が最適化された機械語になる                      │
 │    → CPU のレジスタで直接カウントアップ                     │
 │    → ループ展開で分岐回数も削減                             │
 │    → 数十倍〜数百倍高速                                    │
 └─────────────────────────────────────────────────────────────┘
```

ただし、JIT コンパイル自体に時間がかかる。プログラムの実行中にバックグラウンドでコンパイルするため、**すべてのメソッドをコンパイルするのではなく、頻繁に実行される「ホットスポット」だけを選択的にコンパイルする**。これが HotSpot VM の名前の由来である。

### 3.2 C1 (Client Compiler) -- 素早く軽い最適化

C1 コンパイラは、**コンパイル速度を重視** した軽量な最適化コンパイラである。

```
 ┌─ C1 コンパイラの特徴 ───────────────────────────────────────┐
 │                                                              │
 │  コンパイル速度:  速い（数μs〜数ms）                         │
 │  最適化レベル:    軽い（基本的なインライン化、定数畳み込み等） │
 │  生成コード品質:  中程度                                     │
 │  プロファイリング: コンパイル後も継続的に収集                 │
 │  用途:           起動速度の向上、C2 への橋渡し               │
 │                                                              │
 │  C1 のコンパイルパイプライン:                                │
 │  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────────┐  ┌──────────┐  │
 │  │ HIR  │→│ LIR  │→│レジスタ│→│コード     │→│Code Cache│  │
 │  │(高レベル│ │(低レベル│ │割り当て│ │生成       │ │に格納    │  │
 │  │  IR)  │ │  IR)  │ │      │ │(機械語)   │ │         │  │
 │  └──────┘  └──────┘  └──────┘  └──────────┘  └──────────┘  │
 │                                                              │
 │  HIR = High-level Intermediate Representation                │
 │  LIR = Low-level Intermediate Representation                 │
 └──────────────────────────────────────────────────────────────┘
```

- 関連ソースパス: `src/hotspot/share/c1/c1_Compiler.cpp`
- 関連ソースパス: `src/hotspot/share/c1/c1_LIR.cpp`

### 3.3 C2 (Server Compiler) -- 攻めの最適化

C2 コンパイラは、**ピーク性能を追求** する積極的な最適化コンパイラである。コンパイルに時間がかかるが、生成されるコードは C/C++ コンパイラに匹敵する品質になる。

```
 ┌─ C2 コンパイラの特徴 ───────────────────────────────────────┐
 │                                                              │
 │  コンパイル速度:  遅い（数ms〜数百ms）                       │
 │  最適化レベル:    積極的（インライン化, エスケープ解析,       │
 │                  ループ展開, ベクトル化 等の多数の最適化）    │
 │  生成コード品質:  高い                                       │
 │  プロファイリング: しない（最終最適化コード）                 │
 │  用途:           長時間稼働するサーバーアプリのピーク性能     │
 │                                                              │
 │  C2 のコンパイルパイプライン:                                │
 │  ┌──────┐  ┌──────────────┐  ┌──────────┐  ┌──────────┐    │
 │  │Ideal │→│Global Value  │→│レジスタ   │→│Code Cache│    │
 │  │Graph │  │Numbering +   │  │割り当て   │  │に格納    │    │
 │  │(Sea  │  │ループ最適化 + │  │+ コード   │  │         │    │
 │  │ of   │  │エスケープ解析│  │生成       │  │         │    │
 │  │Nodes)│  │+ etc.        │  │          │  │         │    │
 │  └──────┘  └──────────────┘  └──────────┘  └──────────┘    │
 │                                                              │
 │  Ideal Graph = 「ノードの海 (Sea of Nodes)」と呼ばれる       │
 │  中間表現。制御フローとデータフローを統合的に表現する。       │
 └──────────────────────────────────────────────────────────────┘
```

**C1 と C2 の比較:**

| 項目 | C1 (Client) | C2 (Server) |
|------|-------------|-------------|
| コンパイル速度 | 速い（μs〜ms） | 遅い（ms〜百ms） |
| 最適化の積極性 | 軽い | 攻めの最適化 |
| 中間表現 | HIR → LIR | Ideal Graph (Sea of Nodes) |
| プロファイリング | コンパイル後も収集 | しない |
| インライン化 | 浅い | 深い（複数階層） |
| エスケープ解析 | しない | する |
| ループ展開 | しない | する |
| コードの寿命 | 短い（C2 で置き換わる） | 長い（最終最適化） |

- 関連ソースパス: `src/hotspot/share/opto/compile.cpp`（C2 コンパイラのメイン）
- 関連ソースパス: `src/hotspot/share/opto/node.hpp`（Ideal Graph のノード定義）

## 4. Tiered Compilation -- 段階的コンパイルの全体像

**対象層:** HotSpot C++ 層（`src/hotspot/share/compiler/compileBroker.cpp`）

[jvm-overview.md のセクション 5.3](../jvm-overview.md#53-tiered-compilation段階的コンパイル) で概要を示した Tiered Compilation の内部を詳しく見る。

### 4.1 5つのコンパイルレベル

Tiered Compilation では、コードの実行は **5段階のレベル** を遷移する。

```
 ┌─────────────────────────────────────────────────────────────┐
 │              Tiered Compilation の5レベル                    │
 │                                                             │
 │  Level 0: インタプリタ                                      │
 │    バイトコードを逐次解釈。プロファイリング情報を収集する。  │
 │                                                             │
 │  Level 1: C1 コンパイル（プロファイリングなし）              │
 │    軽い最適化。プロファイリングのオーバーヘッドなし。         │
 │    → 呼び出し回数が少ないメソッドの最終形。                 │
 │                                                             │
 │  Level 2: C1 コンパイル（基本プロファイリングあり）          │
 │    呼び出しカウンタ + バックエッジカウンタを収集。           │
 │    → C2 が忙しい時の一時的な状態。                          │
 │                                                             │
 │  Level 3: C1 コンパイル（フルプロファイリングあり）          │
 │    分岐プロファイル + 型プロファイルも収集。                 │
 │    → C2 へ進むためのメインルート。                          │
 │                                                             │
 │  Level 4: C2 コンパイル（フル最適化）                       │
 │    プロファイリング情報を活用した積極的な最適化。            │
 │    → 長時間稼働するメソッドの最終形。                       │
 │                                                             │
 └─────────────────────────────────────────────────────────────┘
```

**典型的な遷移パス:**

```
 ┌─────────────────────────────────────────────────────────────┐
 │                                                             │
 │  【通常のホットメソッド】                                    │
 │  Level 0 ──→ Level 3 ──→ Level 4                           │
 │  (インタプリタ)  (C1+プロファイル)  (C2 フル最適化)          │
 │                                                             │
 │  【C2 が忙しい場合】                                        │
 │  Level 0 ──→ Level 2 ──→ Level 3 ──→ Level 4              │
 │                (C1 軽量)   (C1+プロファイル)  (C2)           │
 │                                                             │
 │  【単純で小さいメソッド（trivial method）】                  │
 │  Level 0 ──→ Level 1                                       │
 │               (C1 のみ、C2 不要)                            │
 │                                                             │
 │  【脱最適化（投機が外れた）】                                │
 │  Level 4 ──→ Level 0 ──→ Level 3 ──→ Level 4              │
 │  (C2 破棄)   (インタプリタ)  (再プロファイル)  (再 C2)       │
 │                                                             │
 └─────────────────────────────────────────────────────────────┘
```

### 4.2 コンパイルの閾値と判定基準

メソッドが JIT コンパイルの対象になるかは、**呼び出しカウンタ**と**バックエッジカウンタ**で判定される。

```
 ┌─ コンパイル判定の2つのカウンタ ──────────────────────────────┐
 │                                                              │
 │  呼び出しカウンタ (Invocation Counter):                      │
 │    メソッドが呼び出されるたびに +1。                         │
 │    「このメソッドは頻繁に呼ばれている」ことを示す。          │
 │                                                              │
 │  バックエッジカウンタ (Back-Edge Counter):                   │
 │    ループ内のバックエッジ（ループの先頭への戻りジャンプ）     │
 │    を通過するたびに +1。                                     │
 │    「このメソッド内のループが頻繁に回っている」ことを示す。  │
 │                                                              │
 │  判定式 (概略):                                              │
 │    invocation_count + back_edge_count > CompileThreshold      │
 │    → コンパイル対象に選出                                    │
 │                                                              │
 └──────────────────────────────────────────────────────────────┘
```

**主要な閾値パラメータ:**

| パラメータ | デフォルト値 | 説明 |
|-----------|------------|------|
| `-XX:CompileThreshold` | 10,000 (非Tiered) | コンパイルまでの呼び出し回数 |
| `-XX:Tier3InvocationThreshold` | 200 | Level 0 → Level 3 の閾値 |
| `-XX:Tier4InvocationThreshold` | 5,000 | Level 3 → Level 4 の閾値 |

Tiered Compilation が有効（デフォルト）の場合、メソッドは比較的少ない呼び出し回数（約200回）で C1 コンパイルされる。これにより **起動速度が大幅に改善** される。

### 4.3 コンパイルキュー -- バックグラウンドの非同期コンパイル

JIT コンパイルは **メインスレッドとは別のコンパイラスレッド** でバックグラウンド実行される。

```
 ┌─────────────────────────────────────────────────────────────┐
 │            JIT コンパイルの非同期実行                         │
 │                                                             │
 │  main スレッド:                                              │
 │  ──[実行]──[実行]──[閾値到達!]──[実行を継続]──[JIT版で実行]─▶│
 │                         │                        ▲          │
 │                         │ コンパイル要求          │ 差替え   │
 │                         ▼                        │          │
 │  C1 スレッド:  ──[待機]──[C1 コンパイル]──[完了]──┘           │
 │                                                             │
 │  C2 スレッド:  ──[待機]────────[C2 コンパイル]──[完了]──▶     │
 │                                                             │
 │  ★ コンパイル中もアプリケーションは止まらない               │
 │  ★ コンパイルが完了すると、次の呼び出しから JIT 版に切り替わる│
 └─────────────────────────────────────────────────────────────┘
```

**コンパイラスレッドの数:**

| スレッド種別 | デフォルト数 | 調整フラグ |
|------------|------------|-----------|
| C1 コンパイラスレッド | CPU コア数に応じて自動（1〜数本） | `-XX:CICompilerCount` |
| C2 コンパイラスレッド | CPU コア数に応じて自動（1〜数本） | `-XX:CICompilerCount` |

- 関連ソースパス: `src/hotspot/share/compiler/compileBroker.cpp`

### 4.4 OSR (On-Stack Replacement) -- 実行中のループを差し替える

通常の JIT コンパイルは「次にメソッドが呼ばれたとき」に効果が出る。しかし、**長時間実行されるループ** の場合、メソッドが終わるまで JIT の恩恵を受けられない。

OSR (On-Stack Replacement) は、**実行中のメソッドを、そのメソッドの途中からコンパイル済みコードに差し替える** 技術である。

```
 ┌─ OSR なし ──────────────────────────────────────────────────┐
 │                                                              │
 │  void longLoop() {                                          │
 │      for (int i = 0; i < 10_000_000; i++) {                 │
 │          // インタプリタで実行中...                           │
 │          // 途中でコンパイル完了しても、                      │
 │          // このメソッドの呼び出しは終わるまで                │
 │          // インタプリタのまま...                             │
 │      }                                                      │
 │  }                                                          │
 └──────────────────────────────────────────────────────────────┘

 ┌─ OSR あり ──────────────────────────────────────────────────┐
 │                                                              │
 │  void longLoop() {                                          │
 │      for (int i = 0; i < 10_000_000; i++) {                 │
 │          // i=0〜5000: インタプリタで実行                    │
 │          //                                                  │
 │          // バックエッジカウンタが閾値到達！                  │
 │          // → OSR コンパイルをトリガー                       │
 │          //                                                  │
 │          // i=5001〜: ★ ループの途中から JIT コードに切替え  │
 │          // スタックフレームを書き換えて、                    │
 │          // コンパイル済みコードの対応する位置にジャンプ      │
 │      }                                                      │
 │  }                                                          │
 └──────────────────────────────────────────────────────────────┘
```

`-XX:+PrintCompilation` の出力で OSR コンパイルを確認できる。OSR の場合、出力に `%` マークが付く。

```
  123   4 %     3       MyClass::longLoop @ 5 (42 bytes)
                ^                          ^ ^
                |                          | OSR のエントリポイント（バイトコードオフセット）
                |                          |
                OSR マーク
```

## 5. JIT コンパイラの最適化テクニック

**対象層:** HotSpot C++ 層（主に `src/hotspot/share/opto/`）

### 5.1 インライン化 (Inlining) -- 最も重要な最適化

インライン化は、メソッド呼び出しを **呼び出し先の本体で置き換える** 最適化である。JIT 最適化の中で **最も効果が大きい** とされる。

```
 ┌─ インライン化前 ────────────────────────────────────────────┐
 │                                                              │
 │  int calculate(int x) {                                     │
 │      return square(x) + double_(x);                         │
 │  }                                                          │
 │  int square(int n)  { return n * n; }                       │
 │  int double_(int n) { return n * 2; }                       │
 │                                                              │
 │  呼び出しごとに:                                             │
 │    ① 引数をスタックに積む                                   │
 │    ② メソッドディスパッチ（vtable ルックアップ）             │
 │    ③ 新しいスタックフレームを作成                            │
 │    ④ 実行                                                   │
 │    ⑤ 戻り値を返す                                           │
 │    ⑥ スタックフレームを破棄                                  │
 └──────────────────────────────────────────────────────────────┘

 ┌─ インライン化後 ────────────────────────────────────────────┐
 │                                                              │
 │  int calculate(int x) {                                     │
 │      return (x * x) + (x * 2);  // ← メソッド呼び出しが消えた│
 │  }                                                          │
 │                                                              │
 │  → メソッドディスパッチのコスト = 0                          │
 │  → スタックフレームの作成/破棄 = 0                           │
 │  → さらにこの後、定数畳み込みやレジスタ最適化が可能に       │
 └──────────────────────────────────────────────────────────────┘
```

**インライン化が「最も重要」な理由:** インライン化によってメソッドの境界が消え、コンパイラがより広い範囲を見渡せるようになる。これにより定数畳み込み、デッドコード除去、エスケープ解析など **他の最適化の適用範囲が劇的に広がる。**

**インライン化の制限:**

| 制限 | デフォルト値 | フラグ |
|------|------------|--------|
| バイトコードサイズの上限 | 35バイト（頻繁に呼ばれる場合は325バイト） | `-XX:MaxInlineSize`, `-XX:FreqInlineSize` |
| インライン化の深さ上限 | 9段階 | `-XX:MaxInlineLevel` |

### 5.2 エスケープ解析 (Escape Analysis) -- ヒープ割り当ての省略

エスケープ解析は、オブジェクトが **メソッドの外に「逃げない」（エスケープしない）** ことを証明し、ヒープ割り当てを省略する最適化である。

```
 ┌─ エスケープ解析前 ──────────────────────────────────────────┐
 │                                                              │
 │  int sumPoint(int x, int y) {                               │
 │      var p = new Point(x, y);  // ヒープにオブジェクト生成   │
 │      return p.x() + p.y();     // フィールドアクセス         │
 │  }                             // p はこのメソッドでしか使わない│
 │                                                              │
 │  → new Point() が呼ばれるたびにヒープ割り当て + GC 負荷     │
 └──────────────────────────────────────────────────────────────┘

 ┌─ エスケープ解析後 ──────────────────────────────────────────┐
 │                                                              │
 │  int sumPoint(int x, int y) {                               │
 │      // Point はスカラー化される（分解される）               │
 │      int p_x = x;              // フィールドがローカル変数に │
 │      int p_y = y;              // 展開される                 │
 │      return p_x + p_y;         // ヒープ割り当てなし！       │
 │  }                                                          │
 │                                                              │
 │  → オブジェクト生成が完全に消える                            │
 │  → GC 負荷もゼロ                                            │
 └──────────────────────────────────────────────────────────────┘
```

**エスケープの3段階:**

| レベル | 状態 | 最適化 |
|--------|------|--------|
| **NoEscape** | メソッド内で完結 | スカラー置換（ヒープ割り当て省略） |
| **ArgEscape** | 引数として渡されるが、呼び出し先から戻らない | スタック上割り当て |
| **GlobalEscape** | フィールドに保存される等、メソッド外に逃げる | 最適化不可（通常のヒープ割り当て） |

### 5.3 ループ最適化

C2 コンパイラは複数のループ最適化を適用する。

**ループ展開 (Loop Unrolling):**

```
 ┌─ 展開前 ──────────────────┐   ┌─ 展開後（4倍展開）────────────┐
 │                            │   │                                │
 │  for (i = 0; i < 100; i++)│   │  for (i = 0; i < 100; i += 4) │
 │      sum += a[i];          │   │      sum += a[i];              │
 │                            │   │      sum += a[i+1];            │
 │  分岐: 100回              │   │      sum += a[i+2];            │
 │                            │   │      sum += a[i+3];            │
 │                            │   │                                │
 │                            │   │  分岐: 25回（4倍速い）         │
 └────────────────────────────┘   └────────────────────────────────┘
```

**ループ不変式外出し (Loop-Invariant Code Motion):**

```
 ┌─ 最適化前 ──────────────────┐   ┌─ 最適化後 ──────────────────┐
 │                              │   │                              │
 │  for (i = 0; i < n; i++)    │   │  int len = arr.length;       │
 │      if (i < arr.length)    │   │  for (i = 0; i < n; i++)    │
 │          sum += arr[i];     │   │      if (i < len)            │
 │                              │   │          sum += arr[i];     │
 │  arr.length をループ内で     │   │                              │
 │  毎回評価している            │   │  arr.length をループ外に移動 │
 └──────────────────────────────┘   └──────────────────────────────┘
```

### 5.4 定数畳み込みとデッドコード除去

**定数畳み込み (Constant Folding):**

コンパイル時に確定する計算を事前に実行する。

```
 int x = 3 * 4 + 5;     →    int x = 17;
 String s = "Hello" + " " + "World";  →  String s = "Hello World";
```

**デッドコード除去 (Dead Code Elimination):**

実行されないコード、結果が使われない計算を除去する。

```
 int unused = heavyCompute();  // 結果が使われない → 呼び出し自体を除去
 if (false) { ... }            // 到達しないコード → 除去
```

### 5.5 投機的最適化とガード

C2 の最も強力な特徴は **投機的最適化 (Speculative Optimization)** である。プロファイリング情報から「たぶんこうだろう」と仮定し、その仮定に基づいて積極的に最適化する。仮定が外れた場合は **ガード** が発動し、脱最適化する。

```
 ┌─ 投機的最適化の例 ──────────────────────────────────────────┐
 │                                                              │
 │  // 元のコード                                              │
 │  void process(Shape shape) {                                │
 │      double area = shape.area();  // 仮想メソッド呼び出し    │
 │  }                                                          │
 │                                                              │
 │  // プロファイリング情報: shape は 100% Circle だった        │
 │                                                              │
 │  // C2 の投機的最適化後 (擬似コード):                        │
 │  void process(Shape shape) {                                │
 │      if (shape.getClass() != Circle.class) {                │
 │          // ★ ガード: 仮定が外れたら脱最適化               │
 │          deoptimize();  // インタプリタに戻る                │
 │      }                                                      │
 │      // Circle 前提でインライン化された area()               │
 │      double area = ((Circle)shape).radius * Math.PI          │
 │                    * ((Circle)shape).radius;                 │
 │      // → 仮想ディスパッチのコストがゼロに！                │
 │  }                                                          │
 │                                                              │
 └──────────────────────────────────────────────────────────────┘
```

## 6. 脱最適化 (Deoptimization) -- JIT の「やり直し」

**対象層:** HotSpot C++ 層（`src/hotspot/share/runtime/deoptimization.cpp`）

### 6.1 なぜ脱最適化が必要か

JIT コンパイラの投機的最適化は **仮定の上に成り立っている**。仮定が実行時に崩れた場合、JIT が生成したコードは正しく動作しない可能性がある。このとき JVM は以下を行う:

1. JIT コンパイル済みコードの実行を中断する
2. スタックフレームをインタプリタ形式に再構築する
3. インタプリタモードで安全に実行を再開する
4. 必要に応じて再プロファイリングし、再コンパイルする

```
 ┌─────────────────────────────────────────────────────────────┐
 │                    脱最適化の流れ                            │
 │                                                             │
 │  C2 最適化コード: Shape は常に Circle と仮定して最適化       │
 │       │                                                     │
 │       ▼                                                     │
 │  ガードチェック: shape instanceof Circle ?                   │
 │       │                                                     │
 │       ├──→ YES (99.9%) → 最適化コードを続行               │
 │       │                                                     │
 │       └──→ NO  (0.1%)  → ★ 脱最適化！                     │
 │                 │                                           │
 │                 ▼                                           │
 │  ① JIT コードの実行を停止                                  │
 │  ② 現在の状態（ローカル変数、スタック等）をインタプリタ     │
 │     形式のスタックフレームに変換                             │
 │  ③ インタプリタで実行を再開                                 │
 │  ④ 新しいプロファイリング情報を収集                         │
 │  ⑤ 再度 C2 でコンパイル（今度は Circle 以外も考慮）        │
 │                                                             │
 └─────────────────────────────────────────────────────────────┘
```

### 6.2 Uncommon Trap -- 「ありえないはず」の分岐

Uncommon Trap は、JIT コンパイラが「この分岐はほぼ通らないだろう」と判断したコードパスに設置するトラップである。

```
 C2 コンパイル時:
   if (obj == null) {
       // プロファイルでは1回も通っていない → Uncommon Trap を設置
       // 実際のコードは生成しない（デフォルト処理のみ）
   } else {
       // 通常パスのコードを最適化して生成
       obj.process();
   }
```

Uncommon Trap が発動すると脱最適化が起き、インタプリタに戻る。一定回数以上発動すると、JVM は「この仮定は間違っていた」と判断し、再コンパイル時には Uncommon Trap を除去する。

### 6.3 脱最適化の発生条件

| 条件 | 説明 | 例 |
|------|------|-----|
| **型ガードの失敗** | 投機的型推定が外れた | 常に String だと仮定したが Integer が来た |
| **null ガードの失敗** | null でないと仮定したが null が来た | `obj.method()` の obj が null |
| **クラスのロード** | 未ロードのサブクラスがロードされた | 新しい Shape の実装が追加された |
| **コードパッチ** | 最適化に使った前提が変わった | static フィールドの値が変更された |
| **Uncommon Trap** | 「通らないはず」の分岐に到達した | 例外パスに初めて到達 |

**脱最適化は性能低下の原因になる:** 脱最適化が頻繁に発生すると、JIT コンパイル → 脱最適化 → 再コンパイル のサイクルを繰り返すことになる。`-XX:+PrintCompilation` や `-XX:+TraceDeoptimization` で頻繁な脱最適化を検出できる。

- 関連ソースパス: `src/hotspot/share/runtime/deoptimization.cpp`

## 7. コードキャッシュ -- JIT 出力の格納場所

**対象層:** HotSpot C++ 層（`src/hotspot/share/code/codeCache.cpp`）

### 7.1 3セグメント構造 (Java 9+)

JIT コンパイラが生成した機械語コードは **コードキャッシュ (Code Cache)** に格納される。Java 9 以降、コードキャッシュは3つのセグメントに分割されている。

```
 ┌─ コードキャッシュの3セグメント構造 ─────────────────────────┐
 │                                                              │
 │  ┌─ CodeHeap 'non-nmethods' ──────────────────────────────┐ │
 │  │  JVM 内部のスタブコード、アダプター                     │ │
 │  │  (バイトコード→機械語の変換テーブル等)                  │ │
 │  │  デフォルトサイズ: 約 5 MB                              │ │
 │  └────────────────────────────────────────────────────────┘ │
 │                                                              │
 │  ┌─ CodeHeap 'profiled nmethods' ─────────────────────────┐ │
 │  │  C1 コンパイル済みコード（プロファイリング付き）         │ │
 │  │  → C2 で再コンパイルされるまでの一時的な格納場所        │ │
 │  │  → Level 3 のコードがここに入る                        │ │
 │  │  デフォルトサイズ: 約 122 MB                            │ │
 │  └────────────────────────────────────────────────────────┘ │
 │                                                              │
 │  ┌─ CodeHeap 'non-profiled nmethods' ─────────────────────┐ │
 │  │  C2 コンパイル済みコード（最終最適化版）                 │ │
 │  │  → Level 1 と Level 4 のコードがここに入る              │ │
 │  │  → 長期間使用されるコード                               │ │
 │  │  デフォルトサイズ: 約 122 MB                            │ │
 │  └────────────────────────────────────────────────────────┘ │
 │                                                              │
 │  合計デフォルト: 約 240 MB (-XX:ReservedCodeCacheSize)       │
 └──────────────────────────────────────────────────────────────┘
```

コードキャッシュの詳細は [memory-basics.md のセクション 5.4](../memory/memory-basics.md#54-コードキャッシュ-code-cache) を参照。`CodeCacheExplorer.java` でセグメントごとの使用状況を確認できる。

### 7.2 コードキャッシュが満杯になったら

コードキャッシュが満杯になると、**JIT コンパイルが停止** する。以降のすべてのメソッドはインタプリタで実行されるため、パフォーマンスが大幅に劣化する。

```
 コードキャッシュ枯渇時の影響:

 通常時:
   ホットメソッド → [C1] → [C2] → 機械語で高速実行

 コードキャッシュ満杯時:
   ホットメソッド → コンパイル不可 → インタプリタのまま（遅い！）
```

**JVM はコードキャッシュの逼迫を緩和する仕組みを持つ:**
- **コードフラッシング:** 古いコンパイル済みコードやプロファイルコード（C1 Level 3）を Code Cache から除去し、空き領域を確保する
- **Speculative sweeping:** 一定期間呼ばれていないコンパイル済みコードをマークし、回収対象にする

**対策:**

| 対策 | フラグ |
|------|--------|
| コードキャッシュサイズを拡大 | `-XX:ReservedCodeCacheSize=512m` |
| コードフラッシングの有効化（デフォルトで有効） | `-XX:+UseCodeCacheFlushing` |
| JMX でコードキャッシュ使用量を監視 | `CodeCacheExplorer.java` で確認可能 |

- 関連ソースパス: `src/hotspot/share/code/codeCache.cpp`

## 8. 実務での活用 -- JVM フラグとモニタリング

**対象層:** Java 層 / HotSpot C++ 層

### 8.1 JIT 関連の主要 JVM フラグ

| フラグ | デフォルト | 説明 |
|--------|-----------|------|
| `-XX:+TieredCompilation` | 有効 | Tiered Compilation の有効/無効 |
| `-XX:TieredStopAtLevel=N` | 4 | 最大コンパイルレベルを制限（1=C1のみ, 4=C2まで） |
| `-XX:+PrintCompilation` | 無効 | JIT コンパイルのログを表示 |
| `-XX:CompileThreshold=N` | 10000 | コンパイルまでの呼び出し回数（非Tiered時） |
| `-XX:ReservedCodeCacheSize=N` | 240MB | コードキャッシュの最大サイズ |
| `-XX:+UnlockDiagnosticVMOptions -XX:+PrintInlining` | 無効 | インライン化の判定結果を表示 |
| `-Xint` | -- | インタプリタのみで実行（JIT 無効化） |
| `-Xcomp` | -- | 全メソッドを JIT コンパイル（インタプリタ最小化） |

**`-Xint` の用途:** JIT コンパイルの効果を測定したい場合のベースラインとして使う。通常のアプリケーションでは使わない。

**`-Xcomp` の用途:** ほぼ使わない。プロファイリング情報なしでコンパイルするため、投機的最適化が効かず、逆に遅くなる場合がある。

### 8.2 -XX:+PrintCompilation の読み方

```bash
java -XX:+PrintCompilation MyApp.java
```

**出力例:**

```
   時刻 ID  属性  レベル  クラス::メソッド (サイズ)
   ───────────────────────────────────────────────────
    42   1       3       java.lang.String::hashCode (55 bytes)
    43   2       3       java.lang.String::equals (50 bytes)
    45   3  s    4       java.lang.StringLatin1::hashCode (42 bytes)
    48   1       3       java.lang.String::hashCode (55 bytes)   made not entrant
    50   4  %    3       MyApp::hotLoop @ 5 (35 bytes)
```

**各カラムの意味:**

| カラム | 意味 |
|--------|------|
| 時刻 | JVM 起動からのミリ秒 |
| ID | コンパイルの一意 ID |
| 属性 | `%` = OSR, `s` = synchronized, `!` = 例外ハンドラあり, `n` = native |
| レベル | 1-4 のコンパイルレベル（3=C1+プロファイル, 4=C2） |
| クラス::メソッド | コンパイル対象 |
| (サイズ) | バイトコードのサイズ |
| made not entrant | 古いバージョンが無効化された（新バージョンに置き換え） |

### 8.3 Spring Boot と実行エンジン

**起動速度の問題:**

Spring Boot アプリの起動時には数万クラスがロード・初期化される。このとき JIT コンパイルは間に合わず、ほとんどのコードがインタプリタで実行される。

```
 Spring Boot 起動タイムライン:

 0s          1s          2s          3s          4s
 │───────────│───────────│───────────│───────────│
 │  インタプリタで大量のクラス初期化   │           │
 │  [C1 コンパイル開始]              │           │
 │           │ [C2 コンパイル開始]    │           │
 │           │           │           │ 定常状態   │
 │           │           │           │ (C2最適化) │
 │← 起動フェーズ（遅い）→│←ウォームアップ→│← 高速 →│
```

**対策:**

| 対策 | 効果 |
|------|------|
| **CDS / AppCDS** | クラスロードと検証を高速化（起動時間短縮） |
| **AOT コンパイル (GraalVM Native Image)** | ビルド時に機械語にコンパイル（起動時間を数十ms に） |
| **Spring AOT** | Spring の DI 処理をビルド時に実行（ランタイム負荷を削減） |
| **CRaC (Coordinated Restore at Checkpoint)** | JVM の状態をスナップショットし、復元起動する |

**ピーク性能のウォームアップ:**

Spring Boot アプリが本来の性能を発揮するのは、主要なコードパスが C2 コンパイルされた後（起動から数秒〜数十秒後）である。ロードバランサーのヘルスチェックや readiness probe は、ウォームアップ完了を考慮して設定すべきである。

### Q&A 形式の最重要概念

| 質問 | 回答 |
|------|------|
| バイトコードとは？ | javac が生成する中間表現。1命令 = 1バイトのオペコード。約200種類 |
| JVM はレジスタマシン？スタックマシン？ | **スタックマシン**。計算はオペランドスタックを使って行う。移植性のため |
| インタプリタの役割は？ | ① 即時実行 ② プロファイリング ③ 脱最適化のフォールバック |
| Template Interpreter とは？ | JVM 起動時に各バイトコード命令用の機械語テンプレートを生成し、直接実行する高速インタプリタ |
| C1 と C2 の違いは？ | C1 = 高速コンパイル + 軽い最適化、C2 = 低速コンパイル + 積極的な最適化 |
| Tiered Compilation のレベル数は？ | 5レベル（Level 0 〜 Level 4） |
| 最も重要な JIT 最適化は？ | **インライン化**。他の最適化の適用範囲を広げる基盤 |
| エスケープ解析の効果は？ | メソッド内で完結するオブジェクトのヒープ割り当てを省略（スカラー置換） |
| OSR とは？ | On-Stack Replacement。実行中のループを途中から JIT コードに切り替える技術 |
| 脱最適化はなぜ起きる？ | 投機的最適化の前提が崩れた場合。型ガード失敗、新クラスのロード等 |
| コードキャッシュが満杯になると？ | JIT コンパイルが停止し、以降はインタプリタのまま → 性能劣化 |
| `-XX:+PrintCompilation` の `%` は？ | OSR コンパイルであることを示す |
| Spring Boot の起動が遅い原因は？ | 大量のクラスロード + JIT がまだウォームアップしていない |
