## 概要

本ドキュメントは、Java 21 で正式導入 (JEP 444) された**仮想スレッド (Virtual Threads)** の内部構造と活用方法を詳細に解説するものである。仮想スレッドは **Project Loom** の中核成果物であり、JVM の並行処理モデルを根本から変革する機能である。

仮想スレッドに加えて、Project Loom のもう2つの柱である **Structured Concurrency（構造化並行処理）** と **Scoped Values（スコープ付き値）** についても、Java 25 時点の最新状態を含めて解説する。

**対象層:** Java 層 / JNI ネイティブ層 / HotSpot C++ 層

**対象ランタイム:** Azul Zulu 25 (OpenJDK 25 ベース)

**前提知識:** Java のスレッドモデルの基礎（`Thread`, `Runnable`, `ExecutorService`）

## 第1章: なぜ仮想スレッドが必要なのか

### 1.1 C10K 問題とスレッドモデルの限界

従来の Java サーバーアプリケーションでは、1つのリクエストに1つの **プラットフォームスレッド**（OS ネイティブスレッドと 1:1 対応）を割り当てる **thread-per-request** モデルが広く使われてきた。しかし、このモデルには根本的なスケーラビリティの問題がある。

```
┌─────────────────────────────────────────────────────────────────┐
│            従来の thread-per-request モデルの限界                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  クライアント        サーバー (スレッドプール: 200)                │
│  ─────────          ──────────────────────                      │
│                                                                 │
│  req #1    ───→     Thread-1  [██████████░░░░░░]  (DB 待ち)     │
│  req #2    ───→     Thread-2  [██░░░░░░░░░░░░░░]  (API 待ち)   │
│  req #3    ───→     Thread-3  [████░░░░░░░░░░░░]  (File 待ち)  │
│   ...                ...                                        │
│  req #200  ───→     Thread-200 [██████░░░░░░░░░]  (DB 待ち)    │
│                                                                 │
│  req #201  ───→     ⛔ 拒否！ スレッドプール枯渇！               │
│  req #202  ───→     ⛔ 拒否！                                   │
│                                                                 │
│  ████ = CPU 処理中   ░░░░ = I/O 待ち (ブロック中)                │
│                                                                 │
│  問題: スレッドの大半は I/O 待ちで CPU を使っていないが、          │
│       OS スレッドとスタックメモリ (~1MB) を占有し続ける            │
└─────────────────────────────────────────────────────────────────┘
```

プラットフォームスレッドは OS のネイティブスレッドと 1:1 でマッピングされるため、以下の制約がある:

| 制約 | 詳細 |
|---|---|
| **メモリコスト** | 1スレッドあたり約 1MB のスタック領域（デフォルト）。1万スレッドで約 10GB |
| **OS リソース** | スレッド数は OS のカーネルスレッド上限に制約される（Linux: 数万程度） |
| **コンテキストスイッチ** | OS レベルのコンテキストスイッチは数マイクロ秒のオーバーヘッド |
| **生成コスト** | OS スレッドの生成は数十マイクロ秒のコスト |

いわゆる **C10K 問題**（1万同時接続の壁）は、まさにこのスレッドモデルの限界に起因する。現代のマイクロサービスアーキテクチャでは、1つのリクエストが複数のバックエンドサービスに対して I/O 待ちを行うため、スレッドが I/O でブロックされる時間は処理時間全体の大部分を占める。

### 1.2 非同期プログラミングの代償

C10K 問題に対する従来の解決策は、**非同期プログラミング** であった。`CompletableFuture`、リアクティブプログラミング（Project Reactor, RxJava）、コールバックなどの手法を用いて、1つのスレッドで複数のリクエストを多重化する。

```
┌─────────────────────────────────────────────────────────────────┐
│           非同期プログラミングの問題点                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【同期コード（読みやすい）】                                     │
│                                                                 │
│    User user = userRepo.findById(id);                           │
│    Order order = orderService.getLatest(user);                  │
│    return formatResponse(user, order);                          │
│                                                                 │
│  【非同期コード（CompletableFuture）】                            │
│                                                                 │
│    return userRepo.findByIdAsync(id)                            │
│        .thenCompose(user ->                                     │
│            orderService.getLatestAsync(user)                    │
│                .thenApply(order ->                               │
│                    formatResponse(user, order)))                 │
│        .exceptionally(ex -> handleError(ex));                   │
│                                                                 │
│  問題:                                                           │
│  - コードの可読性が著しく低下（"callback hell"）                  │
│  - スタックトレースが断片化し、デバッグが困難                     │
│  - 既存の同期ライブラリ (JDBC 等) が使えない                      │
│  - try-catch-finally の構造化例外処理が使えない                   │
│  - コード全体が "async" に汚染される ("colored function" 問題)    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

非同期プログラミングは **「シンプルな同期コードのスケーラビリティ」と「非同期コードのパフォーマンス」の二者択一** を開発者に強いてきた。仮想スレッドはこのジレンマを解消する。

### 1.3 仮想スレッドの設計思想 -- "thread-per-request" の復権

仮想スレッドの設計思想は極めてシンプルである:

> **「スレッドは安価であるべきだ。開発者はスレッドの数を気にすることなく、thread-per-request スタイルで素直な同期コードを書けるべきだ」**

この思想は JEP 444 の Motivation セクションに明確に記されている。仮想スレッドは OS スレッドではなく **JVM が管理する軽量スレッド** であり、以下の特性を持つ:

- **1つの仮想スレッドのメモリコストは数百バイトから数 KB**（プラットフォームスレッドの約 1/1000）
- **100万の仮想スレッドを同時に実行可能**（数十のプラットフォームスレッド上で多重化）
- **既存の同期 API がそのまま使える**（JDBC, `InputStream`, `Socket` 等）
- **既存の `Thread` API と互換性がある**（`Thread.sleep`, `join`, `interrupt` 等）

```
┌─────────────────────────────────────────────────────────────────┐
│         仮想スレッドによる thread-per-request の復権               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  クライアント        サーバー                                     │
│  ─────────          ──────────────────────                      │
│                                                                 │
│  req #1    ───→     VT-1     [██░░██░░██]                       │
│  req #2    ───→     VT-2     [██░░░░░░██]                       │
│  req #3    ───→     VT-3     [████░░████]                       │
│   ...                ...                                        │
│  req #10000───→     VT-10000 [██░░░░░░██]                       │
│  req #10001───→     VT-10001 [██░░██████]   ← 問題なし！        │
│                                                                 │
│      仮想スレッド (VT) は少数のキャリアスレッド上で多重化:         │
│                                                                 │
│      Carrier-1: [VT-1][VT-5][VT-9]...                          │
│      Carrier-2: [VT-2][VT-6][VT-10]...                         │
│      Carrier-3: [VT-3][VT-7][VT-11]...                         │
│      Carrier-N: [VT-4][VT-8][VT-12]...                         │
│                                                                 │
│      ░░ = I/O 待ち中は VT をアンマウントし、                     │
│           キャリアスレッドは別の VT を実行                        │
└─────────────────────────────────────────────────────────────────┘
```

### 1.4 Project Loom の歴史と JEP の変遷

Project Loom は 2017 年に Ron Pressler 氏を中心に開始された。以下にその主要なマイルストーンを示す:

| 時期 | JEP / イベント | 内容 |

| 2022.09 | JEP 425 (Java 19) | Virtual Threads -- **Preview** |
| 2023.03 | JEP 436 (Java 20) | Virtual Threads -- **Second Preview** |
| 2023.09 | **JEP 444 (Java 21)** | **Virtual Threads -- 正式導入** |
| 2024.03 | JEP 462 (Java 22) | Structured Concurrency -- Second Preview |
| 2024.03 | JEP 464 (Java 22) | Scoped Values -- Second Preview |
| 2024.09 | JEP 480 (Java 23) | Structured Concurrency -- Third Preview |
| 2024.09 | JEP 481 (Java 23) | Scoped Values -- Third Preview |
| 2025.03 | JEP 491 (Java 24) | **synchronized のピン留め解消 -- 正式導入** |
| 2025.03 | JEP 499 (Java 24) | Structured Concurrency -- Fourth Preview |
| 2025.03 | JEP 487 (Java 24) | Scoped Values -- Fourth Preview |
| 2025.09 | JEP 505 (Java 25) | **Structured Concurrency -- 正式化 (GA)** |

## 第2章: プラットフォームスレッド vs 仮想スレッド

### 2.1 プラットフォームスレッドの構造

プラットフォームスレッド（従来の `new Thread()` で生成されるスレッド）は、OS のネイティブスレッドと 1:1 で対応する。

```
┌──────────────────────────────────────────────────────────────────┐
│                プラットフォームスレッドの構造                       │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Java 層                    JVM (HotSpot) 層        OS 層        │
│  ──────                    ──────────────           ─────        │
│                                                                  │
│  java.lang.Thread          JavaThread (C++)      pthread         │
│  ┌──────────────┐          ┌──────────────┐     ┌──────────┐    │
│  │ Thread obj   │ ──1:1──→ │ JavaThread   │ ─→  │ OS Thread│    │
│  │              │          │              │     │          │    │
│  │ - name       │          │ - _osthread  │     │ - stack  │    │
│  │ - priority   │          │ - _stack_base│     │   (~1MB) │    │
│  │ - state      │          │ - _stack_size│     │ - TID    │    │
│  │ - target     │          │ - _thread_id │     │ - state  │    │
│  └──────────────┘          └──────────────┘     └──────────┘    │
│                                                                  │
│  スタック: OS が確保 (~1MB 固定、-Xss で変更可)                   │
│  スケジューリング: OS カーネルが制御                               │
│  コンテキストスイッチ: OS カーネルモード遷移を伴う                 │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.2 仮想スレッドの構造

仮想スレッドは OS スレッドと直接対応しない。代わりに、少数のプラットフォームスレッド（**キャリアスレッド**）上で **M:N スケジューリング** される。

```
┌──────────────────────────────────────────────────────────────────┐
│                  仮想スレッドの構造 (M:N モデル)                    │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Java 層 (仮想スレッド)                                           │
│  ────────────────────                                            │
│  VT-1  VT-2  VT-3  VT-4  VT-5  ...  VT-1000000                 │
│   │     │     │     │     │            │                         │
│   │     │     │     │     │            │                         │
│   └──┬──┘     │     └──┬──┘            │    M:N マッピング       │
│      │        │        │               │    (多数の VT を         │
│      ▼        ▼        ▼               │     少数の CT に         │
│   ┌──────┐ ┌──────┐ ┌──────┐           │     マウント)           │
│   │ CT-1 │ │ CT-2 │ │ CT-3 │  ...  ┌──────┐                    │
│   └──────┘ └──────┘ └──────┘        │ CT-N │                    │
│   キャリアスレッド (= プラットフォームスレッド)                     │
│   N = Runtime.getRuntime().availableProcessors() (デフォルト)     │
│                                                                  │
│  HotSpot C++ 層                                                  │
│  ──────────────                                                  │
│  ForkJoinPool (デフォルトスケジューラ)                              │
│  ┌─────────────────────────────────────┐                         │
│  │ Work Queue │ Work Queue │ Work Queue│  ← ワークスティーリング  │
│  │  [VT-1]   │  [VT-3]   │  [VT-5]  │                          │
│  │  [VT-6]   │  [VT-8]   │  [VT-10] │                          │
│  └─────────────────────────────────────┘                         │
│                                                                  │
│  仮想スレッドのスタック:                                           │
│  - ヒープ上に確保（スタックチャンク方式）                          │
│  - 初期サイズ: 数百バイト                                         │
│  - 必要に応じて動的に拡張/縮小                                    │
│  - GC の管理対象                                                  │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.3 比較表

| 特性 | プラットフォームスレッド | 仮想スレッド |
|---|---|---|
| **OS スレッドとの関係** | 1:1 | M:N (多対少数) |
| **スタックサイズ** | ~1MB 固定 (デフォルト) | 数百バイト〜動的拡張 |
| **生成コスト** | 数十〜数百マイクロ秒 | ~1マイクロ秒 |
| **同時存在数の上限** | 数千〜数万 | 数百万 |
| **スケジューリング** | OS カーネル | JVM (ForkJoinPool) |
| **スタック領域** | OS が確保 (ネイティブメモリ) | Java ヒープ上 (GC 対象) |
| **プーリング** | 必須 (`ExecutorService`) | 不要（使い捨て） |
| **`Thread.isVirtual()`** | `false` | `true` |
| **デーモンスレッド** | デフォルト `false` | 常に `true` |
| **優先度** | 設定可能 (1-10) | 常に `NORM_PRIORITY` (無視される) |
| **`ThreadGroup`** | 所属可能 | 固定の VirtualThreads グループ |

## 第3章: 仮想スレッドの内部構造

### 3.1 キャリアスレッドとの関係

仮想スレッドは単独では実行できない。実際に CPU 上でコードを実行するには、**キャリアスレッド (Carrier Thread)** と呼ばれるプラットフォームスレッドに **マウント (mount)** される必要がある。

「キャリア (carrier)」という名前は「運び手」を意味し、仮想スレッドの実行コンテキストを「運ぶ」プラットフォームスレッドであることを表している。

```
┌─────────────────────────────────────────────────────────────────┐
│              キャリアスレッドと仮想スレッドの関係                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  時刻 T1: VT-1 が CT-1 にマウントされて実行中                    │
│                                                                 │
│  CT-1: ┌─────────────────────────────┐                          │
│        │ VT-1 [実行中: CPU 処理]      │                          │
│        └─────────────────────────────┘                          │
│                                                                 │
│  時刻 T2: VT-1 が I/O でブロック → アンマウント                  │
│                                                                 │
│  CT-1: ┌─────────────────────────────┐                          │
│        │ (空き: 次の VT を待機)       │                           │
│        └─────────────────────────────┘                          │
│                                                                 │
│  VT-1: [アンマウント状態: ヒープ上で待機]                         │
│        ┌─────────────────────────────┐                          │
│        │ Continuation (凍結された     │                          │
│        │  スタックフレーム群)          │                          │
│        └─────────────────────────────┘                          │
│                                                                 │
│  時刻 T3: VT-2 が CT-1 にマウントされて実行開始                  │
│                                                                 │
│  CT-1: ┌─────────────────────────────┐                          │
│        │ VT-2 [実行中: CPU 処理]      │                          │
│        └─────────────────────────────┘                          │
│                                                                 │
│  時刻 T4: VT-1 の I/O 完了 → CT-2 にマウントされて再開           │
│                                                                 │
│  CT-2: ┌─────────────────────────────┐                          │
│        │ VT-1 [実行再開]              │                          │
│        └─────────────────────────────┘                          │
│                                                                 │
│  ※ VT-1 は CT-1 でアンマウントされ、CT-2 で再開された。          │
│     キャリアスレッドは VT の実行ごとに変わり得る。                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

重要なポイント: 仮想スレッドがアンマウントされても、キャリアスレッドは**ブロックされない**。キャリアスレッドはすぐに別の仮想スレッドをマウントして実行を継続できる。これが仮想スレッドのスケーラビリティの源泉である。

### 3.2 マウント/アンマウントの仕組み

マウントとアンマウントは仮想スレッドの中核メカニズムである。

```
┌─────────────────────────────────────────────────────────────────┐
│            マウント / アンマウントのライフサイクル                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  仮想スレッドの状態遷移:                                          │
│                                                                 │
│                 ┌──────────┐                                     │
│                 │  NEW     │                                     │
│                 └────┬─────┘                                     │
│                      │ Thread.start()                            │
│                      ▼                                           │
│           ┌──────────────────┐                                   │
│     ┌────→│  RUNNABLE        │←────────────┐                    │
│     │     │  (スケジューラの  │              │                    │
│     │     │   キューで待機)   │              │                    │
│     │     └────────┬─────────┘              │                    │
│     │              │ マウント                │                    │
│     │              ▼                        │                    │
│     │     ┌──────────────────┐              │                    │
│     │     │  RUNNING         │              │                    │
│     │     │  (CT 上で実行中)  │              │                    │
│     │     └──┬────────────┬──┘              │                    │
│     │        │            │                 │                    │
│     │   ブロック操作    正常完了              │                    │
│     │   (I/O, sleep,      │                 │                    │
│     │    park 等)          │                 │                    │
│     │        │             │                │                    │
│     │        ▼             ▼                │                    │
│     │  ┌──────────┐  ┌──────────┐           │                    │
│     │  │ WAITING  │  │TERMINATED│           │                    │
│     │  │ (アン    │  └──────────┘           │                    │
│     │  │  マウント │                        │                    │
│     │  │  済み)   │                         │                    │
│     │  └─────┬────┘                         │                    │
│     │        │ I/O 完了 / unpark            │                    │
│     └────────┴──────────────────────────────┘                    │
│                                                                 │
│  マウント時の処理:                                                │
│  1. スケジューラがキャリアスレッドを選択                           │
│  2. Continuation のスタックチャンクをキャリアのスタックに復元       │
│  3. 仮想スレッドの実行コンテキストを設定                           │
│  4. 実行再開                                                     │
│                                                                 │
│  アンマウント時の処理:                                            │
│  1. 現在のスタックフレーム群をヒープ上のスタックチャンクに保存      │
│  2. キャリアスレッドから仮想スレッドのコンテキストを切り離す        │
│  3. キャリアスレッドは別の仮想スレッドを実行可能に                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

マウント/アンマウントが発生する主なタイミング:

| 操作 | 動作 |
|---|---|
| `Thread.sleep(Duration)` | アンマウント → 時間経過後にスケジュール |
| `LockSupport.park()` | アンマウント → `unpark()` でスケジュール |
| `BlockingQueue.take()` | アンマウント → 要素到着でスケジュール |
| `InputStream.read()` | アンマウント → データ到着でスケジュール |
| `Socket.connect()` | アンマウント → 接続完了でスケジュール |
| `ReentrantLock.lock()` | アンマウント → ロック取得可能でスケジュール |

### 3.3 Continuation（継続）の内部実装

仮想スレッドのマウント/アンマウントを支える基盤技術が **Continuation（継続）** である。Continuation は「中断可能な計算」を表す抽象であり、実行途中のスタックフレーム群をヒープ上に保存（freeze）し、後から復元（thaw）する機能を提供する。

```
┌─────────────────────────────────────────────────────────────────┐
│              Continuation の freeze / thaw                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【freeze（凍結）: アンマウント時】                               │
│                                                                 │
│  キャリアスレッドのスタック          ヒープ (スタックチャンク)      │
│  ┌─────────────────────┐           ┌─────────────────────┐     │
│  │ Frame: main()       │           │                     │     │
│  │ Frame: handleReq()  │ ──copy──→ │ Frame: handleReq()  │     │
│  │ Frame: queryDB()    │           │ Frame: queryDB()    │     │
│  │ Frame: read()  ← yield          │ Frame: read()       │     │
│  └─────────────────────┘           └─────────────────────┘     │
│  ↑ キャリアのスタックは               ↑ GC 管理下のオブジェクト    │
│    解放され再利用可能                                            │
│                                                                 │
│  【thaw（解凍）: マウント時】                                     │
│                                                                 │
│  ヒープ (スタックチャンク)          キャリアスレッドのスタック      │
│  ┌─────────────────────┐           ┌─────────────────────┐     │
│  │ Frame: handleReq()  │           │ Frame: main()       │     │
│  │ Frame: queryDB()    │ ──copy──→ │ Frame: handleReq()  │     │
│  │ Frame: read()       │           │ Frame: queryDB()    │     │
│  └─────────────────────┘           │ Frame: read()       │     │
│                                    └─────────────────────┘     │
│                                    ↑ read() から実行再開         │
│                                                                 │
│  ※ 実際にはスタック全体をコピーするのではなく、                    │
│    変更されたフレームだけを差分コピーする最適化が行われる           │
│    (incremental freeze / lazy copy)                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

Continuation は Java 層では `jdk.internal.vm.Continuation` クラスとして実装されている（パブリック API ではない）。主要な操作は以下の2つ:

- **`Continuation.yield()`**: 実行を中断し、スタックをヒープに保存する（freeze）
- **`Continuation.run()`**: 保存されたスタックを復元し、中断点から実行を再開する（thaw）

### 3.4 スタック管理（スタックチャンク）

仮想スレッドのスタックは、従来のプラットフォームスレッドのように固定サイズのメモリ領域ではなく、**スタックチャンク (StackChunk)** と呼ばれる Java ヒープ上のオブジェクトとして管理される。

```
┌─────────────────────────────────────────────────────────────────┐
│              スタックチャンクの構造                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  プラットフォームスレッド:                                        │
│  ┌─────────────────────────────────────┐                        │
│  │ OS スタック (~1MB, 固定サイズ)        │ ← ネイティブメモリ      │
│  │ ┌─────────────────────────────────┐ │                        │
│  │ │ Frame-1 │ Frame-2 │ ... │ 空き │ │                        │
│  │ └─────────────────────────────────┘ │                        │
│  └─────────────────────────────────────┘                        │
│                                                                 │
│  仮想スレッド:                                                   │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ StackChunk (Java ヒープ上のオブジェクト, GC 管理対象)       │  │
│  │                                                           │  │
│  │ ┌───────────┐    ┌───────────┐    ┌───────────┐           │  │
│  │ │ Chunk-1   │───→│ Chunk-2   │───→│ Chunk-3   │           │  │
│  │ │ (初期)    │    │ (拡張)    │    │ (拡張)    │           │  │
│  │ │ ~数百B    │    │ ~数KB     │    │ ~数KB     │           │  │
│  │ └───────────┘    └───────────┘    └───────────┘           │  │
│  │                                                           │  │
│  │ - 初期サイズは数百バイト (必要最低限)                       │  │
│  │ - 深い呼び出しチェーンで自動拡張                            │  │
│  │ - 不要になったチャンクは GC が回収                          │  │
│  │ - リンクリスト構造で連結                                    │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                 │
│  メモリ効率の比較 (10万スレッドの場合):                            │
│  - プラットフォームスレッド: 100,000 x 1MB = ~100GB              │
│  - 仮想スレッド (待機中): 100,000 x ~数KB = ~数百MB              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

スタックチャンクが Java ヒープ上にあることの重要な帰結:

1. **GC がスタックチャンクを管理する**: 仮想スレッドが終了すると、スタックチャンクは通常の Java オブジェクトと同様に GC によって回収される
2. **GC はスタックチャンク内の参照を走査する必要がある**: これは GC の STW 停止時間にわずかな影響を与える可能性がある
3. **メモリの動的管理が可能**: スタックは必要に応じて拡張・縮小される

### 3.5 ForkJoinPool がキャリアスレッドプールとして使われる理由

仮想スレッドのデフォルトスケジューラとして `ForkJoinPool` が選択されたのには、明確な技術的理由がある:

1. **ワークスティーリング**: `ForkJoinPool` のワークスティーリングアルゴリズムは、短命なタスク（仮想スレッドのマウント/アンマウント）を効率よく分配する
2. **FIFO モードのサポート**: `ForkJoinPool` は `asyncMode` を有効にすることで FIFO スケジューリングが可能で、これはサーバーのリクエスト処理に適している
3. **既存の成熟した実装**: `ForkJoinPool` は Java 7 以来の成熟した実装であり、大規模なタスクスケジューリングの実績がある
4. **スレッド追加の柔軟性**: ピン留め発生時にキャリアスレッドを動的に追加する "compensation" メカニズムをサポートする

デフォルトのキャリアスレッド数は `Runtime.getRuntime().availableProcessors()` で決まる。`-Djdk.virtualThreadScheduler.parallelism` で変更可能。

### 4.2 VirtualThread クラスの構造

Java 層の `VirtualThread` クラス（`java.lang.VirtualThread`）は `Thread` の非 `public` サブクラスであり、パッケージプライベートである。

```
┌─────────────────────────────────────────────────────────────────┐
│           VirtualThread のクラス階層                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  java.lang.Thread (public)                                      │
│       │                                                         │
│       ├── java.lang.Thread (platform thread)                    │
│       │   └── 内部に OS スレッドへの参照を保持                    │
│       │                                                         │
│       └── java.lang.VirtualThread (package-private)             │
│           ├── cont: Continuation         ← 中断可能な計算本体    │
│           ├── scheduler: Executor        ← スケジューラ          │
│           ├── carrierThread: Thread      ← 現在のキャリア        │
│           ├── state: int                 ← 状態管理              │
│           └── parkPermit: boolean        ← park/unpark 用       │
│                                                                 │
│  VirtualThread の状態定数:                                       │
│  ┌────────────────────────────────────────────────────────┐     │
│  │ NEW        = 0    // 生成直後                           │     │
│  │ STARTED    = 1    // start() 呼び出し済み               │     │
│  │ RUNNING    = 2    // キャリアにマウントされ実行中         │     │
│  │ PARKING    = 3    // park 処理中                        │     │
│  │ PARKED     = 4    // park 完了（アンマウント済み）        │     │
│  │ PINNED     = 5    // ピン留め状態で park                 │     │
│  │ YIELDING   = 6    // yield 処理中                       │     │
│  │ YIELDED    = 7    // yield 完了                         │     │
│  │ TERMINATED = 99   // 実行完了                           │     │
│  └────────────────────────────────────────────────────────┘     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

`VirtualThread` の重要な特性:
- **パブリック API ではない**: `VirtualThread` クラスに直接アクセスすることはできない。`Thread.ofVirtual()` ファクトリメソッドを通じて仮想スレッドを生成する
- **常にデーモンスレッド**: 仮想スレッドは必ずデーモンスレッドとして動作する
- **優先度は無視される**: `setPriority()` は受け付けるが、スケジューリングに影響しない
- **ThreadGroup は固定**: すべての仮想スレッドは "VirtualThreads" グループに所属する

### 4.3 Continuation の freeze/thaw メカニズム

Continuation の freeze（凍結）と thaw（解凍）は HotSpot C++ 層で実装される最も複雑な処理の一つである。

```
┌─────────────────────────────────────────────────────────────────┐
│     Continuation freeze/thaw の処理フロー (層をまたぐ)            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【freeze フロー（仮想スレッドのアンマウント時）】                 │
│                                                                 │
│  Java 層                                                        │
│  ──────                                                         │
│  VirtualThread.park()                                           │
│      │                                                          │
│      └→ Continuation.yield(ContinuationScope)                   │
│            │                                                    │
│  ──────────│────────────────────────────────────────             │
│  JNI 層    │                                                    │
│  ──────    │                                                    │
│            └→ JVM_Yield() (ネイティブメソッド)                   │
│                  │                                              │
│  ────────────────│──────────────────────────────────            │
│  HotSpot C++ 層  │                                              │
│  ──────────────  │                                              │
│                  └→ Continuation::freeze()                      │
│                       │                                         │
│                       ├→ 1. スタックフレームのウォーク            │
│                       │     (現在のフレームから呼び出し元へ)       │
│                       │                                         │
│                       ├→ 2. スタックチャンクの確保/再利用         │
│                       │     (ヒープ上の StackChunk OOP)          │
│                       │                                         │
│                       ├→ 3. フレームデータのコピー                │
│                       │     (レジスタ, ローカル変数, 参照)        │
│                       │                                         │
│                       ├→ 4. OOP マップの更新                    │
│                       │     (GC がスタックチャンク内の参照を      │
│                       │      正しく走査できるように)               │
│                       │                                         │
│                       └→ 5. キャリアスレッドのスタックを巻き戻し  │
│                                                                 │
│  ────────────────────────────────────────────────────           │
│                                                                 │
│  【thaw フロー（仮想スレッドのマウント時）】                       │
│                                                                 │
│  Java 層                                                        │
│  ──────                                                         │
│  VirtualThread.unpark()                                         │
│      │                                                          │
│      └→ scheduler.execute(runContinuation)                      │
│            │                                                    │
│            └→ Continuation.run()                                │
│                  │                                              │
│  ────────────────│──────────────────────────────────            │
│  HotSpot C++ 層  │                                              │
│  ──────────────  │                                              │
│                  └→ Continuation::thaw()                        │
│                       │                                         │
│                       ├→ 1. スタックチャンクから最上位フレームを  │
│                       │     キャリアスレッドのスタックにコピー     │
│                       │                                         │
│                       ├→ 2. フレームポインタの修正                │
│                       │                                         │
│                       ├→ 3. 戻りアドレスの設定                   │
│                       │                                         │
│                       └→ 4. 中断点から実行再開                   │
│                                                                 │
│  ※ thaw は "lazy" に行われる:                                    │
│    全フレームを一度にコピーするのではなく、                        │
│    必要なフレームだけを順次コピーする。                            │
│    これにより thaw のコストが最小化される。                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

freeze/thaw の最適化ポイント:

1. **Incremental freeze**: 前回の freeze 以降に変更されたフレームだけをコピーする
2. **Lazy thaw**: thaw 時に全フレームをコピーするのではなく、実行に必要な最上位のフレームだけを復元する。残りのフレームは必要になった時点でコピーされる
3. **スタックチャンクの再利用**: 同じ仮想スレッドが繰り返し freeze/thaw される場合、以前確保したスタックチャンクを再利用する

### 4.4 スタックチャンクの C++ 実装

`StackChunk` は HotSpot 内部で `stackChunkOop` として表現される特殊なオブジェクトである。

```
┌─────────────────────────────────────────────────────────────────┐
│              stackChunkOop のメモリレイアウト                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌────────────────────────────────────┐                         │
│  │ オブジェクトヘッダ (Mark Word + Klass)│                        │
│  ├────────────────────────────────────┤                         │
│  │ _parent    : stackChunkOop        │ ← 親チャンク (linked list)│
│  │ _size      : int                  │ ← チャンクサイズ          │
│  │ _sp        : int                  │ ← スタックポインタ        │
│  │ _argsize   : int                  │ ← 引数領域サイズ          │

│  ├────────────────────────────────────┤                         │
│  │ スタックデータ (可変長)             │                          │
│  │ ┌──────────────────────────────┐  │                          │
│  │ │ Frame N (最新)               │  │                          │
│  │ │ Frame N-1                    │  │                          │
│  │ │ ...                          │  │                          │
│  │ │ Frame 1 (最古)               │  │                          │
│  │ └──────────────────────────────┘  │                          │
│  └────────────────────────────────────┘                         │
│                                                                 │
│  GC との相互作用:                                                │
│  - スタックチャンクは通常の Java オブジェクトとして GC が管理       │
│  - GC はスタックチャンク内の OOP を走査する必要がある              │
│  - OOP マップ (bitmap) を使って参照の位置を追跡                   │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 第5章: 仮想スレッドのスケジューリング

### 5.1 ForkJoinPool ベースのスケジューラ

仮想スレッドのデフォルトスケジューラは、専用の `ForkJoinPool` インスタンスである。このプールは `java.lang.VirtualThread` 内で以下のように構成される:

```
┌─────────────────────────────────────────────────────────────────┐
│           デフォルトスケジューラの構成                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ForkJoinPool (仮想スレッド専用)                                  │
│  ┌─────────────────────────────────────────────────────┐        │
│  │                                                     │        │
│  │  parallelism = availableProcessors()                │        │
│  │  (変更: -Djdk.virtualThreadScheduler.parallelism)   │        │
│  │                                                     │        │
│  │  maxPoolSize = min(256, 16 * parallelism)           │        │
│  │  (変更: -Djdk.virtualThreadScheduler.maxPoolSize)   │        │
│  │  ※ ピン留め時の compensation 用に上限に余裕がある     │        │
│  │                                                     │        │
│  │  asyncMode = true (FIFO スケジューリング)            │        │
│  │                                                     │        │
│  │  ┌──────────────────────────────────────────────┐   │        │
│  │  │  Worker-1       Worker-2       Worker-N      │   │        │
│  │  │  ┌──────────┐  ┌──────────┐  ┌──────────┐   │   │        │
│  │  │  │ Deque    │  │ Deque    │  │ Deque    │   │   │        │
│  │  │  │ [VT-a]   │  │ [VT-c]   │  │ [VT-e]   │   │   │        │
│  │  │  │ [VT-b]   │  │ [VT-d]   │  │ [VT-f]   │   │   │        │
│  │  │  └──────────┘  └──────────┘  └──────────┘   │   │        │
│  │  └──────────────────────────────────────────────┘   │        │
│  │              ↕ ワークスティーリング ↕                 │        │
│  │                                                     │        │
│  │  Submission Queue (外部からの投入用)                  │        │
│  │  ┌──────────────────────────────────────────────┐   │        │
│  │  │ [VT-x] [VT-y] [VT-z] ...                    │   │        │
│  │  └──────────────────────────────────────────────┘   │        │
│  └─────────────────────────────────────────────────────┘        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

主要なシステムプロパティ:

| プロパティ | デフォルト値 | 説明 |
|---|---|---|
| `jdk.virtualThreadScheduler.parallelism` | `availableProcessors()` | キャリアスレッド数 |
| `jdk.virtualThreadScheduler.maxPoolSize` | `min(256, 16 * parallelism)` | 最大プールサイズ |

### 5.2 ワークスティーリング

`ForkJoinPool` のワークスティーリングは仮想スレッドのスケジューリングにおいて重要な役割を果たす。

```
┌─────────────────────────────────────────────────────────────────┐
│              ワークスティーリングの動作                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Worker-1 (忙しい)         Worker-2 (暇)                        │
│  ┌────────────────┐        ┌────────────────┐                   │

│  │ ┌────────────┐ │        │                │                   │
│  │ │ VT-1 (実行)│ │        │  「暇だな...    │                   │
│  │ │ VT-5       │ │←───steal──│ Worker-1 から │                 │
│  │ │ VT-9       │ │        │  盗もう」       │                   │
│  │ │ VT-13      │ │        │                │                   │
│  │ └────────────┘ │        └────────────────┘                   │
│  └────────────────┘                                             │
│                                                                 │
│  ステップ 1: Worker-2 は自分のキューが空であることを検知           │
│  ステップ 2: ランダムに Worker-1 を選択                           │
│  ステップ 3: Worker-1 の Deque の末尾から VT-13 をスティール       │
│  ステップ 4: Worker-2 が VT-13 をマウントして実行開始              │
│                                                                 │
│  ※ FIFO モード (asyncMode=true) により、                         │
│    submit された順序で処理される公平性を保つ                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.3 ブロッキング操作時の動作

仮想スレッドがブロッキング操作を実行すると、JDK の内部実装がこれを検知し、自動的にアンマウントを行う。この仕組みは JDK のライブラリ層（`java.base` モジュール）に組み込まれている。

```
┌─────────────────────────────────────────────────────────────────┐
│        ブロッキング操作時の処理フロー                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【例: Socket.read() 呼び出し時】                                │
│                                                                 │
│  ユーザーコード                                                  │
│  ──────────                                                     │
│  byte[] data = socket.getInputStream().read();                  │
│      │                                                          │
│  JDK ライブラリ層 (java.base)                                    │
│  ─────────────────────────                                      │
│      └→ sun.nio.ch.NioSocketImpl.read()                        │
│            │                                                    │
│            ├→ データ即座に利用可能？                              │
│            │   Yes → そのまま返却 (アンマウントなし)              │
│            │   No  ↓                                            │
│            │                                                    │
│            ├→ ソケットを non-blocking モードに設定                │
│            │                                                    │
│            ├→ I/O イベントをポーラー (Poller) に登録             │
│            │   ┌──────────────────────────────┐                 │
│            │   │ Poller (epoll / kqueue)       │                 │
│            │   │ ┌────────────────────────────┐│                 │
│            │   │ │ fd=42 → VT-1 (unpark)     ││                 │
│            │   │ │ fd=57 → VT-3 (unpark)     ││                 │
│            │   │ └────────────────────────────┘│                 │
│            │   └──────────────────────────────┘                 │
│            │                                                    │
│            └→ LockSupport.park()                                │
│                  │                                              │
│                  └→ Continuation.yield()                        │
│                       │                                         │
│                       └→ VT-1 アンマウント                      │
│                          キャリアスレッドは別の VT を実行          │
│                                                                 │
│  ...時間経過...                                                  │
│                                                                 │
│  Poller: fd=42 に読み取り可能なデータ到着                         │
│      │                                                          │
│      └→ LockSupport.unpark(VT-1)                               │
│            │                                                    │
│            └→ VT-1 がスケジューラのキューに投入                   │
│                 │                                               │
│                 └→ キャリアスレッドにマウントされ実行再開          │
│                      │                                          │
│                      └→ read() から戻り値を返却                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Poller（ポーラー）** は JDK 内部で OS 固有の I/O 多重化 API を使って実装されている:

| OS | 使用される I/O 多重化 API |
|---|---|
| Linux | `epoll` |
| macOS | `kqueue` |
| Windows | `IOCP` (I/O Completion Ports) |

これにより、仮想スレッドは **ユーザーには同期 I/O に見えるが、内部では非同期 I/O として処理される**。開発者は非同期プログラミングの複雑さを意識する必要がない。

### 5.4 カスタムスケジューラ

仮想スレッドは通常のデフォルトスケジューラ (`ForkJoinPool`) を使うが、`Thread.Builder` には現時点でカスタムスケジューラを指定する公開 API は存在しない。ただし、内部的には `Executor` を受け取る仕組みがあり、将来的にカスタムスケジューラのサポートが検討される可能性がある。

## 第6章: Pinning（ピン留め）問題

### 6.1 ピン留めとは何か

**Pinning（ピン留め）** とは、仮想スレッドがキャリアスレッドから**アンマウントできない状態**のことを指す。ピン留めが発生すると、仮想スレッドがブロッキング操作を行っても、キャリアスレッドごとブロックされてしまう。

```
┌─────────────────────────────────────────────────────────────────┐
│                  ピン留めの問題                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【正常動作（ピン留めなし）】                                     │
│                                                                 │
│  CT-1: [VT-1 実行]→[VT-1 アンマウント]→[VT-2 マウント]→...      │
│  CT-2: [VT-3 実行]→[VT-3 アンマウント]→[VT-4 マウント]→...      │
│                                                                 │
│  → キャリアスレッドは常に有効活用されている                        │
│                                                                 │
│  【ピン留め発生時】                                               │
│                                                                 │
│  CT-1: [VT-1 pinned: synchronized 内で I/O 待ち...........] ← 浪費│
│  CT-2: [VT-3 実行]→[VT-3 アンマウント]→[VT-4 マウント]→...      │
│                                                                 │
│  → CT-1 が長時間ブロックされ、他の仮想スレッドが実行できない       │
│  → キャリアスレッドが枯渇すると、全体のスループットが低下           │
│                                                                 │
│  ※ ピン留め中は "compensation" として追加のキャリアスレッドが      │
│    生成されることがあるが、OS スレッド数の増加を招く                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 synchronized によるピン留め

**Java 23 以前** では、仮想スレッドが `synchronized` ブロックまたはメソッド内でブロッキング操作を行うと、ピン留めが発生した。これは `synchronized` が HotSpot 内部のモニター (ObjectMonitor) と密接に結合しており、Continuation の yield を安全に行えなかったためである。

```java
// Java 23 以前: ピン留めが発生するパターン
synchronized (lock) {
    // この中でブロッキング I/O を実行すると
    // 仮想スレッドがピン留めされ、キャリアスレッドがブロックされる
    var data = inputStream.read();  // ← ピン留め！
}
```

### 6.3 ネイティブメソッドによるピン留め

JNI を通じたネイティブメソッドの実行中は、仮想スレッドは常にピン留めされる。これはネイティブコードが JVM のスタックフレームに依存する可能性があるため、スタックの freeze/thaw が安全に行えないことに起因する。

```
┌─────────────────────────────────────────────────────────────────┐
│             ネイティブメソッドによるピン留め                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  仮想スレッドのスタック:                                          │
│                                                                 │
│  ┌──────────────────────┐                                       │
│  │ Java Frame: main()   │ ← freeze 可能                        │
│  │ Java Frame: process()│ ← freeze 可能                        │
│  │ JNI Frame: native()  │ ← freeze 不可！ネイティブコードが      │
│  │                      │   スタックポインタを直接参照する        │
│  │                      │   可能性がある                         │
│  └──────────────────────┘                                       │
│                                                                 │
│  ネイティブメソッドが完了するまで、                                │
│  仮想スレッドはキャリアスレッドにピン留めされる                     │
│                                                                 │
│  ※ このピン留めは Java 25 時点でも解消されていない                 │
│    (本質的にネイティブコードの安全性を JVM が保証できないため)       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.4 JEP 491: synchronized のピン留め解消 (Java 24)

**JEP 491 (Virtual Threads: Synchronize Without Pinning)** は Java 24 で正式導入され、`synchronized` によるピン留めを解消した。これは Project Loom における重要なマイルストーンである。

JEP 491 の実装では、HotSpot の ObjectMonitor が改修され、仮想スレッドが `synchronized` ブロック内でブロックする際にも、Continuation の yield が可能になった。

```
┌─────────────────────────────────────────────────────────────────┐
│         JEP 491 による synchronized ピン留め解消                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【Java 23 以前の動作】                                          │
│                                                                 │
│  synchronized (lock) {       ObjectMonitor                      │
│      read(socket);  ──────→  モニターが VT のアンマウントを       │
│  }                           ブロック → CT ごとブロック！          │
│                                                                 │
│  【Java 24 以降 (JEP 491)】                                     │
│                                                                 │
│  synchronized (lock) {       ObjectMonitor (改修版)              │
│      read(socket);  ──────→  VT はモニターの所有権を保持した      │
│  }                           まま、安全にアンマウント可能          │
│                              → CT は別の VT を実行！              │
│                                                                 │
│  HotSpot C++ 層の変更:                                           │
│  - ObjectMonitor が仮想スレッドの Continuation を認識              │
│  - モニター所有権が VT に紐づく (CT ではなく)                     │
│  - freeze 時にモニター状態を正しく保存/復元                       │
│                                                                 │
│  関連ソース:                                                     │
│  - src/hotspot/share/runtime/objectMonitor.cpp                  │
│  - src/hotspot/share/runtime/objectMonitor.hpp                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

JEP 491 の恩恵:
- **既存の `synchronized` コードをそのまま仮想スレッドで使える**: `ReentrantLock` への書き換えが不要に
- **サードパーティライブラリの `synchronized` も問題にならない**: JDBC ドライバ等の内部で使われる `synchronized` も安全に
- **migration のハードルが大幅に低下**: 既存アプリケーションを仮想スレッドに移行しやすくなった

### 6.5 ピン留めの検出方法

ピン留めが発生しているかどうかを検出するための手段:

**1. JFR (Java Flight Recorder) イベント:**

```
jdk.VirtualThreadPinned イベント:
- duration: ピン留め期間
- stackTrace: ピン留めが発生したスタックトレース
```

```bash
# JFR でピン留めイベントを記録
java -XX:StartFlightRecording=filename=recording.jfr,settings=profile ...
```

**2. システムプロパティ (デバッグ用):**

```bash
# ピン留め発生時にスタックトレースを出力 (Java 21-23)
java -Djdk.tracePinnedThreads=full ...

# 短縮版 (フレーム数を制限)
java -Djdk.tracePinnedThreads=short ...
```

> **注意:** Java 24 以降では `synchronized` によるピン留めが解消されたため、`jdk.tracePinnedThreads` で検出されるのは主にネイティブメソッドによるピン留めとなる。

**3. スレッドダンプ:**

```bash
# JSON 形式のスレッドダンプ (仮想スレッドも含む)
jcmd <pid> Thread.dump_to_file -format=json threads.json
```

## 第7章: 仮想スレッドの正しい使い方

### 7.1 基本的な生成方法

```java
// 方法 1: Thread.ofVirtual() ビルダー
Thread vt = Thread.ofVirtual()
    .name("my-virtual-thread")
    .start(() -> {
        IO.println("仮想スレッドで実行中");
    });

// 方法 2: Thread.startVirtualThread() (簡易版)
Thread vt = Thread.startVirtualThread(() -> {
    IO.println("仮想スレッドで実行中");
});

// 方法 3: Executors.newVirtualThreadPerTaskExecutor() (推奨)
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10_000; i++) {
        executor.submit(() -> {
            // 各タスクが個別の仮想スレッドで実行される
            var result = callRemoteService();
            processResult(result);
        });
    }
} // try-with-resources で自動的に shutdown & awaitTermination
```

`Executors.newVirtualThreadPerTaskExecutor()` が最も推奨される方法である。理由:
- `try-with-resources` でライフサイクル管理が容易
- `submit()` で `Future` を取得でき、例外処理も容易
- タスクごとに新しい仮想スレッドが生成される（プーリングしない）

### 7.2 やるべきこと / やってはいけないこと

```
┌─────────────────────────────────────────────────────────────────┐
│              やるべきこと / やってはいけないこと                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ✓ やるべきこと (Do)                                             │
│  ────────────────                                               │
│  ✓ I/O バウンドなタスクに仮想スレッドを使う                       │
│  ✓ thread-per-request スタイルで素直な同期コードを書く             │
│  ✓ Executors.newVirtualThreadPerTaskExecutor() を使う            │
│  ✓ タスクごとに仮想スレッドを使い捨てる（短命）                    │
│  ✓ ReentrantLock を使う（Java 23 以前で synchronized を避ける場合）│
│  ✓ try-with-resources で ExecutorService を管理する              │
│  ✓ Semaphore 等でリソースアクセスを制限する（DB 接続数等）         │
│                                                                 │
│  ✗ やってはいけないこと (Don't)                                   │
│  ──────────────────────                                         │
│  ✗ 仮想スレッドをプーリングしない（スレッドプールに入れない）       │
│  ✗ CPU バウンドなタスクに仮想スレッドを使わない                   │
│  ✗ ThreadLocal に大量のデータを格納しない                         │
│  ✗ 仮想スレッドに優先度を設定しても意味がない                     │
│  ✗ Thread.yield() を busy-wait のように使わない                  │
│  ✗ スレッド数を人為的に制限しない（スレッド数は問題ではない）       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 7.3 ThreadLocal の注意点

仮想スレッドは `ThreadLocal` をサポートするが、以下の問題がある:

1. **メモリ消費**: 100万の仮想スレッドがそれぞれ `ThreadLocal` を持つと、メモリ消費が膨大になる
2. **ライフサイクル**: 仮想スレッドは短命であることが多いが、`ThreadLocal` の初期化/クリーンアップのオーバーヘッドは残る
3. **継承**: `InheritableThreadLocal` は親から子への値コピーが発生し、仮想スレッドの大量生成時に問題になり得る

```java
// 問題: ThreadLocal + 仮想スレッド
static final ThreadLocal<Connection> connHolder = new ThreadLocal<>();

// 100万の仮想スレッドがそれぞれ Connection を保持
// → メモリ消費が膨大、DB 接続数も爆発する可能性

// 解決策 1: Semaphore で並行数を制限
static final Semaphore dbPermits = new Semaphore(50);

void handleRequest() {
    dbPermits.acquire();
    try {
        var conn = dataSource.getConnection();
        // ...
    } finally {
        dbPermits.release();
    }
}

// 解決策 2: Scoped Values を使う (Java 25 で正式化)
// → 第9章で解説
```

### 7.4 プーリングしてはいけない理由

仮想スレッドをスレッドプールに入れてはいけない。これは仮想スレッドの設計思想に反する。

```
┌─────────────────────────────────────────────────────────────────┐
│            プーリングが不要な理由                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  プラットフォームスレッドをプーリングする理由:                      │
│  - 生成コストが高い (数十〜数百μs)                               │
│  - メモリコストが高い (~1MB/スレッド)                             │
│  - OS リソースを消費する                                         │
│  → 再利用してコストを分散する必要がある                           │
│                                                                 │
│  仮想スレッドにはこれらの問題がない:                               │
│  - 生成コストが極めて低い (~1μs)                                 │
│  - メモリコストが極めて低い (数百バイト〜数KB)                     │
│  - OS リソースを消費しない                                       │
│  → 使い捨てて良い。再利用する理由がない。                         │
│                                                                 │
│  // NG: プーリング（意味がない上に、スループットを制限する）        │
│  var pool = Executors.newFixedThreadPool(100,                    │
│      Thread.ofVirtual().factory());                              │
│                                                                 │
│  // OK: タスクごとに新しい仮想スレッド                            │
│  var executor = Executors.newVirtualThreadPerTaskExecutor();     │
│                                                                 │
│  プーリングが有害な理由:                                          │
│  - スレッドプールのサイズがボトルネックになる                      │
│  - 仮想スレッドの利点（大量並行実行）が失われる                    │
│  - ThreadLocal のライフサイクルが不明確になる                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 7.5 synchronized を ReentrantLock に置き換えるべきケース

**Java 24 以降（JEP 491 適用後）では、`synchronized` のままで問題ない。** しかし Java 23 以前のランタイムをサポートする必要がある場合は、ピン留めが発生するブロッキング操作を含む `synchronized` ブロックを `ReentrantLock` に置き換えることを検討すべきである。

```java
// Java 23 以前で仮想スレッドを使う場合の書き換え例

// Before: synchronized (ピン留めの可能性)
synchronized (lock) {
    var data = inputStream.read();
    process(data);
}

// After: ReentrantLock (ピン留めなし)
var reentrantLock = new ReentrantLock();
reentrantLock.lock();
try {
    var data = inputStream.read();
    process(data);
} finally {
    reentrantLock.unlock();
}
```

Java 25 (Azul Zulu 25) では JEP 491 が適用されているため、`synchronized` のまま安全に仮想スレッドを使用できる。

## 第8章: Structured Concurrency

### 8.1 並行処理の構造化という概念

**Structured Concurrency（構造化並行処理）** とは、並行タスクのライフサイクルをコードの構造（ブロックスコープ）に結びつける設計パターンである。`go` 文やスレッドプールでタスクを「投げっぱなし」にする非構造化並行処理と対比される。

```
┌─────────────────────────────────────────────────────────────────┐
│     非構造化並行処理 vs 構造化並行処理                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【非構造化並行処理 (Unstructured Concurrency)】                  │
│                                                                 │
│  void handleRequest() {                                         │
│      Future<User> f1 = executor.submit(() -> fetchUser());      │
│      Future<Order> f2 = executor.submit(() -> fetchOrder());    │
│      // f1 が例外で失敗しても f2 は実行され続ける                  │
│      // タスクのキャンセルは手動で管理する必要がある                │
│      // スレッドダンプに親子関係が見えない                         │
│  }                                                              │
│                                                                 │
│  タスクのライフサイクル:                                          │
│  handleRequest() ─┬─→ fetchUser()  ───────→ (いつ終わる？)      │
│                   └─→ fetchOrder() ───────→ (いつ終わる？)      │
│                                                                 │
│  ────────────────────────────────────────────                   │
│                                                                 │
│  【構造化並行処理 (Structured Concurrency)】                      │
│                                                                 │
│  void handleRequest() throws Exception {                        │
│      try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
│          var f1 = scope.fork(() -> fetchUser());                │
│          var f2 = scope.fork(() -> fetchOrder());               │
│          scope.join();           // 全サブタスクの完了を待つ      │
│          scope.throwIfFailed();  // いずれかが失敗したら例外      │
│          return process(f1.get(), f2.get());                    │
│      } // スコープ終了時にすべてのサブタスクが完了している          │
│  }                                                              │
│                                                                 │
│  タスクのライフサイクル:                                          │
│  handleRequest() ─┬─→ fetchUser()  ──┐                          │
│                   └─→ fetchOrder() ──┤                          │
│                                      ├─→ join() → 合流          │
│                                      │                          │
│  すべてのタスクが handleRequest() のスコープ内で開始し完了する      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

構造化並行処理の利点:
- **エラー伝播**: サブタスクの例外が親に自動的に伝播する
- **キャンセル**: 一つのサブタスクが失敗したら、他のサブタスクを自動キャンセルできる
- **可観測性**: スレッドダンプに親子関係が表示される
- **リソースリーク防止**: `try-with-resources` でタスクのライフサイクルが保証される

### 8.3 StructuredTaskScope の使い方

> **注意:** 以下のコード例は Java 25 (JEP 505) 時点の API に基づく。

`StructuredTaskScope` は構造化並行処理の中核クラスであり、サブタスクの fork（分岐）と join（合流）を管理する。

```java
// StructuredTaskScope の基本的な使い方 (Java 25 で正式化)

import java.util.concurrent.StructuredTaskScope;

User fetchUser(long id) throws Exception {
    try (var scope = StructuredTaskScope.open()) {
        // サブタスクを fork
        var userTask = scope.fork(() -> userService.findById(id));
        var profileTask = scope.fork(() -> profileService.getProfile(id));

        // すべてのサブタスクの完了を待つ
        scope.join();

        // 結果を取得
        return new User(userTask.get(), profileTask.get());
    }
    // スコープを抜けるとすべてのサブタスクは完了済みであることが保証される
}
```

### 8.4 ShutdownOnFailure / ShutdownOnSuccess パターン

JEP 499 (Java 24, Fourth Preview) では、`Joiner` を使ってタスクの完了ポリシーを構成する。

```java
// ShutdownOnFailure: いずれかのタスクが失敗したら全体を中断
// (Java 24 Fourth Preview の Joiner API)

record UserOrder(User user, Order order) {}

UserOrder fetchUserAndOrder(long userId) throws Exception {
    try (var scope = StructuredTaskScope.open(
            StructuredTaskScope.Joiner.awaitAllSuccessfulOrThrow())) {

        var userTask = scope.fork(() -> userService.findById(userId));
        var orderTask = scope.fork(() -> orderService.getLatest(userId));

        scope.join();  // いずれかが失敗したら例外がスローされる
                       // 失敗したタスクの例外が伝播し、
                       // 残りのタスクはキャンセルされる

        return new UserOrder(userTask.get(), orderTask.get());
    }
}

// ShutdownOnSuccess: 最初に成功したタスクの結果を使う
// (レース: 複数のソースから最速の応答を採用)

String fetchFromFastestSource(String key) throws Exception {
    try (var scope = StructuredTaskScope.open(
            StructuredTaskScope.Joiner.anySuccessfulResultOrThrow())) {

        scope.fork(() -> cache.get(key));        // キャッシュ
        scope.fork(() -> database.get(key));     // DB
        scope.fork(() -> remoteService.get(key)); // リモート

        return scope.join();  // 最初に成功した結果が返される
                              // 残りのタスクはキャンセルされる
    }
}
```

## 第9章: Scoped Values

### 9.1 ThreadLocal の問題点

`ThreadLocal` は Java 1.2 から存在するスレッドごとのデータ共有メカニズムだが、仮想スレッド時代には以下の問題がある:

| 問題 | 説明 |
|---|---|
| **ミュータブル** | `set()` でいつでも書き換え可能。意図しない変更のリスク |
| **無制限のライフサイクル** | スレッドが生きている限りデータが保持される。メモリリークの原因 |
| **継承コスト** | `InheritableThreadLocal` は子スレッド生成時にコピーが発生 |
| **メモリ消費** | 100万仮想スレッド × ThreadLocal = 大量のメモリ消費 |
| **スコープの不在** | データの有効範囲がコードの構造と無関係 |

### 9.2 JEP の変遷と Java 25 時点の状態

| JEP | Java バージョン | ステータス | 主な変更点 |
|---|---|---|---|
| JEP 429 | Java 20 | Incubator | 初回導入 (`jdk.incubator.concurrent.ScopedValue`) |
| JEP 446 | Java 21 | Preview | `java.lang.ScopedValue` に移動 |
| JEP 464 | Java 22 | Second Preview | 変更なし (フィードバック収集) |
| JEP 481 | Java 23 | Third Preview | API 改善 |
| JEP 487 | Java 24 | Fourth Preview | API 洗練 |
| JEP 502 | Java 25 | **正式 (GA)** | 正式化 |

> **補足:** Scoped Values は Java 25 で **JEP 502 として正式化**された。`--enable-preview` フラグは不要であり、本番環境で利用可能である。

### 9.3 ScopedValue の使い方

```java
// ScopedValue の基本的な使い方 (Java 25 で正式化)

import java.lang.ScopedValue;

// ScopedValue は final static フィールドとして宣言
static final ScopedValue<User> CURRENT_USER = ScopedValue.newInstance();

void handleRequest(Request request) {
    User user = authenticate(request);

    // ScopedValue.where() でスコープ付きの値を設定し、run() で実行
    ScopedValue.where(CURRENT_USER, user)
        .run(() -> {
            // このブロック内（および呼び出し先のメソッド）で
            // CURRENT_USER.get() が有効
            processOrder();
            sendNotification();
        });
    // ブロックを抜けると値は自動的に無効になる
}

void processOrder() {
    // 引数として渡さなくても、呼び出し元のスコープから値を取得
    User user = CURRENT_USER.get();
    IO.println("Processing order for: " + user.name());
}

void sendNotification() {
    // 同じスコープ内であれば、どこからでも取得可能
    User user = CURRENT_USER.get();
    IO.println("Sending notification to: " + user.email());
}
```

### 9.4 ThreadLocal との比較

```
┌─────────────────────────────────────────────────────────────────┐
│           ThreadLocal vs ScopedValue                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ThreadLocal:                                                   │
│  ┌──────────────────────────────────────────────┐               │
│  │ Thread                                       │               │
│  │ ┌──────────────────────────────────────────┐ │               │
│  │ │ ThreadLocalMap                           │ │               │
│  │ │ ┌────────────────────────────────────┐   │ │               │
│  │ │ │ key=TL_USER  → value=User("Bob")  │   │ │               │

│  │ │ │ key=TL_TRACE → value="abc-123"    │   │ │               │

│  │ └──────────────────────────────────────────┘ │               │

│                                                                 │
│  ScopedValue:                                                   │
│  ┌──────────────────────────────────────────────┐               │
│  │ ScopedValue.where(USER, user).run(() -> {    │               │
│  │ ┌────────────────────────────────────────┐   │               │

│  │ │  USER = user (不変, このスコープ内のみ)  │   │               │
│  │ │  ┌────────────────────────────────┐     │   │               │
```

| 特性 | ThreadLocal | ScopedValue |
|---|---|---|
| **可変性** | ミュータブル (`set()` で変更可能) | **イミュータブル** (スコープ内で固定) |
| **ライフサイクル** | スレッド寿命 (手動 `remove()` 必要) | **スコープに束縛** (自動解放) |
| **継承** | `InheritableThreadLocal` (コピー) | **Structured Concurrency と統合** (コピー不要) |
| **パフォーマンス** | ハッシュマップ探索 | **コンパイラ最適化可能** |
| **メモリリスク** | リークしやすい | **リークしない** |
| **仮想スレッドとの相性** | 悪い (100万スレッドで問題) | **良い** (設計がマッチ) |

### 9.5 Structured Concurrency との統合

`ScopedValue` は `StructuredTaskScope` と自然に統合される。親スコープで設定した `ScopedValue` は、fork されたサブタスク（子仮想スレッド）から自動的に参照可能である。

```java
// ScopedValue + StructuredTaskScope の統合 (Java 25 で正式化)

static final ScopedValue<String> REQUEST_ID = ScopedValue.newInstance();

void handleRequest(String reqId) throws Exception {
    ScopedValue.where(REQUEST_ID, reqId)
        .run(() -> {
            try (var scope = StructuredTaskScope.open()) {
                // fork されたサブタスクから REQUEST_ID が参照可能
                scope.fork(() -> {
                    // ThreadLocal と違い、値のコピーは発生しない
                    IO.println("Task A, reqId=" + REQUEST_ID.get());
                    return fetchUser();
                });
                scope.fork(() -> {
                    IO.println("Task B, reqId=" + REQUEST_ID.get());
                    return fetchOrder();
                });
                scope.join();
            }
        });
}
```

`InheritableThreadLocal` との重要な違い: `InheritableThreadLocal` は子スレッド生成時に値を**コピー**する。100万の仮想スレッドを fork する場合、100万回のコピーが発生する。`ScopedValue` はコピーを行わず、**スコープチェーンを辿って値を参照する**ため、はるかに効率的である。

```
│  0K  │  ╚═══╝   ╚═════╝   ╚═════╝   ╚═════╝                     │
│      └───────────────────────────────────────→                  │
│       CPU 集約   混合       I/O 集約  高並行 I/O                  │
│       (計算)    (DB+計算)  (HTTP 呼出) (マイクロSV)              │
│                                                                 │
│  PT = プラットフォームスレッド (200スレッドプール)                  │
│  VT = 仮想スレッド (無制限)                                      │
│                                                                 │
│  ポイント:                                                       │
│  - CPU 集約: VT と PT の差は小さい (ボトルネックは CPU)            │
│  - I/O 集約: VT が圧倒的に有利 (PT はスレッド枯渇が発生)          │
│  - 高並行 I/O: VT の利点が最大化 (数万同時接続)                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 10.4 GC への影響

仮想スレッドのスタックチャンクは Java ヒープ上にあるため、GC との相互作用がある:

| GC への影響 | 詳細 |
|---|---|
| **ヒープ使用量の増加** | 大量の仮想スレッドのスタックチャンクがヒープを消費する |
| **GC ルートの増加** | アクティブな仮想スレッドのスタックチャンクは GC ルートとして扱われる |
| **STW 時間への影響** | 多数のスタックチャンクの走査が STW 時間をわずかに増加させる可能性がある |
| **GC フレンドリー** | 休止中の仮想スレッドのスタックチャンクはヒープ上の通常オブジェクトとして GC が効率的に処理できる |

G1GC / ZGC ともにスタックチャンクの走査を最適化しており、通常の使用では GC への影響は軽微である。ただし、数百万の仮想スレッドを同時に保持する場合は、ヒープサイズとGC チューニングに注意を払う必要がある。

### 10.5 仮想スレッドが有利でないケース

| ケース | 理由 |
|---|---|
| **CPU バウンドなタスク** | 仮想スレッドの利点はブロッキング操作の多重化。CPU を使い切っている場合はプラットフォームスレッドと差がない |
| **短期間の非ブロッキング処理** | マウント/アンマウントのオーバーヘッドが処理時間に対して相対的に大きい |
| **既に十分なスレッドプール** | 同時接続数が少なく、プラットフォームスレッドプールで十分対応できる場合 |
| **リアルタイム性が必要** | 仮想スレッドのスケジューリングは OS スケジューラほど厳密な優先度制御ができない |

## 第11章: 実務での活用

### 11.1 Spring Boot での対応状況

Spring Boot 3.2 以降で仮想スレッドのサポートが追加された。設定は極めてシンプルである:

```properties
# application.properties
spring.threads.virtual.enabled=true
```

この設定により、以下が仮想スレッドで実行される:

| コンポーネント | 仮想スレッド化 |
|---|---|
| Tomcat のリクエスト処理 | 各リクエストが仮想スレッドで処理される |
| `@Async` メソッド | 仮想スレッド上で実行される |
| Spring MVC のコントローラ | 仮想スレッドで呼び出される |
| `@Scheduled` タスク | 仮想スレッドで実行される |

```
┌─────────────────────────────────────────────────────────────────┐
│         Spring Boot + Virtual Threads のアーキテクチャ             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【従来 (spring.threads.virtual.enabled=false)】                 │
│                                                                 │
│  HTTP Request → Tomcat Thread Pool (200) → Controller           │
│                 ┌──────────────────┐      → Service             │
│                 │ PT-1, PT-2, ...  │      → Repository          │
│                 │ PT-200           │      → DB (JDBC)           │
│                 └──────────────────┘                            │
│                 200以上の同時リクエストは待機                      │
│                                                                 │
│  【仮想スレッド有効時 (spring.threads.virtual.enabled=true)】     │
│                                                                 │
│  HTTP Request → VirtualThreadPerTaskExecutor → Controller       │
│                 ┌──────────────────────────┐  → Service         │
│                 │ VT-1, VT-2, ...          │  → Repository      │
│                 │ VT-10000, VT-10001, ...  │  → DB (JDBC)       │
│                 └──────────────────────────┘                    │
│                 同時リクエスト数に上限なし                         │
│                 (ただし DB 接続プール等は別途制限が必要)            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> **注意:** 仮想スレッドを有効にしても、DB 接続プール (HikariCP 等) のサイズは依然としてボトルネックになり得る。Semaphore やコネクションプールの設定で適切に制限する必要がある。

### 11.2 Quarkus / Helidon での対応状況

**Quarkus:**
- Quarkus 3.x 以降で `@RunOnVirtualThread` アノテーションをサポート
- RESTEasy Reactive と組み合わせて、エンドポイント単位で仮想スレッドを指定可能
- Quarkus はリアクティブ基盤 (Vert.x) 上に構築されているため、仮想スレッドはリアクティブとの共存手段として提供される

**Helidon:**
- Helidon 4.x は仮想スレッドを前提として再設計された
- Helidon Nima (Helidon 4 の Web サーバー) は非リアクティブな設計で、仮想スレッドを全面活用
- thread-per-request モデルに回帰し、リアクティブプログラミングの複雑さを排除

### 11.3 JDBC / HTTP Client での活用

**JDBC:**

JDBC のブロッキング API は仮想スレッドと相性が良い。`Connection.prepareStatement()`, `ResultSet.next()` 等のブロッキング呼び出しで仮想スレッドが自動的にアンマウントされる。

```java
// JDBC + 仮想スレッド
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    var futures = orders.stream()
        .map(order -> executor.submit(() -> {
            // 各注文を個別の仮想スレッドで処理
            try (var conn = dataSource.getConnection();
                 var stmt = conn.prepareStatement("SELECT ...")) {
                stmt.setLong(1, order.id());
                var rs = stmt.executeQuery();  // ← アンマウントされる
                return mapResult(rs);
            }
        }))
        .toList();

    // 結果の収集
    for (var future : futures) {
        var result = future.get();
        process(result);
    }
}
```

**java.net.http.HttpClient:**

Java 11 で導入された `HttpClient` は仮想スレッド対応済み。同期 API (`send()`) を仮想スレッドから呼び出すと、内部で適切にアンマウントされる。

```java
// HttpClient + 仮想スレッド
var client = HttpClient.newHttpClient();

try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    var futures = urls.stream()
        .map(url -> executor.submit(() -> {
            var request = HttpRequest.newBuilder(URI.create(url)).build();
            // send() はブロッキングだが、仮想スレッドではアンマウントされる
            return client.send(request, HttpResponse.BodyHandlers.ofString());
        }))
        .toList();

    for (var future : futures) {
        HttpResponse<String> response = future.get();
        IO.println(response.statusCode() + ": " + response.uri());
    }
}
```

### Zulu との差分

Azul Zulu (OSS 版) は OpenJDK mainline のビルドであり、仮想スレッドの実装は HotSpot そのものである。Zulu 25 で仮想スレッドを使用した場合、OpenJDK 25 と完全に同一の動作をする。

ただし、Azul の商用製品 **Azul Platform Prime（旧 Zing）** の環境では以下の点に注意:

| 比較項目 | OpenJDK (Zulu) Virtual Threads | Azul Platform Prime |
|---|---|---|
| **仮想スレッド** | JEP 444 準拠の標準実装 | JEP 444 準拠 (Prime は OpenJDK ベース) |
| **JIT コンパイラ** | C2 | Falcon JIT (LLVM ベース) |
| **GC** | G1GC / ZGC | C4 GC (独自実装) |
| **スタックチャンクの GC 処理** | G1GC / ZGC が走査 | C4 GC が走査 (低レイテンシ) |
| **ピン留め解消 (JEP 491)** | Java 24+ で対応 | Java 24+ で対応 |

Prime の C4 GC は超低レイテンシの GC であるため、大量の仮想スレッドのスタックチャンクが存在する場合でも、GC 停止時間への影響が OpenJDK の G1GC より小さい可能性がある。
