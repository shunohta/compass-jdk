## 1. JVM とは何か

### 1.1 「Write Once, Run Anywhere」を実現する仮想マシン

JVM（Java Virtual Machine）は、**Java プログラムを実行するための仮想的なコンピューター**である。

通常、C や C++ で書いたプログラムは、コンパイルした OS・CPU 向けの機械語にしかならない。Windows 用にコンパイルしたプログラムは macOS では動かないし、x86 用のバイナリは ARM では動かない。

Java はこの問題を **「中間言語」＋「仮想マシン」** という仕組みで解決した。

```
■ C/C++ の場合（プラットフォーム依存）

  ソースコード (.c)
       │
       ▼ コンパイル
  機械語 (Windows/x86 用)  ← Windows でしか動かない
  機械語 (macOS/ARM 用)    ← macOS でしか動かない
  機械語 (Linux/x86 用)    ← Linux でしか動かない


■ Java の場合（プラットフォーム非依存）

  ソースコード (.java)
       │
       ▼ javac でコンパイル
  バイトコード (.class)  ← どの OS でも同じファイル
       │
       ├──→ JVM (Windows 版) → Windows で実行
       ├──→ JVM (macOS 版)   → macOS で実行
       └──→ JVM (Linux 版)   → Linux で実行
```

ポイントは **バイトコード (.class ファイル) はどの環境でも同一** であること。環境の差異は、各プラットフォーム向けに用意された JVM が吸収する。これが「Write Once, Run Anywhere（一度書けば、どこでも動く）」の本質である。

### 1.2 JVM 仕様と実装の関係

JVM には **仕様 (Specification)** と **実装 (Implementation)** がある。JVM 仕様は「JVM はこう振る舞うべきだ」というルールブックであり、実装は「そのルールに従って実際に動くプログラム」である。

我々が使っている **HotSpot VM** は、世界で最も広く使われている JVM 実装であり、C++ で書かれている。HotSpot の詳細（名前の由来、JIT コンパイラの仕組み、各ディストリビューションとの関係など）については [memory-basics.md の「HotSpot とは何か」セクション](memory/memory-basics.md#hotspot-とは何か----この学習シリーズの主役を知る) で詳しく解説している。

### 1.3 なぜ「仮想マシン」と呼ぶのか

「仮想マシン」という名前は大げさに聞こえるかもしれないが、JVM は本物のコンピューターと同じ要素を持っている。

| 本物のコンピューター | JVM（仮想マシン） |
|---------------------|-------------------|
| CPU | 実行エンジン（インタプリタ + JIT コンパイラ） |
| RAM（メモリ） | ランタイムデータエリア（ヒープ、スタック等） |
| ディスクからプログラムを読み込む | クラスローダーが .class を読み込む |
| 機械語を実行 | バイトコードを実行 |
| OS がメモリを管理 | GC がメモリを管理 |

つまり JVM は **ソフトウェアで実装されたコンピューター** であり、バイトコードという「仮想的な機械語」を実行する。だから Virtual Machine（仮想マシン）と呼ぶ。

## 2. JVM の全体構造 -- 主要コンポーネント

### 2.1 全体構造図

Java ソースコードがどのように実行されるかを、JVM の内部構造と合わせて示す。

```
  Hello.java (ソースコード)
       │
       ▼  javac (Java コンパイラ) ← JVM の「外」で動作
  Hello.class (バイトコード)
       │
       ▼
 ┌─────────────────────────────────────────────────────────────────┐
 │                        JVM (HotSpot VM)                        │
 │                                                                │
 │  ┌──────────────────────────────────────────────────────────┐  │
 │  │          クラスローダーサブシステム                        │  │
 │  │  ┌──────────────┐ ┌──────────┐ ┌──────────────────────┐  │  │
 │  │  │   ロード     │→│  リンク  │→│     初期化           │  │  │
 │  │  │ (.class 読込) │ │(検証/準備)│ │(static 実行)         │  │  │
 │  │  └──────────────┘ └──────────┘ └──────────────────────┘  │  │
 │  └──────────────────────────┬───────────────────────────────┘  │
 │                             │ クラス情報を配置                  │
 │                             ▼                                  │
 │  ┌──────────────────────────────────────────────────────────┐  │
 │  │            ランタイムデータエリア (メモリ)                 │  │
 │  │                                                          │  │
 │  │   ┌────────┐ ┌────────┐ ┌──────────┐ ┌──────────────┐   │  │
 │  │   │ ヒープ │ │ スタック │ │Metaspace │ │コードキャッシュ│   │  │
 │  │   │(共有)  │ │(各ﾚｯﾄﾞ) │ │(ｸﾗｽ情報) │ │ (JIT 出力)   │   │  │
 │  │   └────────┘ └────────┘ └──────────┘ └──────────────┘   │  │
 │  │   ┌──────────────┐ ┌──────────────────────────────────┐  │  │
 │  │   │ PC レジスタ  │ │ ネイティブメソッドスタック        │  │  │
 │  │   └──────────────┘ └──────────────────────────────────┘  │  │
 │  └──────────────────────────────────────────────────────────┘  │
 │                             │ バイトコードを読み出す            │
 │                             ▼                                  │
 │  ┌──────────────────────────────────────────────────────────┐  │
 │  │              実行エンジン (Execution Engine)              │  │
 │  │                                                          │  │
 │  │   ┌──────────────┐  ┌─────────────┐  ┌──────────────┐   │  │
 │  │   │ インタプリタ │  │JIT コンパイラ│  │    GC        │   │  │
 │  │   │(逐次実行)    │  │  (C1 / C2)  │  │(メモリ回収)  │   │  │
 │  │   └──────────────┘  └─────────────┘  └──────────────┘   │  │
 │  └──────────────────────────────────────────────────────────┘  │
 │                             │                                  │
 │  ┌──────────────────────────┴───────────────────────────────┐  │
 │  │              JNI (ネイティブメソッドインタフェース)        │  │
 │  └──────────────────────────┬───────────────────────────────┘  │
 └─────────────────────────────┼──────────────────────────────────┘
                               │
                               ▼
                ネイティブメソッドライブラリ (C/C++)
                  (OS 機能, I/O, ネットワーク等)
```

### 2.2 コンポーネント一覧

JVM の主要コンポーネントを「比喩」で整理すると、次のようになる。

| コンポーネント | 比喩 | 役割 |
|--------------|------|------|
| **クラスローダー** | 「図書館の司書」 | .class ファイルを見つけて読み込み、使える状態にする |
| **ランタイムデータエリア** | 「作業場と倉庫」 | プログラム実行中のデータを保管するメモリ空間 |
| **実行エンジン** | 「CPU」 | バイトコードを解釈・実行する頭脳 |
| **JIT コンパイラ** | 「翻訳の達人」 | よく使うバイトコードを高速な機械語に変換する |
| **GC** | 「掃除係」 | 不要になったオブジェクトのメモリを自動回収する |
| **JNI** | 「通訳」 | Java と C/C++ ネイティブコードの橋渡しをする |
| **ネイティブライブラリ** | 「OS の専門家」 | ファイル I/O やネットワーク等、OS 固有の機能を提供 |

JIT コンパイラと GC は実行エンジンの **一部** として位置づけられるが、独立した大きなサブシステムであるため、別途解説する。

## 3. クラスローダーサブシステム (Class Loader Subsystem)

**対象層:** Java 層（`java.lang.ClassLoader`）/ HotSpot C++ 層（`src/hotspot/share/classfile/`）

クラスローダーは、.class ファイル（バイトコード）をディスクやネットワークから読み込み、JVM のメモリ上で使える状態にするサブシステムである。

> **詳細は [classloader-deep-dive.md](runtime/classloader-deep-dive.md) を参照。** 本セクションでは全体像の把握に必要な範囲のみ記述する。

### 3.1 クラスローダーの3階層

Java のクラスローダーは3層構造になっている。

```
  ┌─────────────────────────────────────────────────┐
  │     Bootstrap ClassLoader（起動クラスローダー）   │  ← C++ で実装
  │  java.lang.String, java.util.List 等の            │     (HotSpot C++ 層)
  │  コアライブラリ (java.base モジュール) をロード     │
  └────────────────────┬────────────────────────────┘
                       │ 親
  ┌────────────────────▼────────────────────────────┐
  │   Platform ClassLoader（プラットフォームCL）      │  ← Java で実装
  │  java.sql, java.xml 等の                         │     (Java 層)
  │  プラットフォーム拡張モジュールをロード             │
  └────────────────────┬────────────────────────────┘
                       │ 親
  ┌────────────────────▼────────────────────────────┐
  │   Application ClassLoader（アプリケーションCL）   │  ← Java で実装
  │  自分が書いたクラス、ライブラリの jar をロード      │     (Java 層)
  │  (classpath / modulepath 上のクラス)              │
  └─────────────────────────────────────────────────┘
```

**注意:** Java 8 以前は「Extension ClassLoader」という名前だったが、Java 9 のモジュールシステム（JPMS）導入に伴い「Platform ClassLoader」に改称された。

**HotSpot C++ 層との接点:** Bootstrap ClassLoader だけは例外的に Java コードではなく **HotSpot の C++ コード** で実装されている。これは JVM 自身が動くために必要な最低限のクラス（`java.lang.Object` や `java.lang.Class` 等）を、Java コードが動く前にロードしなければならないためである。

- 関連ソースパス: `src/hotspot/share/classfile/classLoader.cpp`

### 3.2 双親委譲モデル (Parent Delegation Model)

クラスローダーは、クラスのロード要求を受けると **まず親に委譲する** というルールで動作する。これを双親委譲モデル（Parent Delegation Model）と呼ぶ。

```
ロード要求: "com.example.MyService" を読み込んで

Application CL: 「親に聞いてみよう」
       │
       ▼
Platform CL: 「親に聞いてみよう」
       │
       ▼
Bootstrap CL: 「java.base にそんなクラスはない。知らない」
       │
       ▼ (見つからない)
Platform CL: 「自分の管轄にもない。知らない」
       │
       ▼ (見つからない)
Application CL: 「自分のクラスパスにあった！読み込む」
       │
       ▼
ロード完了: com.example.MyService
```

**なぜこの仕組みが必要か:**

1. **セキュリティ:** ユーザーが `java.lang.String` という名前のクラスを作っても、Bootstrap CL が先に本物の `String` を見つけるため、コアライブラリの偽装ができない
2. **一意性:** 同じクラスが複数回ロードされることを防ぎ、型の同一性を保証する
3. **分離:** 各層のクラスローダーが責任範囲を持つことで、モジュール間の境界を維持する

### 3.3 クラスのライフサイクル -- ロード・リンク・初期化

クラスがメモリ上で使える状態になるまでには、3つのフェーズがある。

```
 .class ファイル
       │
       ▼
 ┌─────────────┐
 │  1. ロード   │  バイトコードを読み込み、Class オブジェクトを生成
 │   (Loading)  │  → Metaspace にクラスのメタ情報を配置
 └──────┬──────┘
        ▼
 ┌─────────────────────────────────────────────┐
 │  2. リンク (Linking)                         │
 │  ┌──────────┐ ┌──────────┐ ┌──────────────┐ │
 │  │  検証    │→│  準備    │→│  解決        │ │
 │  │(Verify)  │ │(Prepare) │ │(Resolve)     │ │
 │  │ｸﾗｽﾌｧｲﾙの│ │static変数│ │ｼﾝﾎﾞﾙ参照を  │ │
 │  │整合性確認│ │のﾃﾞﾌｫﾙﾄ値│ │直接参照に変換│ │
 │  └──────────┘ └──────────┘ └──────────────┘ │
 └──────────────────────┬──────────────────────┘
                        ▼
 ┌────────────────────────┐
 │  3. 初期化              │  static イニシャライザ（<clinit>）を実行
 │   (Initialization)     │  static 変数に実際の値を代入
 └────────────────────────┘
```

**各フェーズの詳細:**

| フェーズ | 何をするか | 失敗した場合 |
|---------|-----------|-------------|
| **ロード** | .class のバイト列を読み込み、`java.lang.Class` オブジェクトを生成 | `ClassNotFoundException` |
| **検証 (Verify)** | バイトコードが JVM 仕様に違反していないか確認（スタック操作の不整合、不正な型変換等） | `VerifyError` |
| **準備 (Prepare)** | static フィールドにデフォルト値（int なら 0、参照型なら null）を設定 | -- |
| **解決 (Resolve)** | シンボル参照（クラス名の文字列等）を、メモリ上の実体への直接参照に変換 | `NoClassDefFoundError` |
| **初期化** | `static {}` ブロックや `static final` フィールドの初期値式を実行 | `ExceptionInInitializerError` |

### 3.4 Spring Boot との関連 -- 動的クラスロードの世界

Spring Boot は独自のクラスローダー（`LaunchedURLClassLoader`）を使い、fat JAR（実行可能 JAR）の中にネストされた依存ライブラリの JAR を読み込む。これは標準のクラスローダーでは「JAR の中の JAR」を読めないためである。

```
spring-boot-app.jar
 ├── BOOT-INF/
 │   ├── classes/          ← アプリケーションのクラス
 │   └── lib/              ← 依存ライブラリの JAR（ネスト）
 │       ├── spring-core-6.x.jar
 │       ├── jackson-databind-2.x.jar
 │       └── ...
 └── org/springframework/boot/loader/
     └── LaunchedURLClassLoader  ← ネスト JAR を読むための独自 CL
```

また、Spring の DI（Dependency Injection）コンテナは起動時にクラスパスをスキャンして `@Component` 等のアノテーションが付いたクラスを発見・ロードする。このため、**Spring Boot アプリケーションの起動時間の大部分は、クラスのロードと初期化に費やされている。**

## 4. ランタイムデータエリア (Runtime Data Areas)

**対象層:** Java 層 / HotSpot C++ 層（`src/hotspot/share/runtime/`, `src/hotspot/share/memory/`）

ランタイムデータエリアは、JVM がプログラム実行中にデータを格納するメモリ領域の総称である。

> **詳細は [memory-basics.md](memory/memory-basics.md) を参照。** 本セクションでは全体像の把握に必要な範囲のみ記述する。

### 4.1 メモリ領域の全体図

```
 ┌──────────────────────────────────────────────────────────────────┐
 │                    JVM プロセスのメモリ空間                       │
 │                                                                  │
 │  ┌────────────────────────────── 全スレッド共有 ──────────────┐  │
 │  │                                                            │  │
 │  │  ┌──────────────────────┐  ┌────────────────────────────┐  │  │
 │  │  │       ヒープ         │  │      Metaspace             │  │  │
 │  │  │  (オブジェクト格納)  │  │  (クラス情報, メソッド情報) │  │  │
 │  │  │  -Xms / -Xmx で制御 │  │  ネイティブメモリ上に配置   │  │  │
 │  │  └──────────────────────┘  └────────────────────────────┘  │  │
 │  │                                                            │  │
 │  │  ┌──────────────────────┐  ┌────────────────────────────┐  │  │
 │  │  │  コードキャッシュ    │  │    文字列プール             │  │  │
 │  │  │  (JIT 変換済み機械語)│  │  (intern された文字列)      │  │  │
 │  │  └──────────────────────┘  └────────────────────────────┘  │  │
 │  └────────────────────────────────────────────────────────────┘  │
 │                                                                  │
 │  ┌─────────────────────── スレッドごとに独立 ─────────────────┐  │
 │  │                                                            │  │
 │  │  ┌──────────────────────┐  ┌────────────────────────────┐  │  │
```

## 5. 実行エンジン (Execution Engine)

**対象層:** HotSpot C++ 層（`src/hotspot/share/interpreter/`, `src/hotspot/share/opto/`, `src/hotspot/share/c1/`）

実行エンジンは、クラスローダーがメモリに配置したバイトコードを **実際に実行する** コンポーネントである。JVM の「CPU」に相当する。

> **詳細は [execution-engine-deep-dive.md](runtime/execution-engine-deep-dive.md) を参照。** 本セクションでは全体像の把握に必要な範囲のみ記述する。

### 5.1 インタプリタ -- バイトコードを1命令ずつ実行

JVM はまずインタプリタモードでバイトコードを実行する。バイトコードの各命令（`iload`, `iadd`, `invokevirtual` 等）を1つずつ読み取り、対応する処理を実行する。

```
バイトコード例: 1 + 2 を計算

  iconst_1    ← 定数 1 をスタックに積む
  iconst_2    ← 定数 2 をスタックに積む
  iadd        ← スタックから 2 つ取り出して加算、結果をスタックに積む
  istore_0    ← 結果をローカル変数 0 に保存

インタプリタは上記を 1 命令ずつ逐次実行する
```

インタプリタの特徴は **起動が速い** こと。コンパイルの待ち時間なしにすぐ実行を開始できる。ただし、同じバイトコードを何万回実行しても毎回解釈し直すため、**繰り返しの実行は遅い。**

- 関連ソースパス: `src/hotspot/share/interpreter/templateInterpreter.cpp`

### 5.2 JIT コンパイラ -- ホットスポットを機械語に変換

JIT (Just-In-Time) コンパイラは、**頻繁に実行されるコード（ホットスポット）を検出し、バイトコードを CPU が直接実行できる機械語に変換する** コンポーネントである。HotSpot VM の名前の由来でもある。

```
メソッド invoke() の呼び出し回数を監視:

  invoke() が 1 回目:    インタプリタで実行
  invoke() が 100 回目:  インタプリタで実行
  invoke() が 1,000 回目: まだインタプリタ
     :
  invoke() が ~10,000 回目: ★ ホットスポット検出！
     │
     ▼
  JIT コンパイラが invoke() を機械語に変換
  → 結果をコードキャッシュに格納
     │
     ▼
  invoke() が 10,001 回目以降: 機械語で直接実行 ← 高速！
```

HotSpot VM には2つの JIT コンパイラがある。

| コンパイラ | 別名 | 特徴 |
|-----------|------|------|
| **C1** | Client Compiler | コンパイルが速い。軽い最適化。起動速度重視 |
| **C2** | Server Compiler | コンパイルに時間がかかる。積極的な最適化。ピーク性能重視 |

- C1 関連ソースパス: `src/hotspot/share/c1/`
- C2 関連ソースパス: `src/hotspot/share/opto/`

### 5.3 Tiered Compilation（段階的コンパイル）

現在の HotSpot VM（Java 8 以降のデフォルト）は、C1 と C2 を **組み合わせて** 使う「Tiered Compilation（段階的コンパイル）」を採用している。

```
 実行開始
    │
    ▼
 ┌────────────────────────────────┐
 │  Level 0: インタプリタ          │  バイトコードを逐次実行
 │  (プロファイリング情報を収集)    │  「このメソッドは何回呼ばれたか」
 └───────────────┬────────────────┘  「この分岐はどちら側が多いか」
                 │ 閾値到達
                 ▼
 ┌────────────────────────────────┐
 │  Level 1-3: C1 コンパイル       │  軽い最適化で機械語に変換
 │  (引き続きプロファイリング)      │  起動速度とのバランスを取る
 └───────────────┬────────────────┘
                 │ さらにホットと判明
                 ▼
 ┌────────────────────────────────┐
 │  Level 4: C2 コンパイル         │  積極的な最適化で機械語に変換
 │  (インライン化, ループ展開,      │  ピーク性能を追求
 │   エスケープ解析, etc.)         │
 └────────────────────────────────┘
```

**なぜ段階的にするのか:**

- **起動時:** C2 のフルコンパイルを待っていると起動が遅い。まず C1 で軽く最適化して応答速度を確保
- **定常状態:** 十分なプロファイリング情報が集まってから C2 で最適化すると、より賢い最適化が可能になる
- **結果:** 起動速度とピーク性能の**両立**

C2 が行う代表的な最適化:

| 最適化 | 説明 |
|--------|------|
| **インライン化 (Inlining)** | メソッド呼び出しをなくし、呼び出し先の本体を埋め込む |
| **エスケープ解析 (Escape Analysis)** | オブジェクトがメソッド外に「逃げない」場合、ヒープ割り当てを省略 |
| **ループ展開 (Loop Unrolling)** | ループの反復を複数回分まとめて、分岐コストを削減 |
| **デッドコード除去** | 実行されないコードを除去 |
| **定数畳み込み (Constant Folding)** | コンパイル時に計算可能な式を事前に計算 |

### 5.4 ガベージコレクター (GC)

GC は、ヒープ上で不要になったオブジェクトを自動的に検出し、メモリを回収するサブシステムである。実行エンジンの一部として分類されるが、独立した巨大なサブシステムであり、JVM の性能特性を大きく左右する。

Java 25（Zulu 25）で利用可能な主要 GC:

| GC | 特徴 | 用途 |
|----|------|------|
| **G1GC** (デフォルト) | リージョンベース。停止時間目標を設定可能 | 汎用。ほとんどのアプリケーション |
| **ZGC** | 超低レイテンシ（停止時間 < 1ms）。大容量ヒープ対応 | レイテンシ最重要のアプリ |
| **Shenandoah** | 低レイテンシ。コンカレント・コンパクション | ZGC の代替 |
| **Parallel GC** | スループット最大化。停止時間は長め | バッチ処理 |
| **Serial GC** | シングルスレッド。最小フットプリント | コンテナ、小規模アプリ |

> **GC の詳細（世代別仮説、GC アルゴリズム、G1GC の内部構造、GC ログの読み方、チューニング等）は [gc-deep-dive.md](memory/gc-deep-dive.md) を参照。**

## 6. ネイティブメソッドインタフェース (JNI) とネイティブメソッドライブラリ

**対象層:** JNI 層（Java 層と HotSpot C++ 層の境界）

### 6.1 JNI とは

JNI (Java Native Interface) は、**Java コードから C/C++ 等のネイティブコードを呼び出す（またはその逆の）ための標準インタフェース** である。

Java だけでは実現できない処理 -- OS 固有の機能呼び出し、ハードウェアの直接操作、パフォーマンスクリティカルな計算 -- を行うために存在する。

> **詳細は [jni-deep-dive.md](native/jni-deep-dive.md) を参照。** 本セクションでは全体像の把握に必要な範囲のみ記述する。

### 6.2 身近な利用例

実は、Java の標準ライブラリ自体が内部で大量の JNI を使用している。

| Java API | 内部で呼び出すネイティブコード | 理由 |
|----------|------------------------------|------|
| `java.io.FileInputStream` | OS のファイル読み込みシステムコール | ファイル I/O は OS の機能 |
| `java.net.Socket` | OS のソケット API | ネットワーク通信は OS の機能 |
| `java.lang.Thread#start()` | OS のスレッド生成 API | スレッドは OS が管理 |
| `System.currentTimeMillis()` | OS の時刻取得 API | 時刻は OS が管理 |
| JDBC ドライバ（一部） | データベースのネイティブクライアントライブラリ | 性能・互換性のため |

### 6.3 JNI の処理フロー

```
  Java 層                    JNI 層                 ネイティブ層 (C/C++)
 ─────────────────────────────────────────────────────────────────
  native メソッド呼び出し
  FileInputStream.read()
       │
       ▼
  JVM が native メソッド   JNI フレームワーク
  であることを検出          が引数を変換
       │                  (Java 型 → C 型)
       ▼                       │
                               ▼
                         OS のシステムコール
                         read() を実行
                               │
                               ▼
                          戻り値を変換
                         (C 型 → Java 型)
       │                       │
       ◀───────────────────────┘
  Java コードに結果を返す
```

**JNI のコスト:** Java → ネイティブの境界を越えるたびに、引数の型変換やセキュリティチェックが発生する。このため、**JNI 呼び出しは通常の Java メソッド呼び出しよりも高コスト** である。頻繁に JNI を呼ぶ設計は避けるべきである。

> **補足:** Java 22 で正式機能となった **Foreign Function & Memory API (FFM API / JEP 454)** は、JNI の後継として位置づけられている。JNI よりも安全で、ボイラープレートが少なく、性能面でも改善されている。新規開発では FFM API の利用を検討すべきである。

## 7. Java プログラムの実行フロー -- ソースコードから実行結果まで

### 7.1 全体フロー図

`.java` ファイルから実行結果が出力されるまでの全過程を時系列で示す。

```
  ┌──────────────────────────────────────────────────────────────┐
  │  1. コンパイル (javac)                     ← JVM の「外」    │
  │                                                              │
  │    Hello.java  ──javac──▶  Hello.class (バイトコード)         │
  └──────────────────────────────┬───────────────────────────────┘
                                 │
  ┌──────────────────────────────▼───────────────────────────────┐
  │  2. JVM 起動                                                 │
  │                                                              │
  │    $ java Hello                                              │
  │    → JVM プロセスが OS 上に生成される                          │
  │    → ヒープ、スタック等のメモリ領域を確保                      │
  │    → GC スレッド、JIT コンパイラスレッドが起動                 │
  └──────────────────────────────┬───────────────────────────────┘
                                 │
  ┌──────────────────────────────▼───────────────────────────────┐
  │  3. クラスのロード                                            │
  │                                                              │
  │    クラスローダーが Hello.class を読み込む                     │
  │    → 依存するクラス (java.lang.Object 等) も連鎖的にロード    │
  └──────────────────────────────┬───────────────────────────────┘
                                 │
  ┌──────────────────────────────▼───────────────────────────────┐
  │  4. バイトコード検証 (Verification)                           │
  │                                                              │
  │    バイトコードが JVM 仕様に適合しているか検証                 │
  │    → 不正なスタック操作、型の不整合などを検出                  │
  └──────────────────────────────┬───────────────────────────────┘
                                 │
  ┌──────────────────────────────▼───────────────────────────────┐
  │  5. メモリへの配置                                            │
  │                                                              │
  │    クラス情報 → Metaspace に配置                              │
  │    static 変数 → ヒープ上の Class オブジェクトに関連付け       │
  │    メソッドのバイトコード → Metaspace に配置                   │
  └──────────────────────────────┬───────────────────────────────┘

  │  6. main メソッドの実行開始                                   │

  │  7. 実行中の最適化 (JIT コンパイル)                           │

  │    ホットスポット検出 → C1 コンパイル → C2 コンパイル          │
  │    → 機械語をコードキャッシュに格納                            │

  └──────────────────────────────┬───────────────────────────────┘

  │    main メソッドが return (または全ノンデーモンスレッド終了)   │

  │    → ファイナライザを実行（非推奨、使用すべきでない）          │
  │    → ヒープ、スタック等のメモリを OS に返却                   │
  │    → JVM プロセスが終了                                      │
  └──────────────────────────────────────────────────────────────┘
```

### 7.2 各ステップの詳細

#### ステップ 1: コンパイル (`javac`)

`javac` は JVM とは**独立した**プログラムであり、Java ソースコードをバイトコード（.class ファイル）に変換する。この段階では JVM は起動していない。

```java
// Hello.java -- source code
void main() {
    IO.println("Hello, JVM!");
}
```

バイトコードは CPU アーキテクチャに依存しない中間表現であり、`.class` ファイルはどのプラットフォームでも同一のバイナリになる。

> **補足:** JEP 477/495（Java 25 で標準機能）により、`void main()` のようなシンプルなエントリーポイントを記述できる。上記は `public class Hello` や `public static void main(String[] args)` を省略した形式である。`java Hello.java` で直接実行する場合（ソースコードランチャー）は、`javac` による明示的なコンパイルも省略される。

#### ステップ 2-5: JVM 起動からメモリ配置まで

`java Hello` を実行すると、OS 上に JVM プロセスが生成される。JVM は以下を行う:

1. **メモリ確保:** ヒープ（`-Xms`/`-Xmx`）、スタック（`-Xss`）、Metaspace 等のメモリ領域を OS から確保
2. **内部スレッド起動:** GC スレッド、JIT コンパイラスレッド、シグナルハンドラスレッド等を起動
3. **コアクラスのロード:** Bootstrap ClassLoader が `java.lang.Object`, `java.lang.Class`, `java.lang.String` 等の基本クラスをロード
4. **アプリケーションクラスのロード:** Application ClassLoader が `Hello.class` をロードし、リンク・初期化

#### ステップ 6-8: 実行と最適化

main メソッドの実行が開始されると、インタプリタ、JIT コンパイラ、GC が**並行して**動作する。

```
  時間 ──────────────────────────────────────────────▶

  main スレッド:  [インタプリタ実行]──[C1最適化コード]──[C2最適化コード]──▶
  JIT スレッド:              [C1コンパイル]   [C2コンパイル]
  GC スレッド:    [監視]──[Minor GC]──[監視]──[Minor GC]──[監視]──▶
```

この**並行動作**こそが HotSpot VM の核心的な設計であり、プログラムの実行を止めずに最適化とメモリ管理を行う。

#### ステップ 9: シャットダウン

JVM のシャットダウンは以下の条件で開始される:

- `main` メソッドが正常に終了し、すべてのノンデーモンスレッドが終了した場合
- `System.exit()` が呼ばれた場合
- OS からの終了シグナル（`SIGTERM` 等）を受信した場合

シャットダウン時には、`Runtime.addShutdownHook()` で登録されたフックが実行される。Spring Boot の場合、このフックで DI コンテナの `close()` が呼ばれ、DB コネクションプールの解放やログのフラッシュ等が行われる。
