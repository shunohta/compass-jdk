# Java エディション -- SE / EE / ME の歴史と現在

## 概要

このドキュメントは、Java が「なぜ3つのエディションに分かれたのか」「それぞれがどう進化し、今どうなっているのか」を、1990 年代後半の分岐から 2025 年の Java 25 時代に至るまで詳しく解説する。

Java の世界で最も混乱を生む領域の一つが、この「エディション」の問題である。J2SE、J2EE、J2ME、Java SE、Java EE、Jakarta EE ── 似たような名前が次々と登場し、名前空間の変更（`javax.*` → `jakarta.*`）まで加わって、全体像を掴むのは容易ではない。本ドキュメントはその混乱を解きほぐすことを目的とする。

> **対象読者:** Java を学んでいる開発者、「Java SE と Java EE の違いがよく分からない」方、Jakarta EE への移行を検討している方、Java の全体像を俯瞰したい方。

> **前提知識:** [Java の歴史](java-history.md) を先に読んでおくと、時系列の理解がスムーズになる。

## 1. なぜ Java は3つに分かれたのか

### 1990 年代後半のコンピューティング環境

Java 1.0 (1996年) がリリースされた直後、Java は爆発的に普及した。しかし、その普及先は一つの領域ではなかった。1990 年代後半のコンピューティング環境は、大きく3つの世界に分かれていた。

```
┌─────────────── 1990年代後半のコンピューティング環境 ───────────────┐
│                                                                   │
│  ┌──────────────────┐  ┌──────────────────┐  ┌────────────────┐  │
│  │   サーバーサイド   │  │  デスクトップ /   │  │ モバイル /      │  │
│  │                  │  │  クライアント      │  │ 組み込みデバイス │  │
│  ├──────────────────┤  ├──────────────────┤  ├────────────────┤  │
│  │ ・企業の基幹系    │  │ ・デスクトップ    │  │ ・携帯電話      │  │
│  │   業務システム    │  │   アプリケーション │  │ ・PDA           │  │
│  │ ・Web アプリ      │  │ ・ブラウザ Applet │  │ ・セットトップ   │  │
│  │ ・トランザクション │  │ ・開発ツール      │  │   ボックス      │  │
│  │   処理            │  │ ・GUI アプリ      │  │ ・カーナビ       │  │
│  │ ・メッセージング   │  │                  │  │ ・スマートカード │  │
│  ├──────────────────┤  ├──────────────────┤  ├────────────────┤  │
│  │ 要求: 堅牢性     │  │ 要求: 汎用性     │  │ 要求: 省メモリ  │  │
│  │ スケーラビリティ  │  │ 標準ライブラリ    │  │ 省 CPU          │  │
│  │ 分散トランザクション│  │ 使いやすさ       │  │ 小フットプリント │  │
│  └──────────────────┘  └──────────────────┘  └────────────────┘  │
│         ↓                      ↓                      ↓          │
│      Java EE               Java SE                Java ME        │
│   (Enterprise Edition)  (Standard Edition)     (Micro Edition)   │
└───────────────────────────────────────────────────────────────────┘
```

### 1つの言語では足りなかった

Java 1.0/1.1 の時点では、言語とライブラリは一つのパッケージだった。しかし、3つの世界のニーズは根本的に異なっていた。

| 要件 | サーバーサイド | デスクトップ | モバイル/組み込み |
|------|---------------|-------------|------------------|
| メモリ | 数 GB | 数百 MB | 数百 KB〜数 MB |
| CPU | マルチプロセッサ | シングル〜デュアル | 低クロック |
| 必要な API | トランザクション、メッセージング、分散オブジェクト | GUI、I/O、コレクション | 最小限のコア API |
| ライフサイクル | 24/365 稼働 | ユーザー操作単位 | バッテリー制約 |
| 開発者 | エンタープライズ企業 | 一般開発者 | デバイスメーカー |

一つの巨大な仕様にすべてを詰め込むのは現実的ではなかった。サーバーサイドの開発者にとって KVM（Kilobyte Virtual Machine）の制約は無関係だし、携帯電話に EJB（Enterprise JavaBeans）は載せられない。

**Sun Microsystems の解決策: エディション分割** ── Java という「一つの言語」を維持しつつ、プラットフォームを3つの「エディション」に分ける。これが 1999 年の **Java 2 Platform** 発表で実現された。

## 2. 命名の変遷 -- Java 史上最大の混乱ポイント

Java のバージョニングと命名は、IT 業界でもトップクラスにややこしい。ここで一度、完全に整理する。

### 2.1 前史: エディション分割前 (1996-1998)

最初の Java にはエディションの概念がなかった。

| リリース | 年 | 備考 |
|---------|------|------|
| JDK 1.0 | 1996年1月 | 最初のリリース |
| JDK 1.1 | 1997年2月 | 内部クラス、JDBC、RMI 追加 |

この時点ではすべてが「JDK」であり、Standard も Enterprise も Micro もなかった。

### 2.2 Java 2 Platform の誕生 (1998-1999)

1998年12月、Sun は **JDK 1.2** をリリースした。このリリースは Java にとって極めて大きな転換点だった。Collections Framework、Swing、JIT コンパイラなど大幅な機能追加があり、Sun はこれを単なるマイナーアップデートではなく「第2世代の Java」として位置づけた。

**なぜ「2」なのか:** JDK 1.2 の内部バージョン番号は `1.2` だが、Sun はマーケティング上これを **Java 2 Platform** とリブランディングした。「1.2」というバージョン番号のインパクトの弱さを補い、大幅な進化をアピールする狙いがあった。

このとき同時に、3つのエディションが定義された。

| エディション | 正式名称 | リリース年 |
|-------------|---------|-----------|
| **J2SE** | Java 2 Platform, Standard Edition | 1998年 (JDK 1.2) |
| **J2EE** | Java 2 Platform, Enterprise Edition | 1999年 (J2EE 1.2) |
| **J2ME** | Java 2 Platform, Micro Edition | 1999年 (仕様策定開始) |

### 2.3 J2SE のバージョニング混乱 (2004)

2004年、J2SE 5.0 のリリースでさらに混乱が生じた。

```
┌──────────────────────────────────────────────────────────────────┐
│                J2SE のバージョン番号問題                          │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  内部バージョン    マーケティング名     通称                      │
│  ─────────────    ──────────────────   ──────                    │
│  1.2              J2SE 1.2             Java 2                    │
│  1.3              J2SE 1.3             --                        │
│  1.4              J2SE 1.4             --                        │
│  1.5.0            J2SE 5.0             Java 5, Tiger             │
│                   ↑                                              │
│                   ここで突然「1.5」→「5.0」にジャンプ !           │
│                   理由: 「1.5 はインパクトが弱い」               │
│                   内部的には 1.5 のまま (java -version で確認可)  │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

`java -version` を実行すると `1.5.0` と表示されるのに、公式名称は「J2SE 5.0」。この二重バージョニングは長年にわたって開発者を混乱させた。

### 2.4 「J2」の廃止と Java SE/EE/ME への改名 (2006)

2006年の Java SE 6 で、Sun はついに命名体系を整理した。

```
┌──────────────────────────────────────────────────────────────────┐
│                    2006年の命名リブランディング                    │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  旧名称                          新名称                          │
│  ──────                          ──────                          │
│  Java 2 Platform, Standard Ed.   →  Java Platform, Standard Ed. │
│  (J2SE)                              (Java SE)                   │
│                                                                  │
│  Java 2 Platform, Enterprise Ed. →  Java Platform, Enterprise Ed│
│  (J2EE)                              (Java EE)                  │
│                                                                  │
│  Java 2 Platform, Micro Ed.      →  Java Platform, Micro Ed.   │
│  (J2ME)                              (Java ME)                  │
│                                                                  │
│  変更理由:                                                       │
│  ・「2」はもはや意味をなさない（Java 6 なのに「2」?）             │
│  ・バージョン番号も「1.6」を廃止し単に「6」に統一                 │
│  ・よりシンプルで直感的な命名へ                                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 3. Java SE (Standard Edition) の歴史と現在

### 3.1 Java SE とは何か

Java SE は Java プラットフォームの**基盤**である。以下の3要素で構成される。

```
┌────────────────────────── Java SE ──────────────────────────┐
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │              Java 言語仕様 (JLS)                       │  │
│  │  ・文法、型システム、継承、ジェネリクス                  │  │
│  │  ・パターンマッチング、レコード、sealed class ...        │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │              標準クラスライブラリ (API)                  │  │
│  │  ・java.lang, java.util, java.io, java.net ...         │  │
│  │  ・Collections, Stream, NIO, Concurrency ...           │  │
│  │  ・java.lang.foreign (FFM API), java.util.stream ...   │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │              Java 仮想マシン仕様 (JVMS)                │  │
│  │  ・バイトコード定義、クラスファイル形式                  │  │
│  │  ・メモリモデル、スレッドモデル                         │  │
│  │  ・GC、JIT コンパイラは実装依存                         │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  重要: Java SE は「仕様」であり「実装」ではない              │
│  実装 = OpenJDK, Azul Zulu, Eclipse Temurin, Amazon Corretto│
└─────────────────────────────────────────────────────────────┘
```

**Java SE は「仕様」であり「実装」ではない** ── これは極めて重要な区別である。Java SE は JCP（Java Community Process）によって策定される仕様であり、その仕様を実装したものが JDK（Java Development Kit）である。OpenJDK はその参照実装（Reference Implementation）であり、Azul Zulu や Eclipse Temurin はOpenJDK をベースにした各ベンダーの配布物である。詳しくは [JDK ディストリビューション比較](jdk-distributions-overview.md) を参照。

### 3.4 本プロジェクトにおける Java SE の位置づけ

本プロジェクト (jdk-core) のメインターゲットは **Java SE 25** である。具体的には:

- **参照ランタイム:** Azul Zulu 25 (OpenJDK 25 ベース)
- **言語機能:** JEP 477 (暗黙クラス)、JEP 495 (暗黙 import) など Java 25 の標準機能を使用
- **内部解析対象:** HotSpot VM (OpenJDK のデフォルト JVM 実装) の C++ 層まで含む

Java EE / Jakarta EE や Java ME は本プロジェクトの直接の対象外だが、Java SE がそれらの基盤であることを理解しておくことは重要である。

## 4. Java EE (Enterprise Edition) の波乱の歴史

Java EE は Java 3エディションの中で最も波乱に満ちた歴史を持つ。エンタープライズ Java の栄光、「複雑すぎる」という批判、Spring との競争、そして Oracle から Eclipse Foundation への移管 ── その全貌を追う。

### 4.1 J2EE の誕生 (1999): エンタープライズ Java の幕開け

1999年、Sun は **J2EE 1.2** をリリースした。この仕様は Java をエンタープライズシステムの主力言語に押し上げた立役者である。

**J2EE 1.2 に含まれた主要技術:**

| 技術 | 役割 |
|------|------|
| **Servlet** | HTTP リクエスト/レスポンスを処理するサーバーサイドコンポーネント |
| **JSP** (JavaServer Pages) | HTML 内に Java コードを埋め込むテンプレートエンジン |
| **EJB** (Enterprise JavaBeans) | ビジネスロジックのコンポーネントモデル。トランザクション管理、分散オブジェクトを提供 |
| **JDBC** | データベース接続 (SE から引き継ぎ) |
| **JMS** (Java Message Service) | 非同期メッセージング |
| **JNDI** (Java Naming and Directory Interface) | 名前付けとディレクトリサービス |
| **RMI/IIOP** | 分散オブジェクト通信 |

**アプリケーションサーバー戦争:**
J2EE の仕様が公開されると、各ベンダーがこぞってアプリケーションサーバーを開発・販売した。

```
┌───────── 2000年代初頭: アプリケーションサーバー戦争 ─────────┐
│                                                              │
│  商用サーバー                    オープンソースサーバー       │
│  ┌────────────────────────┐     ┌──────────────────────┐    │
│  │ BEA WebLogic           │     │ JBoss (後の WildFly) │    │
│  │ IBM WebSphere          │     │ Apache Tomcat        │    │
│  │ Oracle Application     │     │  (Servlet/JSP のみ)  │    │
│  │   Server               │     │ Apache Geronimo      │    │
│  │ Sun ONE / GlassFish    │     │ JOnAS                │    │
│  └────────────────────────┘     └──────────────────────┘    │
│                                                              │
│  価格: 数百万〜数千万円          価格: 無料                  │
│  サポート: 手厚い                サポート: コミュニティ       │
│                                                              │
│  → この競争が Java のエンタープライズ市場での地位を確立       │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 J2EE 1.2〜1.4 (1999-2003): 全盛期と「複雑さ」の代名詞

**J2EE 1.3 (2001):**
- EJB 2.0: Entity Bean（データベースとのマッピングを持つコンポーネント）の導入
- JCA (Java Connector Architecture): 既存システムとの接続

**J2EE 1.4 (2003):**
- Web Services サポート (JAX-RPC, SAAJ)
- Management/Deployment API
- J2EE 仕様の最大規模リリース

**しかし、この時期の J2EE には深刻な問題があった。**

EJB 2.x の Entity Bean を書くには、以下のような大量のファイルが必要だった:

```
1つの Entity Bean を作るために必要なもの（EJB 2.x）:
─────────────────────────────────────────────────
  1. Remote Interface         (ビジネスメソッドの定義)
  2. Home Interface           (ライフサイクル管理: create, find)
  3. Bean Implementation      (実際のロジック)
  4. Local Interface           (ローカルアクセス用)
  5. Local Home Interface      (ローカルライフサイクル)
  6. Primary Key Class         (主キーの定義)
  7. ejb-jar.xml              (デプロイメント記述子)
  8. vendor-specific.xml      (サーバー固有の設定)
  ─────────────────────────────────────────────
  合計: 1つのテーブルに対して 6-8 ファイル !

  → 「Hello World」を書くだけでも XML 設定ファイルの山
  → 開発者コミュニティから「EJB は複雑すぎる」の大合唱
```

この「設定地獄」は Java EE 最大の負の遺産であり、後の Spring Framework 台頭の直接的な原因となった。

### 4.3 Spring Framework の登場と競争 (2002-2004)

2002年、Rod Johnson が書籍 *"Expert One-on-One J2EE Design and Development"* を出版した。この本は J2EE の設計パターンを解説しつつ、EJB の複雑さを痛烈に批判し、より軽量な代替アプローチを提案した。翌 2003年、Johnson はその思想を具現化した **Spring Framework** をオープンソースで公開した。

```
┌──────── EJB 2.x vs Spring Framework (2003年頃) ──────────┐
│                                                           │
│  EJB 2.x                     Spring Framework             │
│  ──────────                   ────────────────             │
│  ・重い EJB コンテナが必須    ・軽量な DI コンテナ          │
│  ・多数のインターフェース必須 ・POJO (普通の Java クラス)    │
│  ・XML 設定が山ほど必要      ・XML は最小限 (後にアノテーション)│
│  ・テストが困難               ・単体テスト容易              │
│  ・アプリサーバーに依存       ・Servlet コンテナで十分      │
│  ・学習コスト: 非常に高い     ・学習コスト: 低い            │
│                                                           │
│  結果: 多くの開発者が Spring に流れた                      │
│  → J2EE/Java EE は Spring に強い影響を受けて軽量化へ      │
└───────────────────────────────────────────────────────────┘
```

Spring Framework の成功は Java EE の進化を大きく促した。以降の Java EE のバージョンアップは、いわば「Spring に追いつく」歴史でもある。

### 4.4 Java EE 5 (2006): 軽量化の始まり

Java EE 5 は、Spring からの批判と競争を受けて**大幅に方針転換**したバージョンである。

**主要な変更点:**
- **EJB 3.0:** アノテーションベースに大刷新。Entity Bean を廃止し、POJO + アノテーションでビジネスロジックを記述可能に
- **JPA (Java Persistence API):** Hibernate の影響を強く受けた ORM 仕様。Entity Bean に代わる永続化の標準
- **アノテーションの全面導入:** XML 設定ファイルの大幅削減
- **Convention over Configuration:** 規約に従えば設定不要

```
┌────────── EJB 2.x → EJB 3.0 の劇的変化 ──────────┐
│                                                     │
│  EJB 2.x (必要なファイル: 6-8個)                    │
│  ──────────────────────────                         │
│  public interface AccountHome extends EJBHome {     │
│      Account create(String id) throws ...;          │
│      Account findByPrimaryKey(String id) throws ...;│
│  }                                                  │
│  public interface Account extends EJBObject {       │
│      String getId() throws RemoteException;         │
│      void deposit(int amount) throws ...;           │
│  }                                                  │
│  public class AccountBean implements EntityBean {   │
│      // 大量のライフサイクルメソッドの実装が必須     │
│      public void ejbCreate(...) { ... }             │
│      public void ejbActivate() { ... }              │
│      public void ejbPassivate() { ... }             │
│      public void ejbRemove() { ... }                │
│      ...                                            │
│  }                                                  │
│  + ejb-jar.xml (数十行の XML)                       │
│                                                     │
│          ↓ EJB 3.0 で激変 ↓                         │
│                                                     │
│  EJB 3.0 (必要なファイル: 1個)                      │
│  ──────────────────────────                         │
│  @Entity                                            │
│  public class Account {                             │
│      @Id private String id;                         │
│      private int balance;                           │
│                                                     │
│      public void deposit(int amount) {              │
│          this.balance += amount;                    │
│      }                                              │
│  }                                                  │
│                                                     │
│  → ファイル数: 6-8 → 1、XML: 数十行 → 0行          │
└─────────────────────────────────────────────────────┘
```

> **補足:** JPA の仕様策定には Hibernate の作者 Gavin King が深く関与しており、Hibernate の API が JPA の設計に強い影響を与えた。JPA は仕様であり、Hibernate はその最も有名な実装の一つという関係になった。

### 4.5 Java EE 6-7 (2009-2013): さらなる進化

**Java EE 6 (2009):**
- **CDI (Contexts and Dependency Injection):** Java EE 標準の DI コンテナ。Spring DI に対する Java EE の回答
- **JAX-RS 1.0:** RESTful Web Services の標準 API
- **Bean Validation:** アノテーションベースのバリデーション
- **Web Profile:** フル Java EE のサブセット。Web アプリ開発に必要な仕様だけを集めた軽量プロファイル

**Java EE 7 (2013):**
- **WebSocket API:** リアルタイム双方向通信
- **JSON-P (JSON Processing):** JSON の読み書きの標準 API
- **Batch API:** バッチ処理の標準化
- **Concurrency Utilities for Java EE:** マネージドスレッド

### 4.6 Java EE 8 (2017): Oracle 最後の Java EE

Java EE 8 は Oracle がリリースした**最後の Java EE** となった。

- **Servlet 4.0:** HTTP/2 対応
- **JSON-B (JSON Binding):** JSON とオブジェクト間のバインディング API
- **Security API 1.0:** セキュリティの標準化・簡素化
- **JAX-RS 2.1:** リアクティブ拡張、Server-Sent Events

しかし、Java EE 8 のリリースは大幅に遅延した。Java EE 7 (2013) から4年もの間が空いており、コミュニティからは Oracle の Java EE への投資意欲の低下を疑う声が高まっていた。

### 4.7 Oracle から Eclipse Foundation への移管 (2017-2018)

2017年9月、Oracle は衝撃的な発表を行った: **Java EE の開発を Eclipse Foundation に移管する**。

**なぜ Oracle は Java EE を手放したのか:**

```
┌─────── Oracle が Java EE を手放した背景 ──────────────────┐
│                                                           │
│  1. クラウドシフト                                        │
│     ・Oracle は Oracle Cloud Infrastructure (OCI) に      │
│       リソースを集中させたかった                           │
│     ・Java EE の仕様策定は膨大な労力を要する              │
│                                                           │
│  2. マイクロサービスの台頭                                │
│     ・モノリシックなアプリサーバーモデルの需要が減少       │
│     ・Spring Boot、Dropwizard 等の軽量フレームワークが主流│
│     ・Java EE フルプロファイルは「大きすぎる」             │
│                                                           │
│  3. コミュニティとの関係悪化                              │
│     ・Java EE 8 の開発遅延でコミュニティの信頼を失った    │
│     ・「Oracle は Java EE を塩漬けにしている」との批判     │
│                                                           │
│  4. 収益モデルとの不一致                                  │
│     ・Java EE 自体は直接的な収益源ではなかった            │
│     ・Oracle は Java SE のライセンスと                     │
│       WebLogic の販売に注力する方が合理的                  │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

**移管のプロセス:**

```
2017年9月   Oracle が Java EE の Eclipse Foundation への移管を発表
     │
2018年1月   Eclipse Foundation に「Eclipse Enterprise for Java」
     │      (EE4J) プロジェクトが設立
     │
2018年2-3月 GlassFish、Eclipse Implementation of JASPIC 等の
     │      ソースコードが Eclipse に移管
     │
2018年中    仕様書、TCK (互換性テスト) の移管交渉
     │
     │      ┌─── ここで大問題発生 ──────────────────────┐
     │      │ Oracle は「Java」の商標を手放さなかった !   │
     │      │ → Eclipse Foundation は「Java EE」という   │
     │      │   名前を使い続けることができない            │
     │      └──────────────────────────────────────────┘
     │
2018年2月   コミュニティ投票で新名称を決定
     │      候補: Jakarta EE / Enterprise Profile / Eclipse EE
     │      → **「Jakarta EE」に決定**
     │
2019年9月   Jakarta EE 8 リリース (Eclipse Foundation 初のリリース)
```

### 4.8 「Java EE」から「Jakarta EE」へ -- 名前空間の大移動

Oracle が「Java」の商標を Eclipse Foundation に許諾しなかったため、すべての `javax.*` パッケージ名も変更を余儀なくされた。これは Java エコシステムに巨大なインパクトを与えた。

```
┌──────── javax.* → jakarta.* の名前空間移行 ────────────┐
│                                                         │
│  移行前 (Java EE 8 / Jakarta EE 8)                     │
│  ──────────────────────────────────                     │
│  import javax.servlet.http.HttpServlet;                 │
│  import javax.persistence.Entity;                      │
│  import javax.inject.Inject;                           │
│  import javax.ws.rs.GET;                               │
│                                                         │
│          ↓  Jakarta EE 9 以降  ↓                        │
│                                                         │
│  移行後 (Jakarta EE 9+)                                │
│  ──────────────────────                                │
│  import jakarta.servlet.http.HttpServlet;              │
│  import jakarta.persistence.Entity;                    │
│  import jakarta.inject.Inject;                         │
│  import jakarta.ws.rs.GET;                             │
│                                                         │
│  影響:                                                  │
│  ・すべての Java EE アプリケーションでソース修正が必要   │
│  ・ライブラリ・フレームワークもすべて対応が必要          │
│  ・Spring Framework 6 / Spring Boot 3 で jakarta.* に   │
│    移行 (2022年11月)                                    │
│  ・移行ツール (Eclipse Transformer 等) が提供された      │
│                                                         │
│  注意: javax.* のうち Java SE に属するもの              │
│   (javax.crypto, javax.net.ssl 等) は変更なし           │
│   → Java SE 側の javax.* はそのまま使い続けられる       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 4.9 Jakarta EE 8-11 (2019-2024): Eclipse Foundation 時代

**Jakarta EE 8 (2019年9月):**
- Java EE 8 と**完全互換**
- パッケージ名は `javax.*` のまま（移行の第一歩として互換性を優先）
- 目的: 移管プロセスが正しく機能することの証明

**Jakarta EE 9 / 9.1 (2020年12月 / 2021年5月):**
- **`javax.*` → `jakarta.*` の名前空間移行**を実施（最大の変更点）
- 機能追加はほぼなし。名前空間変更に集中したリリース
- Jakarta EE 9: Java SE 8+ 対応
- Jakarta EE 9.1: Java SE 11+ 対応

**Jakarta EE 10 (2022年9月):**
- **Java SE 11+** を最低要件に引き上げ
- **Core Profile** の導入: マイクロサービス向けの最小プロファイル
- **CDI Lite:** CDI の軽量サブセット。ビルド時にコンパイル可能で、GraalVM Native Image と相性が良い
- Jakarta RESTful Web Services 3.1、Jakarta Persistence 3.1 等のアップデート

**Jakarta EE 11 (2024年6月, 仕様承認):**
- **Java SE 21+** を最低要件に引き上げ（レコード、sealed class、Virtual Threads が利用可能に）
- **Jakarta Data:** リポジトリパターンによるデータアクセス抽象化の新仕様。Spring Data に類似
- **Jakarta Concurrency 3.1:** Virtual Threads（[Virtual Threads 詳細](runtime/virtual-threads-deep-dive.md)）への対応
- **CDI 4.1:** より柔軟な DI サポート
- Jakarta Persistence 3.2、Jakarta Servlet 6.1 等の更新

```
┌─── Jakarta EE のバージョン進化 ──────────────────────────────────────┐
│                                                                       │
│  バージョン  年    Java SE  名前空間    主なテーマ                    │
│  ─────────  ────  ───────  ─────────  ──────────────────────────     │
│  Jakarta 8  2019  8+       javax.*    EE 8 互換 (移管証明)           │
│  Jakarta 9  2020  8+       jakarta.*  名前空間移行                    │
│  Jakarta 9.1 2021 11+      jakarta.*  Java 11 対応                   │
│  Jakarta 10 2022  11+      jakarta.*  Core Profile, CDI Lite         │
│  Jakarta 11 2024  21+      jakarta.*  Jakarta Data, VT 対応          │
│                                                                       │
│  → Java SE の LTS と足並みを揃える形で最低要件を引き上げている       │
└───────────────────────────────────────────────────────────────────────┘
```

**MicroProfile との関係:**

Eclipse MicroProfile は、Java EE/Jakarta EE をマイクロサービス向けに補完するプロジェクトである。

## 5. Java ME (Micro Edition) の盛衰

Java ME は3エディションの中で最もドラマチックな盛衰を経験した。一時は世界で最も広く展開されたプラットフォームだったが、スマートフォンの登場によりほぼ完全に消滅した。

### 5.1 J2ME の誕生 (1999-2000): モバイル Java の夢

Sun は Java の「Write Once, Run Anywhere」を携帯電話や組み込みデバイスに持ち込もうとした。しかし、1999 年の携帯電話のスペックは現在とは比較にならないほど貧弱だった。

```
┌────── 1999年頃の携帯電話スペック ──────────────────────────────┐
│                                                                │
│  CPU:     ARM7 @ 50-100 MHz                                   │
│  RAM:     256 KB - 2 MB                                       │
│  Storage: 1 - 8 MB                                            │
│  Display: 96x64 〜 128x128 ピクセル (モノクロ〜256色)         │
│  Input:   テンキー (0-9, *, #)                                │
│  Network: 9.6 kbps (PDC/GSM)                                  │
│                                                                │
│  → 標準の JVM (HotSpot) は到底載らない                       │
│  → Java SE のライブラリもメモリに収まらない                   │
│  → 専用の小さな VM と最小限の API が必要                      │
└────────────────────────────────────────────────────────────────┘
```

この制約に対応するため、J2ME は**二層構造**で設計された。

```
┌──────────── J2ME のアーキテクチャ ────────────────────────────┐
│                                                               │
│  ┌─────────────────────────────────────────────────────────┐  │
│  │                    プロファイル (Profile)                 │  │
│  │  デバイスの UI やネットワーク API を定義                  │  │
│  │                                                          │  │
│  │  ┌─────────────────┐  ┌────────────────────────────┐    │  │
│  │  │ MIDP            │  │ Foundation Profile         │    │  │
│  │  │ (Mobile Info     │  │ Personal Profile           │    │  │
│  │  │  Device Profile) │  │ Personal Basis Profile     │    │  │
│  │  │ 携帯電話向け     │  │ セットトップボックス等向け │    │  │
│  │  └────────┬────────┘  └──────────┬─────────────────┘    │  │
│  └───────────┼──────────────────────┼──────────────────────┘  │
│              ▼                      ▼                         │
│  ┌─────────────────────────────────────────────────────────┐  │
│  │              コンフィグレーション (Configuration)         │  │
│  │  デバイスの能力に応じた VM と最小 API セットを定義        │  │
│  │                                                          │  │
│  │  ┌─────────────────────┐  ┌─────────────────────────┐   │  │
│  │  │ CLDC                │  │ CDC                     │   │  │
│  │  │ (Connected Limited   │  │ (Connected Device      │   │  │
│  │  │  Device Config.)    │  │  Configuration)         │   │  │
│  │  │                     │  │                         │   │  │
│  │  │ 対象: 携帯電話, PDA │  │ 対象: STB, カーナビ    │   │  │
│  │  │ VM: KVM             │  │ VM: CVM (ほぼフル JVM) │   │  │
│  │  │ メモリ: 128-512 KB  │  │ メモリ: 2 MB+          │   │  │
│  │  │ float: なし(CLDC1.0)│  │ float: あり            │   │  │
│  │  │ ファイナライザ: なし │  │ フルGC: あり           │   │  │
│  │  └─────────────────────┘  └─────────────────────────┘   │  │
│  └─────────────────────────────────────────────────────────┘  │
│                                                               │
│  KVM = Kilobyte Virtual Machine                              │
│  ・名前の通り「キロバイト単位」のメモリで動く VM             │
│  ・JIT なし（インタプリタのみ）                               │
│  ・フル JVM の機能を大幅に削減                                │
└───────────────────────────────────────────────────────────────┘
```

**主要な制約 (CLDC 1.0):**
- 浮動小数点演算なし（`float`/`double` が使えない!）
- ファイナライザなし、弱参照なし
- リフレクションなし
- スレッドグループなし
- `java.lang`, `java.util`, `java.io` のサブセットのみ
- クラスローダーのカスタマイズ不可

### 5.3 スマートフォンの台頭と衰退 (2007-)

2007年の iPhone と 2008年の Android の登場は、Java ME の存在意義を根本から覆した。

**Java ME が敗れた理由:**
1. **UI パラダイムの不一致:** MIDP の Canvas/Form はテンキー操作前提。タッチスクリーンに対応できなかった
2. **API の貧弱さ:** スマートフォンの GPS、カメラ、加速度センサー等のリッチなハードウェアに対応する標準 API がなかった
3. **開発者体験の差:** Android は Java SE に近い開発体験を提供。Java ME の制約的な API とは比較にならなかった
4. **エコシステムの分断:** 端末ごとの API 差異、画面サイズ差異への対応が困難だった
5. **パフォーマンス:** KVM のインタプリタ実行では、ネイティブアプリや Dalvik の JIT には太刀打ちできなかった

### 5.4 Java ME の現在

**実質的にはほぼ終了した技術** である。

- Oracle は **Java ME Embedded** を IoT 向けに提供していたが、積極的な開発は停止している
- Java ME 8 (2014) が最後のメジャーリリース
- 新規の Java ME アプリ開発はほぼ皆無
- 組み込み/IoT 分野では、GraalVM Native Image や Eclipse IoT プロジェクトが代替選択肢となっている
- フィーチャーフォン (KaiOS) で Java ME アプリが動く端末がアフリカ・インド等で一部残存しているが、新規エコシステムの発展はない

> **歴史的意義:** Java ME は「世界で最も広く展開されたアプリケーションプラットフォーム」だった事実は重要である。30 億台以上のデバイスに搭載された実績は、Java の "Write Once, Run Anywhere" がモバイル時代に一定の成功を収めたことを示している。しかし、スマートフォン革命という非連続的な変化には対応できなかった。

## 8. よくある混乱の整理

### Q1: 「Java EE は死んだのか?」

**A: 死んでいない。Jakarta EE として活発に開発が続いている。**

「Java EE」という**名前**は確かに使われなくなった。しかし仕様自体は Eclipse Foundation に移管され、**Jakarta EE** として進化を続けている。Jakarta EE 11 (2024) は Java SE 21 を最低要件とし、Virtual Threads や Jakarta Data など最新の機能を取り込んでいる。

ただし、かつての「Java EE vs Spring」という対立構図はもはや意味をなさない。Spring Boot は Jakarta EE の仕様（Servlet、JPA、Bean Validation 等）を積極的に利用しており、両者は「競争」よりも「共存・補完」の関係にある。

### Q2: 「javax.* と jakarta.* の違いは?」

**A: 同じ仕様の、Oracle 時代の名前空間と Eclipse Foundation 時代の名前空間。**

```
javax.servlet.http.HttpServlet    ← Java EE 8 以前 (Oracle 時代)
jakarta.servlet.http.HttpServlet  ← Jakarta EE 9 以降 (Eclipse 時代)
```

Oracle が「Java」の商標を Eclipse に許諾しなかったため、`javax` (Java Extension の略) を `jakarta` に変更する必要があった。API の設計や機能はほぼ同じだが、パッケージ名が異なるため**バイナリ互換性はない**。

**注意:** `javax.crypto`、`javax.net.ssl`、`javax.sql` など **Java SE に属する `javax.*`** パッケージはこの変更の対象外であり、そのまま使い続けられる。変更されたのは Java EE (エンタープライズ) 由来の `javax.*` パッケージのみである。

### Q3: 「Spring と Java EE/Jakarta EE の関係は?」

**A: Spring は Java EE の仕様を「利用する」フレームワークであり、「代替」ではなく「上位レイヤー」。**

```
┌─────────────────────────────────────────────────────┐
│              Spring Boot アプリケーション             │
│  ┌───────────────────────────────────────────────┐  │
│  │         Spring Framework 独自の機能            │  │
│  │  ・Spring MVC (Web フレームワーク)             │  │
│  │  ・Spring DI (依存性注入)                      │  │
│  │  ・Spring Security                            │  │
│  │  ・Spring Data                                │  │
│  │  ・Spring Boot Auto-configuration             │  │
│  └──────────────────────┬────────────────────────┘  │
│                         │ 利用                       │
│  ┌──────────────────────▼────────────────────────┐  │
│  │      Jakarta EE 仕様 (Spring が利用する部分)   │  │
│  │  ・jakarta.servlet (Servlet API)              │  │
│  │  ・jakarta.persistence (JPA)                  │  │
│  │  ・jakarta.validation (Bean Validation)       │  │
│  │  ・jakarta.transaction (JTA)                  │  │
│  │  ・jakarta.inject (DI 標準)                   │  │
│  └───────────────────────────────────────────────┘  │
│                         │ 実行基盤                   │
│  ┌──────────────────────▼────────────────────────┐  │
│  │              Java SE 25 (OpenJDK)             │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

### Q4: 「Android は Java なのか?」

**A: 言語は Java (と Kotlin) だが、プラットフォームは Java SE/ME のいずれでもない。**

Android は複雑な歴史を持つ。

- **言語:** Java (と Kotlin) を使用。Java SE の言語仕様に準拠
- **API:** Java SE のクラスライブラリの**一部**を再実装 (`java.util`, `java.io` 等) + Android 独自 API
- **VM:** 初期は Dalvik VM、Android 5.0+ は ART (Android Runtime)。いずれも JVM ではなく、DEX バイトコードを実行する独自のランタイム
- **Java ME との関係:** なし。MIDP/CLDC とは一切の互換性がない
- **Oracle との訴訟:** Oracle は Google に対し Java API の著作権侵害を主張。2021年に最高裁判所が Google のフェアユース（公正使用）を認めて決着

```
  Java SE / OpenJDK          Android
  ─────────────────          ────────
  .java ソース                .java / .kt ソース
       │                          │
       ▼                          ▼
  javac → .class ファイル    javac/kotlinc → .class → d8 → .dex
       │                                                  │
       ▼                                                  ▼
  JVM (HotSpot) が実行      ART (Android Runtime) が実行
  JVM バイトコード           DEX バイトコード
```

### Q5: 「Java SE と JDK の違いは?」

**A: Java SE は「仕様」、JDK は「実装」。**

| 用語 | 何か | 具体例 |
|------|------|--------|
| **Java SE** | 仕様。JLS (言語仕様) + JVMS (VM 仕様) + API 仕様 | Java SE 25 Specification |
| **JDK** | Java SE の実装。コンパイラ + ランタイム + ツール | OpenJDK 25, Azul Zulu 25 |
| **JRE** | JDK のサブセット (実行環境のみ、開発ツールなし) | Java 8 まで別配布。Java 9+ では JDK に統合 |
| **OpenJDK** | Java SE の参照実装 (オープンソース) | https://openjdk.org |

> 詳しくは [JDK ディストリビューション比較](jdk-distributions-overview.md) を参照。
