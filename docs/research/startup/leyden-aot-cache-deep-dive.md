## 概要

本ドキュメントは、Project Leyden が推進する **AOT (Ahead-of-Time) キャッシュ**の仕組みを詳細に解説するものである。Java の起動時間とウォームアップ時間の短縮という長年の課題に対し、CDS (Class Data Sharing) から AOT Cache への進化を追い、JEP 483 (Ahead-of-Time Class Loading & Linking) を中心に、Java 25 時点での全体像を整理する。

**対象層:** Java 層 / HotSpot C++ 層（CDS / AOT サブシステム）

**対象ランタイム:** Azul Zulu 25 (OpenJDK 25 ベース / HotSpot VM)

**前提知識:** [jvm-overview.md](../jvm-overview.md) で解説した JVM の全体構造（クラスローダー、実行エンジン、ランタイムデータエリア）を前提とする。特に [classloader-deep-dive.md](../runtime/classloader-deep-dive.md) のクラスのロード・リンク・初期化フローの理解が役に立つ。

## 第1章: Project Leyden の背景と目標

### 1.1 Java の起動時間はなぜ遅いのか

Java アプリケーションの起動が遅い原因は、JVM が起動時に大量の作業を「毎回ゼロから」やり直していることにある。以下の図でその全体像を示す。

```
┌─────────────────────────────────────────────────────────────────┐
│                  Java アプリケーション起動の流れ                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. JVM 初期化                                                   │
│     ├─ HotSpot VM のブートストラップ                               │
│     ├─ ヒープ / メタスペースの確保                                  │
│     └─ GC スレッドの起動                                          │
│                                                                 │
│  2. クラスのロード（数千〜数万クラス）          ← 最大のボトルネック   │
│     ├─ JAR / JMOD からバイトコードを読み込み                       │
│     ├─ 各クラスの検証 (Verification)                              │
│     ├─ リンク (Resolution / Preparation)                         │
│     └─ 初期化 (static initializer の実行)                        │
│                                                                 │
│  3. JIT コンパイル                              ← ウォームアップ   │
│     ├─ インタプリタモードで実行開始                                 │
│     ├─ プロファイルデータを収集                                    │
│     ├─ C1 コンパイル（Tier 1-3）                                  │
│     └─ C2 コンパイル（Tier 4）                                    │
│                                                                 │
│  4. アプリケーション固有の初期化                                    │
│     ├─ DI コンテナの構築（Spring 等）                              │
│     ├─ リフレクションによるメタデータ収集                           │
│     └─ コネクションプールの確立                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**具体的な数字で見る問題:**

| 処理 | 典型的な所要時間 | 備考 |
|------|-----------------|------|
| JVM 起動 + java.base のロード | 50〜100ms | 約 1,200 クラス |
| Spring Boot アプリの起動 | 2〜10秒 | 10,000〜30,000 クラス |
| JIT によるピークパフォーマンス到達 | 30秒〜数分 | C2 の最適化完了まで |

問題の本質は、**クラスのロード・リンク・検証** がアプリケーションの起動ごとに毎回実行されることにある。たとえば `java.lang.String` の検証は起動するたびに同じ結果になるのに、毎回やり直している。この「繰り返しの無駄」を排除するのが Project Leyden の核心的なアイデアである。

### 1.2 Project Leyden とは

**Project Leyden** は、Java の起動時間 (startup time) とウォームアップ時間 (warmup time) を大幅に短縮するための OpenJDK のプロジェクトである。2020年に Mark Reinhold によって提案された。

> "Leyden" という名前は、1745年に発明された最初のコンデンサ（蓄電器）である**ライデン瓶** (Leyden jar) に由来する。電荷を「蓄えて」後で使うように、起動時に必要な情報を「蓄えて」後で使うというメタファーである。

**目標:**
- Java の起動時間を桁違いに短縮する
- ウォームアップ時間（JIT がピークパフォーマンスに達するまでの時間）を短縮する
- **Java の既存エコシステムとの完全な互換性を維持する**（ここが最重要）

最後の点が GraalVM Native Image との決定的な違いである（1.5 節で詳述）。

### 1.3 "Shift and Constrain" コンセプト

Project Leyden のアーキテクチャ上の中核概念は **"Shift and Constrain"**（シフトと制約）である。

```
                         "Shift and Constrain"

  ┌──────────────────────────────────────────────────────────────┐
  │                                                              │
  │   【従来の Java】                                              │
  │                                                              │
  │   ソースコード → コンパイル → JAR → 毎回ゼロから起動              │
  │                  (javac)            ┌─────────────────────┐  │
  │                                     │ Load → Link → Init │  │
  │                                     │ JIT コンパイル       │  │
  │                                     │ すべて実行時に実施   │  │
  │                                     └─────────────────────┘  │
  │                                                              │
  │   【Leyden: Shift（時間軸のシフト）】                           │
  │                                                              │
  │   ソースコード → コンパイル → JAR → Training → AOT Cache → 高速起動│
  │                  (javac)              Run                     │
  │                                      ↓                       │
  │                              ┌───────────────────┐           │
  │                              │ Load + Link 済み  │           │
  │                              │ JIT コード保存    │           │
  │                              │ ヒープ snapshot   │           │
  │                              └───────────────────┘           │
  │                                                              │
  └──────────────────────────────────────────────────────────────┘
```

**Shift（シフト）:**
実行時 (run time) に行っていた作業を、それより前の時点に「シフト」する。具体的には:
- クラスのロード・リンクを **事前に** 実行してキャッシュに保存
- JIT コンパイルの結果を **事前に** 保存
- ヒープ上のオブジェクトを **事前に** 生成して保存

**Constrain（制約）:**
シフトを可能にするために、プログラムの振る舞いに一定の **制約** を課す。たとえば:
- AOT キャッシュ生成時と利用時で、クラスパスが一致する必要がある
- クラスの再定義（Instrumentation）は制限される場合がある
- シフトする範囲が大きいほど、制約も厳しくなる

Leyden の巧みな点は、**制約のレベルを段階的に選べる** ことにある。制約を少なくすれば互換性は高いが効果も限定的、制約を多くすれば効果は大きいが互換性に影響する。開発者が自分のアプリケーションに合ったレベルを選択できる。

### 1.4 Condenser（コンデンサ）の概念

**Condenser（コンデンサ）** は、Leyden のアーキテクチャにおける中心的な抽象概念である。名前の通り「凝縮する」という意味で、実行時の状態を凝縮してキャッシュに保存する変換器のことを指す。

```
┌─────────────────────────────────────────────────────────────┐
│                 Condenser パイプライン                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   アプリケーション                                            │
│         │                                                   │
│         ▼                                                   │
│   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐      │
│   │ Condenser 1 │──▶│ Condenser 2 │──▶│ Condenser 3 │      │
│   │ クラスの     │   │ リンク結果   │   │ JIT コード  │      │
│   │ ロード      │   │ の保存      │   │ の保存      │      │
│   └─────────────┘   └─────────────┘   └─────────────┘      │
│         │                 │                 │               │
│         ▼                 ▼                 ▼               │
│   ┌─────────────────────────────────────────────────┐       │
│   │            AOT Cache ファイル (.aot)              │       │
│   │  ┌──────────┬──────────┬──────────┬──────────┐  │       │
│   │  │ クラス   │ リンク    │ JIT     │ ヒープ   │  │       │
│   │  │ メタデータ│ 情報     │ コード  │ オブジェクト│  │       │
│   │  └──────────┴──────────┴──────────┴──────────┘  │       │
│   └─────────────────────────────────────────────────┘       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

各 Condenser は独立した変換ステップであり、パイプラインとして連結される。これにより:

1. **段階的な導入が可能:** すべての Condenser を一度に使う必要はない
2. **新しい最適化の追加が容易:** 新しい Condenser を追加するだけでパイプラインを拡張できる
3. **制約のスコープが明確:** 各 Condenser がどの制約を必要とするかが明示される

### 1.5 GraalVM Native Image との違い

Project Leyden と GraalVM Native Image はどちらも「起動を速くする」ことを目指すが、アプローチが根本的に異なる。

```
┌───────────────────────────────────────────────────────────────┐
│              GraalVM Native Image                             │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  Java コード ──▶ 静的解析 (Closed-World) ──▶ ネイティブバイナリ  │
│                                                               │
│  特徴:                                                        │
│  ・Closed-World Assumption: 実行時にロード可能なクラスを          │
│    ビルド時にすべて決定する                                      │
│  ・JVM なし: HotSpot を含まないネイティブ実行ファイルを生成       │
│  ・リフレクション: 事前に設定ファイルで指定が必要                  │
│  ・動的クラスロード: 原則不可                                    │
│                                                               │
└───────────────────────────────────────────────────────────────┘

┌───────────────────────────────────────────────────────────────┐
│              Project Leyden (AOT Cache)                       │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  Java コード ──▶ Training Run ──▶ AOT Cache ──▶ HotSpot で実行│
│                                                               │
│  特徴:                                                        │
│  ・Open-World: 実行時に新しいクラスをロード可能                   │
│  ・JVM あり: 通常の HotSpot VM 上で実行                         │
│  ・リフレクション: 制限なし（通常通り動作）                       │
│  ・動的クラスロード: 可能                                       │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

**比較表:**

| 項目 | GraalVM Native Image | Project Leyden |
|------|---------------------|----------------|
| 実行モデル | Closed-World | Open-World |
| 実行環境 | SubstrateVM（独自ランタイム） | HotSpot VM |
| リフレクション | 設定ファイルが必要 | 制限なし |
| 動的クラスロード | 原則不可 | 可能 |
| JIT コンパイル | なし（AOT のみ） | あり（AOT + JIT のハイブリッド） |
| ピークパフォーマンス | JIT がないため限定的 | JIT により最大パフォーマンス到達 |
| 起動時間の短縮幅 | 非常に大きい（10〜100倍） | 大きい（2〜5倍、将来さらに改善） |
| 互換性 | 多くの制約あり | 高い互換性 |
| ビルド時間 | 長い（数分〜十数分） | 短い（数秒〜数十秒） |
| デバッグ | 困難 | 通常通り |

**重要なポイント:** Leyden は「GraalVM Native Image の代替」ではなく、**「通常の HotSpot 実行を高速化するもの」** である。両者は排他的ではなく、ユースケースに応じて使い分ける（第9章で詳述）。

## 第2章: CDS から AOT Cache への進化

### 2.1 CDS (Class Data Sharing) の歴史

**CDS (Class Data Sharing)** は、Java 5 で導入されたクラスデータの共有メカニズムである。Project Leyden の AOT Cache は、この CDS を基盤として進化したものだ。

CDS のアイデアは単純である。「クラスのロード・リンク・検証の結果を**ファイルに保存**し、次回起動時にそのファイルを**メモリマップ**して再利用する」というものだ。

```
┌───────────────────────────────────────────────────────────────┐
│                   CDS の基本原理                                │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│   【初回起動（CDS なし）】                                      │
│                                                               │
│   JAR ──▶ バイトコード読込 ──▶ 検証 ──▶ リンク ──▶ メタスペース  │
│            (I/O コスト大)     (CPU)    (CPU)                    │
│                                                               │
│   【2回目以降（CDS あり）】                                     │
│                                                               │
│   CDS アーカイブ ──▶ mmap() ──▶ メタスペース                    │
│   (.jsa ファイル)     (ほぼ瞬時)                                │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 2.2 Static CDS の仕組み

Java 5〜11 で提供されていた初期の CDS は **Static CDS** と呼ばれ、JDK の基盤クラス（`java.base` モジュール等）のみを対象としていた。

```bash
# Static CDS アーカイブの生成（JDK のインストール時に自動実行される）
java -Xshare:dump

# 生成されるファイル
$JAVA_HOME/lib/server/classes.jsa

# 利用（デフォルトで有効）
java -Xshare:on MyApp     # 明示的に有効化
java -Xshare:auto MyApp   # デフォルト（CDS ファイルがあれば使う）
java -Xshare:off MyApp    # 無効化
```

**Static CDS の限界:**
- JDK のクラスのみが対象で、アプリケーションクラスは含められない
- アーカイブの内容が JDK ビルド時に固定される
- 効果は限定的（数十ミリ秒の改善）

### 2.3 Dynamic CDS / AppCDS

**Java 10** で **AppCDS (Application Class Data Sharing)** が導入され、**アプリケーションクラス**もキャッシュ対象に含められるようになった。さらに **Java 13** で **Dynamic CDS** が導入され、クラスリストを手動で作成する手間が不要になった。

```
┌───────────────────────────────────────────────────────────────┐
│                CDS の進化                                       │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  Java 5  : Static CDS      → JDK クラスのみ                    │
│  Java 10 : AppCDS           → アプリクラスも対象（クラスリスト要）│
│  Java 13 : Dynamic CDS      → クラスリスト不要（自動記録）       │
│  Java 19 : デフォルト CDS 改善                                  │
│  Java 24 : AOT Cache (JEP 483) → CDS を超えた包括的キャッシュ   │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

**Dynamic CDS の使い方（Java 13〜23）:**

```bash
# Step 1: アプリを実行してクラスリストを自動記録
java -XX:ArchiveClassesAtExit=app.jsa -jar myapp.jar

# Step 2: 次回起動時にキャッシュを利用
java -XX:SharedArchiveFile=app.jsa -jar myapp.jar
```

### 2.4 AOT Cache への進化 (Java 24+)

**Java 24** で JEP 483 (Ahead-of-Time Class Loading & Linking) が正式に導入され、CDS は **AOT Cache** へと大きく進化した。従来の CDS が「クラスメタデータのキャッシュ」だったのに対し、AOT Cache は以下を包括的にキャッシュする統合的な仕組みである:

- **クラスメタデータ**（従来の CDS と同等）
- **リンク結果**（定数プールの解決など）← JEP 483 で追加
- **ヒープオブジェクト**（String インスタンスなど）← 拡張中
- **JIT コンパイル済みコード**（将来的に）← 開発中

```
┌───────────────────────────────────────────────────────────────┐
│         CDS → AOT Cache の進化                                 │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  ┌─────────────────┐     ┌──────────────────────────────┐     │
│  │  従来の CDS      │     │  AOT Cache (Java 24+)        │     │
│  │  (.jsa ファイル)  │     │  (.aot ファイル)              │     │
│  │                  │     │                              │     │
```

## 第3章: AOT Class Loading & Linking (JEP 483)

### 3.1 JEP 483 の概要

**JEP 483: Ahead-of-Time Class Loading & Linking** は、Java 24 で正式 (GA) として導入され、Java 25 でも引き続き利用可能な機能である。

このJEP の核心は、**クラスのロードとリンクの結果を事前にキャッシュし、次回起動時にキャッシュから瞬時に復元する**ことで起動時間を短縮することにある。

従来の CDS がクラスの**バイトコード表現のパース結果**（InstanceKlass の構造体）をキャッシュしていたのに対し、JEP 483 はさらに踏み込んで **リンクの結果** もキャッシュする。リンクとは、定数プール中のシンボリック参照を実際のメモリアドレスに解決する処理のことである。

```
┌───────────────────────────────────────────────────────────────┐
│              JEP 483 が追加でキャッシュする内容                   │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│   従来の CDS:                                                  │
│     ✓ クラスファイルのパース結果（InstanceKlass）                │
│     ✓ メソッドのバイトコード                                    │
│     ✓ フィールドレイアウト                                      │
│     ✓ vtable / itable の構造                                   │
│                                                               │
│   JEP 483 で追加:                                              │
│     ✓ 定数プールエントリの解決結果（Constant Pool Resolution）   │
│       - クラス参照 → 実際の Klass* ポインタ                     │
│       - フィールド参照 → オフセット値                            │
│       - メソッド参照 → Method* ポインタ                         │
│     ✓ クラス階層の検証結果                                      │
│     ✓ バイトコードリライティングの結果                           │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 3.2 3ステップワークフロー

JEP 483 では、AOT Cache の利用を **3ステップのワークフロー** で行う。`-XX:AOTMode` フラグと `-XX:AOTCacheOutput` / `-XX:AOTCache` フラグで制御する。

```

│                   3ステップワークフロー                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │

│  ┌─────────────────────────────────────────────────────────┐    │
│  │ java -XX:AOTMode=record                                 │    │
│  │      -XX:AOTCacheOutput=app.aot                         │    │
│  │      -jar myapp.jar                                     │    │
│  │                                                         │    │
│  │ → アプリを実行し、ロード/リンクされたクラスを記録         │    │
│  │ → app.aot.config が生成される                            │    │
│  └─────────────────────────────────────────────────────────┘    │
│         │                                                       │
│         ▼ app.aot.config                                        │
│  Step 2: Create（キャッシュ生成）                                │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ java -XX:AOTMode=create                                 │    │
│  │      -XX:AOTCacheOutput=app.aot                         │    │
│  │      -jar myapp.jar                                     │    │
│  │                                                         │    │
│  │ → config を読み込み、AOT Cache (.aot) を生成             │    │
│  │ → アプリケーションコードは実行されない                    │    │
│  └─────────────────────────────────────────────────────────┘    │
│         │                                                       │
│         ▼ app.aot                                               │
│  Step 3: On（キャッシュ利用）                                    │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ java -XX:AOTMode=on                                     │    │
│  │      -XX:AOTCache=app.aot                               │    │
│  │      -jar myapp.jar                                     │    │
│  │                                                         │    │
│  │ → AOT Cache からクラスを瞬時にロード                     │    │
│  │ → 起動時間が大幅に短縮                                   │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3 トレーニングラン（Training Run）の詳細

**トレーニングラン** は、アプリケーションを通常通り実行しながら、JVM がどのクラスをロード・リンクしたかを記録するステップである。

```bash
# 基本的なトレーニングラン
java -XX:AOTMode=record \
     -XX:AOTCacheOutput=app.aot \
     -jar myapp.jar

# Spring Boot アプリの場合
java -XX:AOTMode=record \
     -XX:AOTCacheOutput=app.aot \
     -jar myapp.jar &
# アプリが起動したらリクエストを送って主要パスを実行
curl http://localhost:8080/api/health
curl http://localhost:8080/api/users
# 完了したらアプリを停止
kill %1
```

**トレーニングランの要点:**
- アプリケーションの**典型的なワークロード**を実行することが重要
- 実行されなかったコードパスのクラスはキャッシュに含まれない
- ただし、含まれなかったクラスは通常通り実行時にロードされるため、**動作の正しさには影響しない**
- トレーニングランの出力は `.aot.config` という中間ファイルに記録される

### 3.4 AOT キャッシュの生成

**Create ステップ** では、トレーニングランで記録した情報をもとに AOT Cache ファイルを生成する。

```bash
java -XX:AOTMode=create \
     -XX:AOTCacheOutput=app.aot \
     -jar myapp.jar
```

このステップでは:
1. `.aot.config` を読み込む
2. 記録されたクラスをロード・リンク・検証する
3. 結果をメモリマップ可能な形式で `.aot` ファイルに書き出す
4. **アプリケーションの `main` メソッドは実行されない**

生成される `.aot` ファイルのサイズは、含まれるクラス数に依存する。典型的な Spring Boot アプリでは数十〜数百 MB になる。

### 3.5 AOT キャッシュの利用

```bash
java -XX:AOTMode=on \
     -XX:AOTCache=app.aot \
     -jar myapp.jar
```

利用時の挙動:
1. `.aot` ファイルを `mmap()` でメモリにマップ
2. キャッシュ済みクラスはロード・リンク・検証をスキップ
3. キャッシュに含まれないクラスは通常通りロード

```
┌───────────────────────────────────────────────────────────────┐
│               AOT Cache 利用時のクラスロードフロー              │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│   クラスロード要求                                             │
│        │                                                      │
│        ▼                                                      │
│   ┌──────────────┐                                            │
│   │ AOT Cache に │── Yes ──▶ キャッシュから復元（mmap）        │
│   │ 存在する?    │            ・ロード不要                     │
│   └──────┬───────┘            ・リンク不要                     │
│          │                    ・検証不要                       │
│         No                    ・ほぼゼロコスト                 │
│          │                                                    │
│          ▼                                                    │
│   通常のクラスロード                                           │
│   ・JAR からバイトコード読込                                   │
│   ・パース → 検証 → リンク                                    │
│   ・（従来通りの処理）                                         │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 3.6 キャッシュに含まれる情報

AOT Cache ファイル（`.aot`）には以下の情報が含まれる:

| カテゴリ | 内容 | 詳細 |
|---------|------|------|
| クラスメタデータ | InstanceKlass 構造体 | フィールドレイアウト、メソッドテーブル、vtable/itable |
| 定数プール | 解決済みエントリ | クラス参照、フィールドオフセット、メソッドポインタ |
| バイトコード | リライト済みバイトコード | `_fast_aload_0` 等の最適化バイトコード |
| 文字列 | intern された String | リテラル文字列の共有 |
| クラス階層 | 検証結果 | スーパークラス / インタフェースの整合性 |
| メソッドデータ | Method 構造体 | シグネチャ、例外テーブル、ラインナンバーテーブル |

### 3.7 制約と注意点

**AOT Cache 使用時の制約:**

1. **クラスパスの一致:** トレーニングラン、キャッシュ生成、利用時のクラスパスは**完全に一致**する必要がある（JAR ファイルのパスと内容）。一致しない場合、キャッシュは無効化される。

2. **JVM バージョンの一致:** キャッシュ生成時と利用時の JVM バージョンは同一でなければならない。パッチバージョンの違いでもキャッシュは無効化される。

3. **一部の JVM フラグの一致:** ヒープサイズなどの一部のフラグは、キャッシュ生成時と利用時で同じ値が必要。

4. **Agent の影響:** `-javaagent` で bytecode instrumentation を行うエージェントは、トレーニングラン時にも指定する必要がある。

5. **グレースフルデグラデーション:** キャッシュが無効（バージョン不一致等）の場合、JVM はキャッシュを無視して通常通り起動する。エラーにはならない。

```
┌───────────────────────────────────────────────────────────────┐
│             AOT Cache の無効化条件                               │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  ✗ JVM バージョンが異なる                                      │
│  ✗ クラスパス (JAR) の内容が変更された                          │
│  ✗ 互換性のない JVM フラグが変更された                          │
│  ✗ .aot ファイルが破損している                                  │
│                                                               │
│  → いずれの場合も、JVM は通常起動にフォールバック               │
│  → アプリケーションの正しさには影響しない                       │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

## 第4章: AOT Object Caching

### 4.1 ヒープオブジェクトのキャッシュとは

AOT Object Caching は、**トレーニングラン中に Java ヒープ上に生成されたオブジェクト**を AOT Cache に保存し、次回起動時にヒープ上に直接復元する仕組みである。これにより、起動時にオブジェクトを再生成するコストを削減できる。

```
┌───────────────────────────────────────────────────────────────┐
│             AOT Object Caching の仕組み                         │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  【従来】                                                      │
│  起動 → クラスロード → static 初期化 → オブジェクト生成         │
│                                          │                    │
│                                    ┌─────┴─────┐             │
│                                    │ String    │             │
│                                    │ リフレクション│            │
│                                    │ Enum      │             │
│                                    │ ...       │             │
│                                    └───────────┘             │
│                                                               │
│  【AOT Object Caching】                                       │
│  起動 → mmap() で AOT Cache をマップ                           │
│         │                                                     │
│         ▼                                                     │
│  ┌──────────────────────────────────┐                         │
│  │ AOT Cache のヒープ領域           │                         │
│  │  ┌─────────┬──────────────┐     │                         │
│  │  │ String  │ リフレクション │     │ → ヒープ上に直接復元    │
│  │  │ 定数    │ データ       │     │   (再生成不要)          │
│  │  └─────────┴──────────────┘     │                         │
│  └──────────────────────────────────┘                         │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 4.2 String のキャッシュ

Java アプリケーションの起動時に大量の `String` オブジェクトが生成される。クラスの定数プールに含まれるリテラル文字列、リフレクション API が返すメソッド名・パラメータ名、アノテーション値などが代表的である。

AOT Cache はこれらの `String` インスタンスを**ヒープスナップショット**としてキャッシュに保存する。これは従来の CDS でもある程度行われていたが（Shared Strings）、AOT Cache ではスコープがアプリケーション全体のクラスに拡大されている。

**キャッシュされる String の例:**
- クラス名、メソッド名、フィールド名（リフレクション用）
- アノテーションの文字列値（`@RequestMapping("/api/users")` 等）
- `String.intern()` で内部化された文字列
- 定数プール中のリテラル文字列

### 4.3 リフレクションデータのキャッシュ

Java のリフレクション API（`Class.getDeclaredMethods()` 等）は、呼び出すたびに内部的にかなりのオブジェクト生成を行う。AOT Object Caching では、これらのリフレクション関連オブジェクトもキャッシュの対象である。

これは特に Spring Framework のような DI コンテナにとって大きな効果がある。Spring は起動時に大量のリフレクション処理を行い、コンポーネントスキャン、アノテーション解析、Bean 定義の構築などで数万のリフレクション関連オブジェクトを生成する。

### 4.4 Java 25 での状態

**Java 25 時点での AOT Object Caching の状態:**

| 機能 | 状態 | 備考 |
|------|------|------|
| 基本的な String キャッシュ | **実装済み** | CDS 由来。AOT Cache でも有効 |
| アプリケーション String の拡張キャッシュ | **開発中** | Java 25 mainline で改善が進行 |
| リフレクションデータのキャッシュ | **開発中** | JEP としてはまだ正式提案されていない |
| 任意ヒープオブジェクトのキャッシュ | **研究段階** | 将来の目標 |

> **注意:** Java 25 時点では、AOT Object Caching は包括的な JEP として正式に提案・確定されていない。CDS に組み込まれた String/シンボルの共有は利用可能だが、アプリケーション全体のヒープオブジェクトをキャッシュする機能は開発途上である。OpenJDK mainline では関連する改善が段階的にマージされている。

## 第5章: AOT コードキャッシュ

### 5.1 JIT コンパイル済みコードの事前キャッシュ

JIT コンパイルは Java のピークパフォーマンスを支える重要な仕組みだが、ウォームアップ時間の主要因でもある。AOT コードキャッシュは、**トレーニングラン中に JIT コンパイルされたネイティブコードを AOT Cache に保存**し、次回起動時に再コンパイルなしで利用する仕組みである。

AOT コードキャッシュの重要な側面として、**プロファイルデータの保存**がある。JIT コンパイラ（特に C2）は、インタプリタとC1 が収集したプロファイルデータ（分岐の頻度、型プロファイル、呼び出し回数など）に基づいて最適化を行う。

## 第6章: 実践的な使い方

### 6.1 基本の3ステップ

```bash
# ===== Step 1: Training Run（トレーニングラン）=====
# アプリケーションを通常通り実行し、クラスのロード情報を記録する
java -XX:AOTMode=record \
     -XX:AOTCacheOutput=app.aot \
     -jar myapp.jar

# ===== Step 2: Create（AOT Cache 生成）=====
# 記録した情報から AOT Cache を生成する
# ※ アプリケーションの main() は実行されない
java -XX:AOTMode=create \
     -XX:AOTCacheOutput=app.aot \
     -jar myapp.jar

# ===== Step 3: Production Run（本番実行）=====
# AOT Cache を利用して高速に起動する
java -XX:AOTMode=on \
     -XX:AOTCache=app.aot \
     -jar myapp.jar
```

**1コマンドの自動キャッシュ（Java 24+ の簡易モード）:**

Java 24 以降では、より簡便な方法も利用できる。JVM がキャッシュの存在を自動的に判断し、なければ生成する:

```bash
# 自動キャッシュ: 初回はキャッシュ生成、2回目以降はキャッシュ利用
java -XX:AOTMode=auto \
     -XX:AOTCacheOutput=app.aot \
     -jar myapp.jar
```

### 6.2 Spring Boot での活用

Spring Boot アプリケーションでは、起動時に大量のクラスロード（10,000〜30,000 クラス）とリフレクション処理が発生するため、AOT Cache の効果が特に大きい。

```bash
# Spring Boot アプリケーションでの AOT Cache 活用

# Step 1: Training Run
# Spring の ApplicationContext が完全に初期化されるまで実行する
java -XX:AOTMode=record \
     -XX:AOTCacheOutput=myapp.aot \

# アプリが起動するまで待つ
sleep 10

# 主要なエンドポイントにリクエストを送り、関連クラスのロードを促す
curl -s http://localhost:8080/actuator/health > /dev/null
curl -s http://localhost:8080/api/users > /dev/null
curl -s http://localhost:8080/api/products > /dev/null

# アプリを停止
kill $APP_PID
wait $APP_PID 2>/dev/null

# Step 2: AOT Cache 生成
java -XX:AOTMode=create \
     -XX:AOTCacheOutput=myapp.aot \
     -jar target/myapp-0.0.1-SNAPSHOT.jar

# Step 3: 本番実行
java -XX:AOTMode=on \
     -XX:AOTCache=myapp.aot \
     -jar target/myapp-0.0.1-SNAPSHOT.jar
```

**Spring Boot 3.4+ の CDS サポート:**

Spring Boot 3.4 以降では、CDS/AOT Cache のサポートが統合されている。`spring-boot:build-aot-cache` Maven ゴールを使うとさらに簡単になる:

```bash
# Spring Boot Maven Plugin による AOT Cache 生成
./mvnw spring-boot:build-aot-cache
```

### 6.3 Quarkus での活用

Quarkus はもともと起動速度を重視した設計だが、AOT Cache との組み合わせでさらなる高速化が可能:

### 6.4 コンテナ環境での活用

コンテナ環境では、Docker のマルチステージビルドを活用して AOT Cache を含むイメージを構築する。

```dockerfile
# ===== Stage 1: ビルド =====
FROM eclipse-temurin:25-jdk AS build
WORKDIR /app
COPY . .
RUN ./mvnw package -DskipTests

# ===== Stage 2: AOT Cache 生成 =====
FROM azul/zulu-openjdk:25 AS aot-cache
WORKDIR /app
COPY --from=build /app/target/myapp.jar .

# Training Run
RUN java -XX:AOTMode=record \
         -XX:AOTCacheOutput=app.aot \
         -jar myapp.jar &\
    APP_PID=$! && \
    sleep 10 && \
    curl -s http://localhost:8080/actuator/health > /dev/null && \
    kill $APP_PID && \
    wait $APP_PID 2>/dev/null || true

# AOT Cache 生成
RUN java -XX:AOTMode=create \
         -XX:AOTCacheOutput=app.aot \
         -jar myapp.jar

# ===== Stage 3: 本番イメージ =====
FROM azul/zulu-openjdk:25-jre AS production
WORKDIR /app
COPY --from=aot-cache /app/myapp.jar .
COPY --from=aot-cache /app/app.aot .

ENTRYPOINT ["java", \
            "-XX:AOTMode=on", \
            "-XX:AOTCache=app.aot", \
            "-jar", "myapp.jar"]
```

**コンテナ環境での注意点:**
- AOT Cache 生成とプロダクション実行で**同じ JVM バージョン** を使うこと
- AOT Cache 生成時の Training Run では、アプリケーションが正常に起動できる環境（DB 接続等）が必要
- `.aot` ファイルはアーキテクチャ固定（`linux/amd64` で生成したキャッシュは `linux/arm64` では使えない）

### 6.5 パフォーマンス測定の方法

```bash
# ===== 起動時間の計測 =====

# 方法1: time コマンド（シンプル）
time java -jar myapp.jar                              # ベースライン
time java -XX:AOTMode=on -XX:AOTCache=app.aot \
     -jar myapp.jar                                    # AOT Cache あり

# 方法2: JVM フラグで詳細計測
java -Xlog:class+load:file=classload.log \
     -XX:AOTMode=on -XX:AOTCache=app.aot \
     -jar myapp.jar

# 方法3: Spring Boot の起動時間ログ
# "Started MyApplication in X.XXX seconds" を比較

# ===== クラスロード統計の確認 =====
java -Xlog:class+load=info:stdout \
     -XX:AOTMode=on -XX:AOTCache=app.aot \
     -jar myapp.jar 2>&1 | head -20
# "source: shared objects file" → AOT Cache から読み込まれたクラス
# "source: file:/path/to.jar"  → 通常ロードされたクラス

# ===== AOT Cache の統計 =====
# キャッシュの内容を確認
java -Xlog:cds -XX:AOTMode=on -XX:AOTCache=app.aot \
     -version
```

AOT Cache ファイル（`.aot` / `.jsa`）は、複数の**リージョン**から構成されるバイナリファイルである。


**FileMapHeader（`filemap.hpp`）の主要フィールド:**


### 7.3 メモリマップドファイルの活用

AOT Cache の高速ロードの鍵は `mmap()` (メモリマップドファイル) にある。

```
┌───────────────────────────────────────────────────────────────┐
│              mmap() による高速ロード                             │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│   【従来のクラスロード】                                        │
│                                                               │
│   JAR (ディスク)                                               │
│     │  read() でユーザ空間にコピー                              │
│     ▼                                                         │
│   バッファ (ユーザ空間)                                         │
│     │  パース                                                  │
│     ▼                                                         │
│   InstanceKlass (ヒープ/メタスペース)                           │
│     │  検証・リンク                                            │
│     ▼                                                         │
│   使用可能状態                                                 │
│                                                               │
│   コスト: I/O + メモリコピー + CPU (パース+検証+リンク)         │
│                                                               │
│                                                               │
│   【mmap() によるロード】                                       │
│                                                               │
│   .aot ファイル (ディスク)                                      │
│     │  mmap() でページテーブルにマッピング                      │
│     ▼                                                         │
│   メタスペース (仮想アドレス空間)                               │
│     │  ページフォルト時にオンデマンドでロード                   │
│     ▼                                                         │
│   使用可能状態（アクセスされたページだけ物理メモリに配置）      │
│                                                               │
│   コスト: mmap() システムコール1回 (ほぼゼロコスト)             │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

**mmap() の利点:**

1. **ゼロコピー:** ファイルの内容がカーネルのページキャッシュから直接参照される。ユーザ空間へのコピーが不要
2. **デマンドページング:** 実際にアクセスされたページだけが物理メモリに読み込まれる
3. **共有メモリ:** 複数の JVM プロセスが同じ `.aot` ファイルをマップすると、物理メモリを共有できる（コンテナ環境で有効）
4. **Read-Only リージョンの共有:** Read-Only Region は `MAP_PRIVATE` ではなく `MAP_SHARED` でマップできるため、プロセス間で物理ページを共有

### 8.1 起動時間の改善

AOT Cache による起動時間の改善は、アプリケーションの種類とクラスのロード数に大きく依存する。以下は OpenJDK コミュニティで報告されている典型的な数値である。

```
│  小規模 CLI ツール       ~200ms   ~130ms    ~70ms     ~3x     │

│  Spring Boot (中)       ~5.0s    ~3.5s     ~1.8s     ~2.8x   │
│  Micronaut              ~1.5s    ~1.0s     ~0.5s     ~3x     │
│  Quarkus                ~1.0s    ~0.7s     ~0.4s     ~2.5x   │
│                                                               │
│  ※ 数値は目安。環境・アプリ構成・JVM バージョンにより変動する  │
```

**改善が大きいケース:**
- ロードされるクラス数が多いアプリケーション（Spring Boot、Jakarta EE）
- リフレクションを多用するフレームワーク
- 短命なプロセス（Lambda、CLI ツール）

**改善が限定的なケース:**
- ロードされるクラス数が少ないシンプルなアプリケーション
- 起動時間の大部分が外部リソース（DB 接続、ネットワーク）の待ちである場合

### 8.3 ウォームアップ時間の短縮

ウォームアップ時間の短縮は、主に将来の AOT コードキャッシュ（第5章）によって実現される見込みである。Java 25 時点では:

- **JEP 483 (AOT Class Loading & Linking) による効果:** クラスのロード・リンクが高速化されることで、JIT コンパイラが「早く仕事を始められる」。間接的にウォームアップの短縮に寄与。
- **将来の AOT コードキャッシュによる効果:** JIT コンパイル済みコードがキャッシュから復元されれば、ウォームアップ自体が不要になる（または大幅に短縮される）。

### 9.2 CRaC との比較

**CRaC (Coordinated Restore at Checkpoint)** は、Azul が主導する OpenJDK プロジェクトで、Linux の CRIU (Checkpoint/Restore In Userspace) を利用して JVM プロセスのスナップショットを取り、復元する方式である。

```
┌───────────────────────────────────────────────────────────────┐
│           CRaC vs Leyden AOT Cache                            │
├───────────────┬──────────────────────┬────────────────────────┤
│ 項目          │ CRaC                 │ Leyden AOT Cache       │
├───────────────┼──────────────────────┼────────────────────────┤
│ 仕組み        │ プロセス全体の        │ クラス/コードの         │
│               │ チェックポイント      │ 事前キャッシュ          │
├───────────────┼──────────────────────┼────────────────────────┤
│ 起動時間      │ 極めて高速            │ 高速                   │
│               │ (プロセス復元)        │ (キャッシュロード)      │
├───────────────┼──────────────────────┼────────────────────────┤
│ OS 依存       │ Linux のみ (CRIU)     │ 全プラットフォーム      │
├───────────────┼──────────────────────┼────────────────────────┤
│ リソース管理  │ 注意が必要             │ 透過的                 │
│               │ (ファイル/ソケット     │ (通常の起動フロー)      │
│               │  の再取得)            │                        │
├───────────────┼──────────────────────┼────────────────────────┤
│ アプリ改修    │ Resource API の        │ 不要                   │
│               │ 実装が必要な場合あり   │                        │
├───────────────┼──────────────────────┼────────────────────────┤
│ セキュリティ  │ メモリダンプに         │ クラスメタデータのみ    │
│               │ 機密情報の可能性       │                        │
├───────────────┼──────────────────────┼────────────────────────┤
│ JDK サポート  │ Azul Zulu で提供       │ OpenJDK mainline       │
│               │ (OpenJDK にも提案中)   │ (Java 24+ で正式)      │
└───────────────┴──────────────────────┴────────────────────────┘
```

**CRaC の独自の利点:**
- JIT コンパイル済み状態、ヒープの内容、すべてが復元される
- 復元後は即座にピークパフォーマンスで動作可能
- ウォームアップが完全にゼロ

**CRaC の制約:**
- Linux 専用（CRIU が Linux カーネルの機能）
- チェックポイント時にオープンしていたファイルやソケットは復元時に再取得が必要
- `Resource` インタフェースの実装が必要になる場合がある
- チェックポイントイメージにヒープの内容（パスワード等）が含まれるセキュリティ上の考慮
