## HotSpot とは何か -- この学習シリーズの「主役」を知る

本シリーズでは「HotSpot C++ 層」という言葉が頻繁に登場する。まず、この HotSpot が何者なのかを整理しておこう。

### Java の「仕様」と「実装」は別物

Java には **JVM Specification（JVM 仕様）** という公式の仕様書がある。これは「JVM はこう動くべきだ」というルールブックであり、メモリ構造やバイトコードの実行規則などが定義されている。しかし、仕様書だけではプログラムは動かない。仕様に従って実際に動く **プログラム** が必要であり、それが **JVM の実装** である。

**HotSpot** は、そのJVM 仕様を実装した **C++ で書かれたプログラム**の名前である。世界で最も広く使われている JVM 実装であり、OpenJDK プロジェクトの中核をなしている。

```
関係図:

JVM 仕様書 (ルールブック)
   │
   │ 「この仕様に従って実装してください」
   │
   ├──→ HotSpot VM (C++ で実装)  ← Oracle OpenJDK / Azul Zulu の中身
   ├──→ Eclipse OpenJ9            ← IBM 由来の別実装
   └──→ GraalVM                   ← Oracle が開発する別アプローチの実装
```

### 名前の由来 -- 「ホットスポット」を見つけて最適化する

プログラムの実行において、すべてのコードが均等に実行されるわけではない。実際には **全体の 20% 程度のコードが、実行時間の 80% を占める** と言われている（パレートの法則）。この「頻繁に実行される箇所」のことを英語で **hot spot（ホットスポット）** と呼ぶ。

HotSpot VM は、プログラム実行中にこのホットスポットを自動的に検出し、その部分だけを **JIT（Just-In-Time）コンパイラ** で機械語に変換して高速化する。つまり、「熱い場所を見つけて最適化する」から HotSpot という名前がついている。

```
プログラム実行の流れ:

  バイトコード (.class)
       │
       ▼
  インタプリタ（最初はそのまま実行）
       │
       │  「このメソッド、もう 10,000 回呼ばれたぞ」 ← ホットスポット検出
       │
       ▼
  JIT コンパイラ（C1/C2）が機械語に変換
       │
       ▼
  次回以降は機械語で直接実行 → 高速！
```

### Oracle OpenJDK も Azul Zulu も、中身は同じ HotSpot

「Oracle の JDK」「Azul Zulu」「Amazon Corretto」「Microsoft Build of OpenJDK」など、JDK のディストリビューションは数多く存在する。しかし、**これらの中身はすべて同じ HotSpot VM がベース** になっている。違いはサポート体制、パッチの適用タイミング、一部の独自機能の有無程度であり、コアのエンジンは共通である。

```
OpenJDK (HotSpot VM のソースコード)
   │
   ├──→ Oracle JDK      (Oracle が配布)
   ├──→ Azul Zulu       (Azul Systems が配布) ← 我々の正装ランタイム
   ├──→ Amazon Corretto  (AWS が配布)
   └──→ Microsoft Build  (Microsoft が配布)
         ...すべて HotSpot VM がベース
```

> **補足:** Azul には商用製品の **Zing/Azul Platform Prime** もあり、こちらは独自の **Falcon JIT コンパイラ** や **C4 GC** を搭載している。ただし、Zulu（OSS 版）は HotSpot そのものである。

### 「HotSpot C++ 層」とは

この学習シリーズで「**HotSpot C++ 層**」と言うとき、それは **JVM 自体のソースコード（C++ で書かれた部分）** を指している。

我々が普段書く Java コードは「Java 層」に属する。そして、その Java コードを実行するエンジン自体は C++ で書かれている。メモリの確保、GC の実行、スレッドの管理、JIT コンパイルなど、JVM の根幹を担う処理はすべてこの C++ 層で行われている。

```
3 つの層:

┌─────────────────────────────────────┐
│         Java 層                      │  ← 我々が書くコード
│  (Java ソースコード / バイトコード)    │    例: UserService.java
├─────────────────────────────────────┤
│         JNI / ネイティブ層            │  ← Java と C++ の橋渡し
│  (Java Native Interface)             │    例: System.arraycopy() の内部
├─────────────────────────────────────┤
│         HotSpot C++ 層               │  ← JVM エンジン本体
│  (C++ ソースコード)                   │    例: thread.cpp, collectedHeap.cpp
└─────────────────────────────────────┘
```

この文書では、各セクションの冒頭に **「対象層」** を明記している。「HotSpot (C++) 層」と書かれている箇所は、JVM エンジンそのもの（C++ の世界）の話をしていると理解してほしい。

---

## 概要

JVM が管理するメモリ領域のうち、最も基本的かつ重要な 2 つが **ヒープ (Heap)** と **スタック (Stack)** である。この 2 つは「何を格納するか」「誰が所有するか」「いつ解放されるか」がまったく異なる。本レポートでは、その違いを物理的な仕組みから Spring Boot 開発の実務まで、段階的に掘り下げる。

**対象層:** 主に Java 層の概念整理を行いつつ、GC やスタック管理については HotSpot (C++) 層の実装にも言及する。

---

## 1. ヒープとスタック -- 役割と「寿命」の違い

### 1.1 スタック -- スレッド専用の「作業台」

**対象層: Java 層 / HotSpot (C++) 層**

スタックは各スレッドが **専有する** メモリ領域で、メソッド呼び出しのたびに **フレーム (Stack Frame)** が積み上げられる。メソッドが終了すると、そのフレームは自動的に取り除かれる。この「後入れ先出し (LIFO: Last-In, First-Out)」の構造がスタックの本質である。

```
スレッド A のスタック（下から上へ積まれる）
┌─────────────────────┐
│ methodC() のフレーム  │ ← 現在実行中（スタックの一番上）
├─────────────────────┤
│ methodB() のフレーム  │
├─────────────────────┤
│ methodA() のフレーム  │
├─────────────────────┤
│ main() のフレーム     │ ← 最初に積まれた
└─────────────────────┘
```

**スタックフレームに格納されるもの:**

| 格納対象 | 具体例 |
|---------|--------|
| プリミティブ型のローカル変数 | `int count = 0;` の `count` の値そのもの (0) |
| オブジェクト参照（Reference） | `var list = new ArrayList<>()` の `list` が保持する参照値（ヒープ上のオブジェクトを指す） |
| メソッドの戻りアドレス | このメソッドが終わったら、呼び出し元のどこに戻るかの情報 |
| オペランドスタック | 計算途中の一時的な値 |

> **重要な区別:** `var list = new ArrayList<>()` と書いたとき、変数 `list` 自体（参照値）はスタックに置かれるが、`ArrayList` オブジェクトの**本体**はヒープに置かれる。スタックにあるのはあくまで「ヒープ上のどこにオブジェクトがあるか」を示す**参照（Reference）** である。
>
> **Java の参照と C/C++ のポインタの違い:** Java の参照は C/C++ のポインタとは明確に異なる概念である。Java の参照ではメモリアドレスを直接操作できず、ポインタ演算（アドレスの加減算）も不可能である。さらに、GC がオブジェクトをヒープ内で移動させた場合でも、Java の参照は自動的に追従し、プログラマがアドレスの変化を意識する必要はない。HotSpot の内部実装（C++ 層）ではオブジェクトへの参照を `oop`（ordinary object pointer）というポインタとして扱っているが、この実装詳細は Java プログラマには完全に隠蔽されている。

### 1.2 ヒープ -- 全スレッド共有の「倉庫」

**対象層: Java 層 / HotSpot (C++) 層**

ヒープは全スレッドから共有されるメモリ領域で、`new` キーワードによって確保されるオブジェクトの本体が置かれる場所である。

```
ヒープ（全スレッドから参照可能）
┌────────────────────────────────────────┐
│  ArrayList@0x1A2B  ←─ スレッドAの list が参照  │
│  String@0x3C4D     ←─ スレッドBの name が参照  │
│  User@0x5E6F       ←─ 誰からも参照されていない  │  ← GC の回収対象
│  HashMap@0x7G8H    ←─ 複数スレッドから参照      │
└────────────────────────────────────────┘
```

**ヒープに格納されるもの:**

| 格納対象 | 具体例 |
|---------|--------|
| オブジェクトのインスタンス本体 | `new User("Alice")` で生成された User オブジェクト |
| 配列の本体 | `new int[100]` の 100 要素分の実データ |
| クラスのインスタンスフィールド | オブジェクト内部の `String name` などのフィールド値 |
| 文字列リテラル（String Pool） | `"hello"` などの文字列定数（ヒープ内の専用領域） |

### 1.3 寿命の対比

| 特性 | スタック | ヒープ |
|------|---------|--------|
| **所有者** | スレッドごとに 1 つ | 全スレッドで共有 |
| **寿命** | メソッドの開始から終了まで | 最後の参照がなくなるまで |
| **解放方法** | スタックポインタの移動（自動） | GC による回収（自動だが非決定的） |
| **サイズ** | 比較的小さい（デフォルト 512KB〜1MB 程度） | 大きい（数百 MB〜数十 GB） |
| **速度** | 非常に高速（ポインタ演算のみ） | スタックより遅い（割り当て・GC コストあり） |
| **断片化** | 発生しない | 発生する可能性がある |

### 1.4 メモリ配置パターン一覧 -- 変数の種類別「どこに何が置かれるか」

**対象層: Java 層（概念モデル）/ HotSpot (C++) 層（実際の配置）**

セクション 1.1〜1.3 で「スタックとヒープは役割が違う」ことを理解した。では、**具体的にどの変数がどこに置かれるのか**。ここでは「変数の種類」ごとに、**変数自体（参照や値を保持する箱）** と **実体（オブジェクト本体や値そのもの）** がそれぞれどのメモリ領域に配置されるかを、一覧表と詳細解説で整理する。

#### 1.4.1 配置パターン早見表

| # | 変数の種類 | 宣言例 | 変数（箱）の場所 | 実体（中身）の場所 | GC 対象？ | 備考 |
|---|-----------|--------|-----------------|-------------------|----------|------|
| 1 | ローカル変数のプリミティブ型 | `int count = 42;` | スタック | スタック（値が直接格納） | いいえ | メソッド終了で自動消滅 |
| 2 | ローカル変数の参照型 | `String name = "hello";` | スタック（参照） | ヒープ（オブジェクト本体） | はい（実体のみ） | 参照はスタック、本体はヒープ |
| 3 | インスタンスフィールドのプリミティブ型 | `class Foo { int x; }` | ヒープ（親オブジェクト内） | ヒープ（親オブジェクト内に値が埋め込み） | はい（親オブジェクトごと） | 親オブジェクトの一部としてヒープに存在 |
| 4 | インスタンスフィールドの参照型 | `class Foo { String name; }` | ヒープ（親オブジェクト内に参照） | ヒープ（参照先オブジェクト） | はい（両方） | 親の中に参照、参照先は別のヒープ領域 |
| 5 | static フィールドのプリミティブ型 | `static int MAX = 100;` | ヒープ（Class オブジェクト内） | ヒープ（Class オブジェクト内に値が埋め込み） | 通常いいえ（※） | クラスがアンロードされない限り生存 |
| 6 | static フィールドの参照型 | `static List<String> cache = ...;` | ヒープ（Class オブジェクト内に参照） | ヒープ（参照先オブジェクト） | 参照先は条件付き（※） | static フィールド自体が GC Root になる |
| 7 | メソッド定義 | `void doSomething() { ... }` | メタスペース（バイトコード） | 実行時はスタック（フレーム）、JIT 後はコードキャッシュ | いいえ | 3 つの場所にまたがる（後述） |

> **※ static フィールドの GC について:** static フィールドは **GC Root**（GC の探索起点）として扱われる。つまり、static フィールドが参照しているオブジェクトは「到達可能」と判定され、GC に回収されない。static フィールド自体がクラスと運命を共にするため、クラスがアンロードされない限り（通常のアプリケーションではほぼアンロードされない）、static フィールドとその参照先は事実上 GC 対象外となる。ただし、static フィールドに `null` を代入すれば、参照先オブジェクトは GC 対象になりうる。

#### 1.4.2 各パターンの詳細解説

##### パターン 1: ローカル変数のプリミティブ型

```
int count = 42;
double rate = 3.14;
boolean active = true;
```

**配置:**

```
スタック（現在のメソッドのフレーム内）
┌───────────────────────┐
│ count  : 42            │  ← 値そのものがスタックに直接格納
│ rate   : 3.14          │
│ active : true (1)      │
└───────────────────────┘

ヒープ: 何も置かれない
```

**なぜスタックなのか（設計上の理由）:**

プリミティブ型のローカル変数は「メソッドの実行中だけ必要」「サイズが固定（int は 4 バイト、long は 8 バイト等）」「他のスレッドから直接アクセスされない」という 3 条件を満たす。この性質は LIFO 構造のスタックと完璧に合致するため、高速なスタック上に直接格納される。ヒープに置く理由がなく、わざわざ GC の管理対象にする必要もない。

**HotSpot の実装詳細:** スタックフレーム内の**ローカル変数配列 (Local Variable Array)** の各スロットに格納される。int/float は 1 スロット（32 ビット）、long/double は 2 スロット（64 ビット）を占有する。JVM 仕様 (Chapter 2.6.1) で定義されたフレーム構造に準拠している。

##### パターン 2: ローカル変数の参照型

```
String name = "hello";
var list = new ArrayList<>();
```

**配置:**

```
スタック                         ヒープ
┌────────────────┐              ┌──────────────────────────┐
│ name : ref ────│─────────────→│ String オブジェクト "hello"│
│ list : ref ────│──────┐       └──────────────────────────┘
└────────────────┘      │       ┌──────────────────────────┐
                        └──────→│ ArrayList オブジェクト     │
                                │  (内部配列 elementData)   │
                                └──────────────────────────┘
```

**これが最重要パターン。** 初学者が最も混乱しやすいのがここである。

- **変数 `name` 自体**（参照値 = ヒープ上のアドレスを保持する箱）→ **スタック**
- **`"hello"` という String オブジェクトの本体** → **ヒープ**（文字列リテラルは String Pool 経由）

参照とは「ヒープ上のどこにオブジェクトがあるかを示す住所メモ」である。住所メモ（参照）はスタック上の作業台に置かれるが、住所の先にある建物（オブジェクト本体）はヒープという「街」に建っている。

**なぜこの分離が必要なのか:** オブジェクトのサイズは可変であり（ArrayList は要素の追加で内部配列が拡張される）、寿命もメソッドの実行期間を超える可能性がある（メソッドの戻り値としてオブジェクトを返す場合など）。スタックは LIFO かつ固定サイズなので、可変サイズ・可変寿命のオブジェクト本体をスタックに置くことはできない。そのため、本体はヒープに、参照（固定サイズ = 通常 64 ビット / Compressed Oops 有効時は 32 ビット）だけをスタックに置く設計になっている。

##### パターン 3: インスタンスフィールドのプリミティブ型

```
class Foo {
    int x = 10;
    double y = 2.5;
}
// メソッド内で: var foo = new Foo();
```

**配置:**

```
スタック                         ヒープ
┌────────────────┐              ┌─────────────────────────────┐
│ foo : ref ─────│─────────────→│ Foo オブジェクト             │
└────────────────┘              │  ┌────────────────────────┐ │
                                │  │ オブジェクトヘッダ      │ │
                                │  ├────────────────────────┤ │
                                │  │ x : 10     (4 bytes)   │ │ ← 値がオブジェクト内に
                                │  │ y : 2.5    (8 bytes)   │ │    直接埋め込まれている
                                │  └────────────────────────┘ │
                                └─────────────────────────────┘
```

パターン 1（ローカル変数のプリミティブ型がスタックに直接格納される）と異なり、**インスタンスフィールドのプリミティブ型はヒープに置かれる**。これは「親オブジェクト（Foo）がヒープに存在するため、その一部であるフィールドも当然ヒープにある」という理屈である。

比喩で言えば、引っ越し用の段ボール箱（Foo オブジェクト）がヒープという「倉庫」に置かれていて、箱の中に入っている文房具（int x, double y）も当然倉庫の中にある。文房具だけがスタック（作業台）に飛び出しているわけではない。

**HotSpot の実装詳細:** オブジェクトのメモリレイアウトは `src/hotspot/share/oops/instanceKlass.cpp` と `src/hotspot/share/oops/fieldInfo.hpp` で管理される。HotSpot はフィールドの並び順を**型サイズの降順**に再配置して、アラインメントのパディングを最小化する最適化を行う（フィールドパッキング）。

##### パターン 4: インスタンスフィールドの参照型

```
class Foo {
    String name = "Alice";
    List<Integer> scores = new ArrayList<>();
}
// メソッド内で: var foo = new Foo();
```

**配置:**

```
スタック                     ヒープ
┌──────────────┐            ┌──────────────────────────────────┐
│ foo : ref ───│───────────→│ Foo オブジェクト                  │
└──────────────┘            │  ┌─────────────────────────────┐ │
                            │  │ オブジェクトヘッダ           │ │
                            │  ├─────────────────────────────┤ │
                            │  │ name   : ref ───│──────────────│──→ String "Alice"
                            │  │ scores : ref ───│──────────────│──→ ArrayList オブジェクト
                            │  └─────────────────────────────┘ │
                            └──────────────────────────────────┘
```

パターン 3 と似ているが、フィールドの型が参照型の場合、親オブジェクトの中に格納されるのは**参照（Reference）** であり、参照先のオブジェクト本体はヒープ上の**別の場所**に存在する。

つまり、ヒープの中でオブジェクト同士が参照を通じて繋がり合い、「オブジェクトグラフ」を形成する。セクション 2.2 で解説した「参照グラフの複雑さ」は、まさにこのパターンの積み重ねで生まれるものである。

##### パターン 5: static フィールドのプリミティブ型

```
class Config {
    static int MAX_RETRY = 3;
    static long TIMEOUT_MS = 5000L;
}
```

**配置:**

```
ヒープ
┌──────────────────────────────────────────────┐
│ java.lang.Class<Config> オブジェクト          │
│  ┌─────────────────────────────────────────┐ │
│  │ MAX_RETRY  : 3        (4 bytes)         │ │ ← Class オブジェクト内に
│  │ TIMEOUT_MS : 5000L    (8 bytes)         │ │    値が埋め込まれている
│  └─────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
         ↑
         │ GC Root として常に到達可能
```

static フィールドはインスタンスに属さず、**クラスそのもの**に属する。HotSpot の実装では、クラスがロードされると `java.lang.Class` オブジェクトがヒープ上に生成され、static フィールドの値はこの **Class オブジェクトの中** に格納される。

**ローカル変数のプリミティブ型との違い:** ローカル変数はスタック上に置かれてメソッド終了時に消えるが、static フィールドはヒープ上の Class オブジェクト内にあり、クラスがアンロードされるまで存続する。事実上、アプリケーションの寿命と同じである。

**よくある混乱: 「static はメタスペースに置かれる」は正確ではない。** メタスペースに格納されるのはクラスの**定義（メタデータ）**—— メソッド定義、フィールド定義、定数プールなど —— であり、static フィールドの**値**はヒープ上の Class オブジェクトに格納される。Java 8 より前の HotSpot では static フィールドの値が PermGen（メソッドエリアの一部）に置かれていたため、この混乱が生じやすい。

> **関連ソース (HotSpot C++):** `src/hotspot/share/oops/instanceKlass.cpp` — `InstanceKlass::allocate_instance()` でインスタンスの確保、`java_lang_Class` で Class オブジェクトと static フィールドの関連が管理される。`src/hotspot/share/classfile/javaClasses.cpp` — `java_lang_Class` の実装。

##### パターン 6: static フィールドの参照型

```
class AppCache {
    static List<String> cache = new ArrayList<>();
    static Map<String, Object> registry = new HashMap<>();
}
```

**配置:**

```
ヒープ
┌──────────────────────────────────────────────────────────┐
│ java.lang.Class<AppCache> オブジェクト                     │
│  ┌──────────────────────────────────────────────────┐    │
│  │ cache    : ref ──────────────→ ArrayList オブジェクト   │
│  │ registry : ref ──────────────→ HashMap オブジェクト     │
│  └──────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────┘
         ↑
         │ GC Root
```

パターン 5 の参照型バージョン。Class オブジェクトの中に参照が格納され、参照先のオブジェクト（ArrayList、HashMap 等）はヒープ上の別の場所に存在する。

**これがセクション 3.3 で紹介した「メモリリーク」の温床になる。** static フィールドは GC Root であるため、ここから参照されているオブジェクトチェーンはすべて「到達可能」と判定される。`cache.add(...)` で要素を追加し続けると、ArrayList の内部配列が拡張され続け、GC がまったく回収できないままヒープを圧迫する。これがセクション 3.3 パターン 1 で示した「static な Collection への無制限な追加」の正体である。

#### 1.4.3 メソッドの「定義」「実行」「最適化」-- 3 つの場所

メソッドは「変数」とは性質が異なるが、「どこに置かれるのか」という疑問は非常に多い。結論として、メソッドは **3 つの異なる場所** にまたがって存在する。

```
┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│  ① メタスペース（定義）                                               │
│  ┌──────────────────────────────────────────────────────┐            │
│  │  クラスメタデータ: UserService.class                  │            │
│  │   ├─ findById() のバイトコード (CONSTANT_Methodref)   │            │
│  │   ├─ save() のバイトコード                            │            │
│  │   ├─ フィールド定義: repository                       │            │
│  │   └─ 定数プール: 文字列リテラル、メソッド参照          │            │
│  └──────────────────────────────────────────────────────┘            │
│       │                                                              │
│       │ findById() が呼ばれた！                                       │
│       ▼                                                              │
│  ② スタック（実行 -- 呼び出しごとにフレームが積まれる）                  │
│  ┌──────────────────────────────────────────────────────┐            │
│  │  findById() のスタックフレーム                        │            │
│  │   ├─ ローカル変数配列: [this, id, result]             │            │
│  │   ├─ オペランドスタック: (計算途中の値)                 │            │
│  │   └─ 戻りアドレス: 呼び出し元に戻るための情報          │            │
│  ├──────────────────────────────────────────────────────┤            │
│  │  main() のスタックフレーム                            │            │
│  └──────────────────────────────────────────────────────┘            │
│       │                                                              │
│       │ 「findById()、もう 10,000 回呼ばれたぞ」                       │
│       ▼                                                              │
│  ③ コードキャッシュ（最適化 -- JIT コンパイル後の機械語）                │
│  ┌──────────────────────────────────────────────────────┐            │
│  │  findById() の最適化済み機械語コード                   │            │
│  │  (C1/C2 JIT コンパイラが生成、次回以降はここから直接実行)│            │
│  └──────────────────────────────────────────────────────┘            │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

**① メタスペース -- メソッドの「設計図」が置かれる場所**

クラスファイル (.class) がクラスローダーによってロードされると、そのバイトコード（メソッドの命令列）やメソッドシグネチャなどの情報はメタスペースに格納される。これは「このメソッドはこういう処理をする」という**定義**であり、何度呼び出しても変わらない静的な情報である。

**② スタック -- メソッドが「実行中」に使う場所**

メソッドが呼び出されるたびに、スタック上に**スタックフレーム**が1つ積まれる。同じメソッドが同時に 100 回呼ばれれば、100 個のフレームがそれぞれのスレッドのスタック上に存在する。定義は 1 つだが、実行のための「作業領域」はそのつど独立して確保される。

比喩で言えば、料理のレシピ（定義 = メタスペース）は 1 枚だけだが、10 人の料理人が同時にそのレシピを見ながら料理すれば、まな板と食材（実行 = スタックフレーム）は 10 セット必要になる。

**③ コードキャッシュ -- 「高速版」が置かれる場所**

HotSpot の JIT コンパイラ（C1/C2）が「このメソッドは頻繁に呼ばれている（ホットスポットだ）」と判断すると、バイトコードを**ネイティブの機械語**に変換し、その結果をコードキャッシュに格納する。以降の呼び出しでは、メタスペースのバイトコードをインタプリタで逐次実行するのではなく、コードキャッシュの機械語を CPU が直接実行するため、大幅に高速化される。

> **関連ソース (HotSpot C++):**
> - `src/hotspot/share/oops/method.cpp` -- メソッドメタデータの管理（メタスペース内）
> - `src/hotspot/share/interpreter/abstractInterpreter.cpp` -- バイトコードの逐次実行
> - `src/hotspot/share/code/codeCache.cpp` -- JIT コンパイル済みコードの管理
> - `src/hotspot/share/compiler/compileBroker.cpp` -- JIT コンパイルの制御

**読み取りのポイント:**

- **左端のスタック** に直接値が置かれるのはパターン 1（ローカルのプリミティブ型）のみ

#### 1.4.5 よくある誤解の訂正

| 誤解 | 正しい理解 |
|------|-----------|
| 「String はスタックに置かれる」 | String はオブジェクトであり、本体は**ヒープ**に置かれる。スタックに置かれるのは String オブジェクトを指す**参照（Reference）** だけである |
| 「プリミティブ型は常にスタック」 | ローカル変数のプリミティブ型はスタックだが、インスタンスフィールドのプリミティブ型は**ヒープ**（親オブジェクト内）に置かれる。「プリミティブ型 = スタック」は半分だけ正しい |
| 「static 変数はメタスペースに置かれる」 | メタスペースに置かれるのはクラスの**定義（メタデータ）** である。static フィールドの**値**はヒープ上の `java.lang.Class` オブジェクト内に格納される。Java 7 以前の PermGen 時代の名残で混同されやすい |
| 「メソッドはメタスペースにある」 | 半分正しい。メソッドの**定義（バイトコード）** はメタスペースにあるが、メソッドの**実行**はスタック上のフレームで行われ、JIT 後の**機械語**はコードキャッシュに格納される。「メソッドの所在」は 3 箇所にまたがる |
| 「参照型の変数はヒープに置かれる」 | **ローカル変数** の参照型は、参照自体は**スタック**に置かれる。ヒープに置かれるのは参照**先のオブジェクト本体**である。ただし、インスタンスフィールドや static フィールドの参照型は、参照自体もヒープ内（親オブジェクトまたは Class オブジェクトの中）にある |
| 「配列はスタックに置かれる」 | `int[] arr = new int[10];` の場合、`arr`（参照）はスタックだが、10 要素分のデータ本体は**ヒープ**に置かれる。配列もオブジェクトである |

> **Escape Analysis（エスケープ解析）に関する補足:** HotSpot の C2 JIT コンパイラは「エスケープ解析」という最適化を行う。メソッド内で生成されたオブジェクトが「そのメソッドの外に逃げない（他のメソッドに渡されない、フィールドに代入されない）」と判断された場合、ヒープではなく**スタック上にオブジェクトを割り当てる**（スカラー置換）ことがある。これは JIT コンパイラによる実行時最適化であり、Java 言語仕様上は「すべてのオブジェクトはヒープに配置される」という原則に変わりはない。あくまで HotSpot の内部最適化として、意味的に等価な範囲でスタック配置に置き換えることがある、と理解しておけばよい。
>
> **関連ソース (HotSpot C++):** `src/hotspot/share/opto/escape.cpp` -- エスケープ解析の実装

## 2. なぜスタックは GC 不要なのに、ヒープには GC が必要なのか

### 2.1 スタックの解放 -- スタックポインタを動かすだけ

**対象層: HotSpot (C++) 層**

スタックが GC を必要としない理由は、その構造的な性質に由来する。ここでは核心となる **スタックポインタ (SP)** の役割を、直感的に理解できるよう掘り下げる。

#### SP とは何か -- 「ここまでが使用中」の境界線

HotSpot の内部では、各スレッドが **スタックポインタ (SP: Stack Pointer)** という CPU レジスタを持っている。SP は「スタック上のどこまでが現在使用中か」を示す **境界線** である。

ホワイトボードで考えると分かりやすい。巨大なホワイトボードがあり、下から順に書き込みをしていくとする。そのとき、**「ここまで書きました」という横線** が SP にあたる。横線の下側は使用中の有効な領域であり、横線の上側は「まだ何も書かれていない（あるいは、以前の書き込みが残っているかもしれないが、無効な）空き領域」である。

```
スタックメモリのイメージ（下が底、上が天井）

  アドレス高
  ┌─────────────────────────────┐
  │                             │  ← 未使用領域
  │        (空き / 無効)         │     SP の外側は「存在しないもの」扱い
  │                             │
  ├─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┤ ◀◀◀ SP（境界線）: 「ここまでが使用中」
  │   methodA() のフレーム       │  ← 使用中（有効）
  ├─────────────────────────────┤
  │   main() のフレーム          │  ← 使用中（有効）
  └─────────────────────────────┘
  アドレス低
```

**重要なのは、SP はただの「線」であるということ。** SP の値（アドレス）を変更すれば、境界線が移動し、使用中の範囲が変わる。メモリの中身を消す必要はない。

#### 「解放」の正体 -- OS にメモリを返すことではない

スタックにおける「解放」という言葉は、多くの初心者に誤解を与える。ここでの「解放」とは、**OS にメモリを返却すること（`free` や `munmap`）ではない。** SP という境界線を後退させて、「この領域はもう使っていません」と宣言するだけのことである。

ホワイトボードの比喩に戻ろう。あなたが書いた内容を消しゴムで消す必要はない。単に **「ここから先は無効です」という線を引き直すだけ** でいい。以前書いた文字はホワイトボード上にまだ残っているが、「無効」と宣言された以上、誰もそれを読みに来ない。そして次に誰かがホワイトボードに書き込むとき、その領域は上から新しい内容で塗り潰される。

#### ステップバイステップ -- SP の動きを追う

以下の図で、メソッド呼び出しと終了に伴う SP の動きを追跡する。

```
【Step 1】main() が実行中
    SP はここ ─────────────────▶ ├ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┤
                                 │  main() のフレーム     │
                                 └────────────────────────┘

【Step 2】main() が methodA() を呼び出す → SP が上昇
    SP はここ ─────────────────▶ ├ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┤
                                 │  methodA() のフレーム  │ ← 新たに確保
                                 ├────────────────────────┤
                                 │  main() のフレーム     │
                                 └────────────────────────┘

【Step 3】methodA() が methodB() を呼び出す → SP がさらに上昇
    SP はここ ─────────────────▶ ├ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┤
                                 │  methodB() のフレーム  │ ← 新たに確保
                                 ├────────────────────────┤
                                 │  methodA() のフレーム  │
                                 ├────────────────────────┤
                                 │  main() のフレーム     │
                                 └────────────────────────┘

【Step 4】methodB() が終了 → SP が下降（「解放」）
    SP はここ ─────────────────▶ ├ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┤
                                 │  methodA() のフレーム  │
                                 ├────────────────────────┤
                                 │  main() のフレーム     │
                                 └────────────────────────┘
                                 ※ methodB() のデータはメモリ上にまだ物理的に残っている。
                                   しかし SP の外側なので「存在しない」扱い。

【Step 5】methodA() が methodC() を呼び出す → SP が再び上昇
    SP はここ ─────────────────▶ ├ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┤
                                 │  methodC() のフレーム  │ ← methodB() の残骸を上書き！
                                 ├────────────────────────┤
                                 │  methodA() のフレーム  │
                                 ├────────────────────────┤
                                 │  main() のフレーム     │
                                 └────────────────────────┘
```

Step 4 → Step 5 が核心である。methodB() のフレームが使っていた**まったく同じメモリ領域**を、methodC() のフレームが上書きしている。消去のための操作は一切行われない。

#### なぜこれが圧倒的に高速なのか

スタックの確保・解放が高速な理由は極めてシンプルである。**SP というレジスタの値を 1 つ変更するだけ** だからだ。

CPU レジスタの書き換えは、1 クロックサイクル程度で完了する。ヒープの `malloc` / `free` のように空きブロックリストを走査したり、GC のように生存オブジェクトの到達可能性を解析したりする必要は一切ない。メソッド呼び出し 1 回につき、SP を「上に動かす」（確保）か「下に戻す」（解放）かの、事実上 **O(1)** の操作で完結する。

これこそが、スタックに GC が不要な理由であり、スタックベースのメモリ管理が数十年にわたって採用され続けている理由でもある。

#### HotSpot の実装

HotSpot (C++) 層の実装では、スレッドの生成時に連続したメモリ領域をスタック用に一括で確保する（`os::create_thread` 内で `pthread_attr_setstacksize` 等を呼び出す）。以降、そのスレッドが生きている間は新たな OS レベルのメモリ確保・解放は発生せず、SP の移動（ポインタ演算）だけでスタックフレームの管理が行われる。

> - `src/hotspot/os/linux/os_linux.cpp` -- Linux 上でのスタック割り当て

### 2.2 ヒープの解放 -- なぜ「掃除」が必要なのか

**対象層: HotSpot (C++) 層**

ヒープは GC なしでは管理できない。その理由は 3 つある。

#### 理由 1: 生成と消滅が不規則

スタックとは異なり、ヒープ上のオブジェクトは**任意の順序で**生成・不要化する。あるオブジェクトが先に作られても、後から作られたオブジェクトより先に不要になることがある。LIFO のような単純なルールでは解放の順序を決められない。

```
時刻 1: objA を確保  [objA][    ][    ]
時刻 2: objB を確保  [objA][objB][    ]
時刻 3: objC を確保  [objA][objB][objC]
時刻 4: objB が不要に [objA][ 穴 ][objC]  ← 中間に「穴」ができる
```

#### 理由 2: 参照グラフの複雑さ

ヒープ上のオブジェクトは相互に参照し合い、複雑なグラフ構造を形成する。あるオブジェクトが本当に不要かどうかは、ルート（GC Roots）からの到達可能性を調べなければ判断できない。

```
GC Roots（スタック上の変数、static フィールドなど）
    │
    ├──→ UserService ──→ UserRepository ──→ DataSource
    │
    └──→ OrderList ──→ Order ──→ User
                                   ↑
                        (循環参照の可能性もある)
```

この到達可能性の判定を行うのが GC であり、HotSpot は様々な GC アルゴリズム（G1GC、ZGC、Shenandoah 等）でこれを実現している。

#### 理由 3: 断片化（Fragmentation）

オブジェクトの確保と解放が不規則に繰り返されると、ヒープ上に「穴」が点在する**断片化**が発生する。十分な合計空き容量があっても、連続した空き領域がなければ新しいオブジェクトを確保できない。

GC（特にコンパクション機能を持つもの）は、生存オブジェクトを一方向に詰め直すことで断片化を解消する。

```
GC 前:  [obj][ 穴 ][obj][ 穴 ][ 穴 ][obj]
          ↓ コンパクション ↓
GC 後:  [obj][obj][obj][        空き        ]
```

> - `src/hotspot/share/gc/g1/` -- G1GC の実装
> - `src/hotspot/share/gc/z/` -- ZGC の実装
> - `src/hotspot/share/gc/shared/collectedHeap.cpp` -- ヒープの基底クラス

### 2.3 GC アルゴリズム入門 -- 掃除のやり方は一つじゃない

**対象層: HotSpot (C++) 層**

ここまでで「ヒープにはGCが必要」ということが分かった。では、その GC とは具体的にどんな仕組みなのか。そして、なぜ複数の GC アルゴリズムが存在するのか。初心者向けに整理する。

#### GC とは何か

**GC（Garbage Collection、ガベージコレクション）** とは、ヒープ上の「もう使われていないオブジェクト」を自動的に見つけて回収する仕組みである。C言語のように開発者が `free()` を呼ぶ必要はなく、JVM が「このオブジェクトはもう誰からも参照されていない」と判断した時点で、自動的にメモリを解放する。

```
GC の基本動作:

  ① GC Roots（スタック上の変数、static フィールドなど）から辿れるオブジェクトをマーク
  ② マークされなかったオブジェクト ＝ 到達不能 ＝ ゴミ → 回収

  GC Roots ──→ [A] ──→ [B] ──→ [C]    ← A, B, C は生存

               [D]       [E]             ← D, E は到達不能 → ゴミとして回収
```

#### なぜ GC アルゴリズムが複数あるのか

GC の設計には根本的なトレードオフが存在する。

- **スループット（処理速度）** を優先すると、GC がまとめて一括処理するためアプリケーション全体の停止時間が長くなる
- **レイテンシ（停止時間の短さ）** を優先すると、GC が小刻みに動くため、スループットが若干犠牲になる

すべてを完璧に満たす万能な GC は存在しない。用途に応じた最適解が異なるため、HotSpot には複数の GC アルゴリズムが搭載されている。

```
トレードオフの関係:

  スループット重視 ◀──────────────────────▶ レイテンシ重視
  (一括掃除、長い停止)                       (こまめ掃除、短い停止)

  Serial GC   Parallel GC     G1GC        Shenandoah    ZGC
  ◀────────────────────────────────────────────────────────▶
```

#### 主要な GC アルゴリズムの概要

##### Serial GC -- 一人の掃除人が全部掃除する

一番シンプルな GC。**1つのスレッドだけ**を使ってヒープ全体を掃除する。掃除中はアプリケーションの全スレッドが停止する（Stop-The-World）。

比喩で言えば、**オフィスの全員が仕事を止めて、掃除担当が一人で全フロアを掃除する**イメージ。シンプルで確実だが、全員が待たされる。

- **向いている用途:** ヒープが小さい（数十〜数百MB）アプリケーション、組み込み環境
- **JVMフラグ:** `-XX:+UseSerialGC`

##### Parallel GC -- 複数人で一斉に掃除する

Serial GC の並列版。**複数の GC スレッドを使って**同時に掃除する。掃除時間は短くなるが、掃除中はやはり全員が待たされる。

比喩で言えば、**掃除チームが一斉にフロアに入り、手分けして掃除する**イメージ。掃除は速いが、掃除中はやっぱり全員待ち。

- **向いている用途:** バッチ処理など、スループット（単位時間あたりの処理量）が最重要なアプリケーション
- **JVMフラグ:** `-XX:+UseParallelGC`

##### G1GC (Garbage-First) -- ゴミが多いところから優先的に掃除する

**Java 9 以降のデフォルト GC** であり、現在最も広く使われている GC アルゴリズム。ヒープ全体を小さな **リージョン（Region）** に分割し、ゴミが多いリージョンから優先的に回収する。名前の「Garbage-First（ゴミが一番多いところから）」はこの戦略に由来する。

比喩で言えば、**オフィスを小さな区画に分け、一番散らかっている区画から優先的に掃除する**イメージ。全体を一度に掃除する必要がなく、停止時間を予測可能な範囲に抑えられる。

```
G1GC のヒープ構造（リージョンベース）:

┌────┬────┬────┬────┬────┬────┬────┬────┐
│ E  │ E  │ S  │ O  │ O  │ E  │ H  │ 空 │
├────┼────┼────┼────┼────┼────┼────┼────┤
│ O  │ 空 │ E  │ O  │ 空 │ O  │ O  │ E  │
└────┴────┴────┴────┴────┴────┴────┴────┘

E = Eden（新規オブジェクト）  S = Survivor  O = Old  H = Humongous（巨大オブジェクト）

→ ゴミが多いリージョンを優先的に回収するので「Garbage-First」
```

- **向いている用途:** 一般的な Web アプリケーション（Spring Boot の標準選択）
- **特徴:** 停止時間の目標値を `-XX:MaxGCPauseMillis` で指定可能（デフォルト 200ms）
- **JVMフラグ:** `-XX:+UseG1GC`（Java 9+ ではデフォルトなので通常は指定不要）

> **関連ソース (HotSpot C++):** `src/hotspot/share/gc/g1/` -- G1GC の実装

##### ZGC -- 超低レイテンシの次世代 GC

**ヒープサイズが数テラバイト級でも、停止時間を 1ms 以下に抑える** ことを目標に設計された GC。Java 15 で正式化（Production Ready）され、Java 21 で世代別（Generational）に進化した。

比喩で言えば、**掃除ロボットが仕事中のオフィスを、誰の邪魔もせずに自動で掃除し続ける**イメージ。ほぼすべての処理をアプリケーションと並行して実行するため、停止時間が極めて短い。

- **向いている用途:** 低レイテンシが求められるリアルタイム系、大規模ヒープのアプリケーション
- **特徴:** ヒープサイズに関係なく停止時間が一定（スケーラブル）
- **JVMフラグ:** `-XX:+UseZGC`（Java 21+ では `-XX:+UseZGC -XX:+ZGenerational` で世代別モード）

> **関連ソース (HotSpot C++):** `src/hotspot/share/gc/z/` -- ZGC の実装

##### Shenandoah -- もう一つの低レイテンシ GC

Red Hat が開発した低レイテンシ GC で、ZGC と同じ「停止時間をヒープサイズに依存させない」という目標を持つ。最大の特徴は、**コンパクション（オブジェクトの移動による断片化解消）をアプリケーションと並行して実行できる**点。

- **向いている用途:** ZGC と同様、低レイテンシが求められるアプリケーション
- **注意:** Oracle 配布の JDK には含まれていないが、Red Hat 系ディストリビューション (Azul Zulu を含む) では利用可能
- **JVMフラグ:** `-XX:+UseShenandoahGC`

> **関連ソース (HotSpot C++):** `src/hotspot/share/gc/shenandoah/` -- Shenandoah の実装

#### GC アルゴリズム比較表

| GC | 停止時間 | スループット | 対応ヒープサイズ | 主な用途 | JVM フラグ |
|----|---------|------------|----------------|---------|-----------|
| **Serial** | 長い | 低い | 小（〜数百MB） | 小規模アプリ、組み込み | `-XX:+UseSerialGC` |
| **Parallel** | 中程度 | **最も高い** | 中〜大 | バッチ処理、スループット重視 | `-XX:+UseParallelGC` |
| **G1GC** | やや短い | 高い | 中〜大（〜数十GB） | **Web アプリの標準**（デフォルト） | `-XX:+UseG1GC` |
| **ZGC** | **極めて短い（<1ms）** | 高い | 小〜**超大（数TB）** | 低レイテンシ、大規模ヒープ | `-XX:+UseZGC` |
| **Shenandoah** | **極めて短い** | 高い | 中〜大 | 低レイテンシ | `-XX:+UseShenandoahGC` |

#### Spring Boot 開発者向けの実用的アドバイス

**大半のケースでは、デフォルトの G1GC で十分である。** Java 9 以降、何も指定しなければ G1GC が使われる。Spring Boot アプリケーションにおいて、GC の選択で悩む必要があるのは以下の場合だけである。

- **レスポンスタイムの P99 が厳しい（数ms以下）マイクロサービス** → ZGC を検討
- **超大規模ヒープ（数十GB以上）を使うアプリケーション** → ZGC を検討
- **バッチ処理でスループットを最大化したい** → Parallel GC を検討

```bash
# 一般的な Spring Boot アプリ（デフォルトの G1GC で OK）
java -Xms1g -Xmx2g -jar myapp.jar

# 低レイテンシが必要な場合（ZGC を明示指定）
java -Xms1g -Xmx2g -XX:+UseZGC -jar myapp.jar

# バッチ処理でスループット優先（Parallel GC を指定）
java -Xms4g -Xmx4g -XX:+UseParallelGC -jar myapp.jar
```

> **より詳細な GC の仕組み**（世代別仮説、ヒープ内部構造、Minor/Major/Full GC の違い、OOM の原因、G1GC の詳細動作、GC ログの読み方、チューニング実務）については [gc-deep-dive.md](gc-deep-dive.md) を参照。

### 2.4 対比まとめ

| 観点 | スタック | ヒープ |
|------|---------|--------|
| **構造** | 線形 (LIFO) | グラフ（参照の網） |
| **解放の順序** | 常に末端から（予測可能） | 不規則（予測不能） |
| **解放コスト** | ポインタ移動のみ（O(1)） | 到達可能性解析 + コンパクション |
| **断片化** | 構造上発生しない | 必然的に発生する |
| **GC の必要性** | 不要 | 必須 |

## 3. Spring Boot 開発者が意識すべきメモリの実務

### 3.1 Thread-per-request モデルとスタック

**対象層: Java 層**

Spring Boot のデフォルト（Tomcat 組み込みサーバー）では、HTTP リクエストごとに 1 つのスレッドが割り当てられる **Thread-per-request** モデルが採用されている。

```
[リクエスト A] ──→ [スレッド 1] ──→ Controller → Service → Repository
[リクエスト B] ──→ [スレッド 2] ──→ Controller → Service → Repository
[リクエスト C] ──→ [スレッド 3] ──→ Controller → Service → Repository
```

各スレッドは自分専用のスタックを持つため、**同時接続数が増えるとスタックメモリの総消費量も比例して増加する。** Tomcat のデフォルトスレッドプール上限は 200 であり、各スレッドのスタックサイズがデフォルト 1MB の場合、スタックだけで最大約 200MB を消費する計算になる。

```
スタック総消費量 = スレッド数 x スタックサイズ(-Xss)
               = 200 x 1MB
               = 200MB
```

### 3.2 DI コンテナとヒープ -- シングルトン Bean の長寿命

**対象層: Java 層**

Spring の DI コンテナが管理する Bean は、デフォルトで**シングルトンスコープ**である。つまり、アプリケーションの起動時にヒープ上に生成され、**アプリケーションが終了するまでヒープに存在し続ける。**

```
ヒープ上の Spring Bean（シングルトン）
┌─────────────────────────────────────────────┐
│  UserController@0x1A   (起動時に生成、終了まで生存)  │
│  UserService@0x2B      (起動時に生成、終了まで生存)  │
│  UserRepository@0x3C   (起動時に生成、終了まで生存)  │
│  DataSource@0x4D       (起動時に生成、終了まで生存)  │
└─────────────────────────────────────────────┘
```

これらのシングルトン Bean 自体はヒープを圧迫する主因にはなりにくい。問題になるのは、**これらの Bean が内部に保持するコレクションやキャッシュが際限なく成長する場合**である。

### 3.3 メモリリークが起きやすいパターン

Spring Boot アプリケーションで特に注意すべきメモリリークのパターンを示す。

#### パターン 1: static な Collection への無制限な追加

```java
// DANGER: static field lives as long as the class is loaded
@Service
class EventLogger {
    static final List<String> logs = new ArrayList<>();

    void log(String message) {
        logs.add(message); // grows indefinitely, never cleared
    }
}
```

`static` フィールドはクラスがロードされている限りヒープ上に存在し続ける。ここに要素を追加し続けると、GC が回収できないままヒープが枯渇する。

#### パターン 2: リスナー・コールバックの登録解除忘れ

```java
// DANGER: listener holds reference to EventProcessor, preventing GC
@Component
class EventProcessor {
    @Autowired EventBus bus;

    @PostConstruct
    void init() {
        bus.register(this); // registers this as listener
    }

    // missing @PreDestroy to call bus.unregister(this)
    // if this bean's scope were prototype, it would leak
}
```

イベントリスナーの登録を解除しないと、対象オブジェクトへの参照が残り続け、GC の回収対象にならない。

#### パターン 3: ThreadLocal の未クリア

```java
// DANGER: in a thread pool, ThreadLocal values survive across requests
class RequestContext {
    static final ThreadLocal<Map<String, Object>> context = new ThreadLocal<>();

    static void set(Map<String, Object> data) {
        context.set(data);
    }

    // must call context.remove() after each request
}
```

スレッドプール環境では、スレッドが再利用されるため `ThreadLocal` の値が次のリクエストに持ち越される。明示的に `remove()` しないと、意図しないデータ保持とメモリリークの原因になる。

### 3.4 JVM 起動オプション -- `-Xss` と `-Xmx`

| オプション | 対象 | 意味 | 典型的な設定 |
|-----------|------|------|-------------|
| `-Xss` | スタック | 1 スレッドあたりのスタックサイズ | `-Xss512k`（深い再帰がなければ十分） |
| `-Xmx` | ヒープ | ヒープの最大サイズ | `-Xmx2g`（アプリ規模に応じて調整） |
| `-Xms` | ヒープ | ヒープの初期サイズ | `-Xms512m`（起動時の GC を減らすため `-Xmx` と同値にすることも多い） |

```bash
# Spring Boot アプリケーションの起動例
java -Xss512k -Xms1g -Xmx2g -jar myapp.jar
```

**`-Xss` を小さくしすぎると:** 深いメソッド呼び出しチェーンで `StackOverflowError` が発生する。Spring のプロキシ機構（AOP）はメソッド呼び出しの段数を増やすため、極端に小さい値は危険である。

**`-Xmx` を小さくしすぎると:** ヒープが不足し `OutOfMemoryError: Java heap space` が発生する。逆に大きくしすぎると、GC の停止時間（Pause Time）が長くなる可能性がある。

### 4.1 MemoryExplorer -- スタックとヒープの ASCII アート可視化

`System.identityHashCode()` を疑似アドレスとして活用し、スタック上のローカル変数とヒープ上のオブジェクトの対応関係を **矢印付き ASCII アート** で段階的に可視化するツール。6 つのステップで構成されている:

| Step | 内容 | 学べること |
|------|------|-----------|
| 1 | プリミティブ変数（int, double, boolean） | 値がスタックに直接格納され、ヒープは空であること |
| 2 | オブジェクト生成（record, String） | スタックに参照、ヒープに本体が配置されること |
| 3 | 配列生成 | 配列もオブジェクトであり、ヒープに配置されること |
| 4 | メソッド呼び出し（StackWalker 使用） | スタックフレームが積み上がる様子 |
| 5 | 同一オブジェクトへの複数参照 | 同じアドレスを指す「参照の共有」 |
| 6 | 参照を null にする | GC の回収対象になる仕組み |

### 4.2 StackFrameVisualizer -- スタックフレームの積み上がり

意図的に深い呼び出しチェーン（`methodA → methodB → methodC`）を作成し、各フレームでローカル変数が独立していることを確認する。StackWalker API で実際のスタックを表示。

### 4.3 StackOverflowDemo -- StackOverflowError の発生実験

無限再帰でスタックフレームを際限なく積み上げ、`-Xss` で設定されたスタック上限に達した瞬間の `StackOverflowError` を捕捉する。`-Xss` の値を変えると到達深度が変化する。

### 4.4 HeapGcObserver -- ヒープ枯渇と GC の観察

2MB のバイト配列をリストに保持し続け、ヒープを一方的に消費していく。`-verbose:gc` と併用すると、GC が何度も走るのに回収量がほぼ 0 である様子が確認でき、「参照を持ち続ける = GC が無力化する」ことを実感できる。最終的に `OutOfMemoryError` が発生する。

### 4.5 MemoryDiagnostics -- JVM メモリ診断ユーティリティ

`ManagementFactory` API を使って JVM の内部状態を一覧表示。Spring Boot の Actuator が内部で使っている仕組みと同じ MXBean を直接操作する。ヒープ使用率のバーグラフ、メモリプールの役割解説、GC 統計のヘルスチェック、全スレッド一覧と役割を出力する。

### 4.6 MetaspaceExplorer -- Metaspace の使用状況確認

セクション 5.3「メソッドエリア / Metaspace」で解説した Metaspace 領域の実際の使用状況を `MemoryPoolMXBean` と `ClassLoadingMXBean` で確認するツール。NON_HEAP タイプのメモリプールから Metaspace 関連（"Metaspace" / "Compressed Class Space"）を抽出して使用量・確保量・最大値を表示し、現在ロード済みのクラス数・累計ロード/アンロード数も出力する。Java 8 での PermGen → Metaspace への刷新の経緯も Javadoc に記載されている。

### 4.7 CodeCacheExplorer -- Code Cache の使用状況確認

セクション 5.4「コードキャッシュ」で解説した Code Cache 領域の実際の使用状況を確認するツール。Java 9 以降の HotSpot が Code Cache を 3 セグメント（`non-nmethods` / `profiled nmethods` / `non-profiled nmethods`）に分割して管理していることを `MemoryPoolMXBean` で可視化する。さらに `CompilationMXBean` で JIT コンパイラ名と累計コンパイル時間を表示し、ホットスポット検出を誘発する計算処理を実行して Code Cache の使用量変化を確認できる。

## 5. ヒープとスタック「以外」の JVM メモリ領域

ここまでヒープとスタックを詳しく見てきたが、JVM のメモリ空間にはそれ以外にも重要な領域がいくつも存在する。JVM 仕様（Chapter 2.5 Run-Time Data Areas）に基づき、すべてのメモリ領域を初心者向けに解説する。

### 5.1 PC レジスタ (Program Counter Register)

**対象層: HotSpot (C++) 層**

各スレッドが **1 つずつ専有する** 小さなポインタ。「今バイトコードのどの命令を実行しているか」を記録している。

比喩で言えば、**本を読むときに指で押さえている「今読んでいる行」**のようなもの。スレッドが次にどの命令を実行すべきかを追跡するための仕組みである。

- **サイズ:** 非常に小さい（アドレス 1 つ分 = 通常 8 バイト）
- **ネイティブメソッド実行中:** JVM 仕様上、PC レジスタの値は「未定義 (undefined)」となる。ネイティブコードは JVM のバイトコードではないため、バイトコード位置を追跡する意味がない
- **開発者が意識する必要:** ほとんどない。JVM が内部的に使うだけの領域

> **JVM 仕様:** 「If the method currently being executed by the thread is not a `native` method, the `pc` register contains the address of the Java Virtual Machine instruction currently being executed.」

### 5.2 ネイティブメソッドスタック (Native Method Stack)

**対象層: JNI / ネイティブ層 / HotSpot (C++) 層**

JNI（Java Native Interface）経由で **C/C++ のネイティブコード** を呼び出すときに使われるスタック領域。Java のスタック（JVM スタック）とは別に、ネイティブコード用のスタックフレームを管理する。

比喩で言えば、**日本語の書類棚（JVM スタック）とは別に設けた英語の書類棚**のようなもの。Java の世界とネイティブの世界でそれぞれ独立した「作業台」が必要になる。

```
Java メソッドの呼び出し:       ネイティブメソッドの呼び出し:

JVM スタック                  ネイティブメソッドスタック
┌─────────────┐              ┌──────────────────────┐
│ methodB()   │              │ native_function()    │
├─────────────┤    JNI 経由   ├──────────────────────┤
│ methodA()   │ ──────────▶  │ JNI_bridge()         │
└─────────────┘              └──────────────────────┘
```

- **HotSpot の実装では:** JVM スタックとネイティブメソッドスタックは **統合されている**（同一のスタック領域を共用する）。これは JVM 仕様が許容している実装上の選択であり、HotSpot の特徴の一つ
- **開発者が意識する場面:** JNI を直接使うことは稀だが、JDBC ドライバやネイティブライブラリ（OpenCV、TensorFlow 等）を使う場合、内部的にこのスタックが使われている

### 5.3 メソッドエリア / Metaspace

**対象層: HotSpot (C++) 層**

JVM 仕様上は **「Method Area（メソッドエリア）」** と呼ばれる領域で、HotSpot の実装では **「Metaspace（メタスペース）」** という名前で実現されている。

#### 何が格納されるか

```
Metaspace に格納されるもの:

┌────────────────────────────────────────────┐
│  クラス名        : java.util.ArrayList      │
│  親クラス        : java.util.AbstractList   │
│  メソッド定義     : add(), get(), size() ... │
│  フィールド定義   : elementData, size        │
│  定数プール       : 文字列リテラル、数値定数  │
│  アノテーション情報: @Override, @Service ...  │
└────────────────────────────────────────────┘
```

つまり、**「クラスそのものの設計図」** が格納される場所である。オブジェクトのインスタンス（実体）はヒープに置かれるが、「そのクラスにはどんなメソッドがあるか」「どんなフィールドがあるか」といったメタ情報はここに置かれる。

#### PermGen から Metaspace への進化

Java 8 より前は、この領域は **PermGen（Permanent Generation、永久世代）** と呼ばれ、**ヒープの一部** として管理されていた。しかし、PermGen には固定サイズの制限があり、クラスを大量にロードするアプリケーション（Spring、Hibernate 等のフレームワーク）で `OutOfMemoryError: PermGen space` が頻発するという問題があった。

Java 8 以降、この領域は **ヒープの外のネイティブメモリ** に移動し、名前も Metaspace に変更された。ネイティブメモリを使うことで、必要に応じて動的に拡張できるようになった。

```
Java 7 以前:
┌──── ヒープ ──────────────────┐
│  Young Gen  │  Old Gen  │ PermGen │  ← PermGen はヒープの一部（固定サイズ）
└─────────────────────────────────┘

Java 8 以降:
┌──── ヒープ ────────────┐  ┌─ Metaspace ─┐
│  Young Gen  │  Old Gen  │  │ ネイティブ    │  ← ヒープ外に移動（動的拡張可能）
└─────────────────────────┘  └──────────────┘
```

- **サイズ制御:** `-XX:MaxMetaspaceSize`（デフォルトは無制限 = ネイティブメモリの限界まで）
- **Spring Boot との関係:** Spring や Hibernate は大量のクラスを動的に生成する（プロキシ、AOP など）。そのため、Metaspace の消費量が大きくなる傾向がある。`-XX:MaxMetaspaceSize=256m` のように上限を設定しておくと、メモリリーク時に早期に気付ける

> **関連ソース (HotSpot C++):**
> - `src/hotspot/share/memory/metaspace/` -- Metaspace の実装
> - `src/hotspot/share/oops/instanceKlass.cpp` -- クラスメタデータの管理

### 5.4 コードキャッシュ (Code Cache)

**対象層: HotSpot (C++) 層**

**JIT コンパイラが生成した機械語コード** を格納するための専用メモリ領域。

冒頭の「HotSpot とは何か」で説明した通り、HotSpot VM はプログラム実行中に「ホットスポット」を検出し、JIT コンパイラ（C1/C2）でバイトコードを機械語に変換する。この変換結果の機械語が保存される場所がコードキャッシュである。

```
バイトコード → JIT コンパイル → 機械語 → コードキャッシュに格納

┌─── コードキャッシュ ──────────────────────────────┐
│  UserService.findById() の機械語コード            │
│  OrderRepository.save() の機械語コード            │
│  String.hashCode() の機械語コード                 │
│  ... 頻繁に呼ばれるメソッドほど最適化される ...     │
└─────────────────────────────────────────────────┘
```

- **サイズ制御:** `-XX:ReservedCodeCacheSize`（デフォルトは 240MB 程度）
- **コードキャッシュが満杯になると:** JIT コンパイルが停止し、以降のメソッドはインタプリタで実行されるため、パフォーマンスが急激に低下する。ログに `CodeCache is full` という警告が出る
- **開発者が意識する場面:** 大規模アプリケーション（メソッド数が非常に多い場合）でコードキャッシュ不足になることがある

> **関連ソース (HotSpot C++):**
> - `src/hotspot/share/code/codeCache.cpp` -- コードキャッシュの管理

### 5.5 ダイレクトメモリ (Direct Memory / Off-heap)

**対象層: Java 層 / JNI 層**

`ByteBuffer.allocateDirect()` によって確保される **ヒープ外のネイティブメモリ** 領域。通常のヒープ上のオブジェクトとは異なり、GC の管理対象外である。

```java
// Direct memory allocation -- lives outside the Java heap
var buffer = ByteBuffer.allocateDirect(1024 * 1024); // 1MB of off-heap memory
```

#### なぜヒープ外にメモリを確保するのか

ファイル I/O やネットワーク I/O を行うとき、ヒープ上の byte 配列をそのまま OS に渡すことはできない（GC がオブジェクトを移動する可能性があるため）。通常はヒープからネイティブメモリにコピーしてから OS に渡すが、ダイレクトメモリを使えば **コピーなしで直接 OS に渡せる** ため、I/O 性能が向上する。

```
通常の ByteBuffer:       ヒープ → コピー → ネイティブメモリ → OS
ダイレクト ByteBuffer:   ネイティブメモリ → OS（コピー不要！）
```

- **サイズ制御:** `-XX:MaxDirectMemorySize`（デフォルトは `-Xmx` と同値）
- **主な利用者:** NIO（New I/O）、Netty、Apache Kafka など I/O 集約型のライブラリ
- **注意点:** GC の管理外なので、`ByteBuffer` オブジェクト自体が GC で回収されるまでネイティブメモリは解放されない。大量に確保すると `OutOfMemoryError: Direct buffer memory` が発生する

### 5.6 文字列プール (String Pool)

**対象層: Java 層 / HotSpot (C++) 層**

**文字列リテラル** が格納される、ヒープ内の特殊な領域。同じ内容の文字列リテラルが重複して生成されるのを防ぎ、メモリを節約する仕組みである。

```java
// Both s1 and s2 point to the SAME object in the String Pool
var s1 = "hello";
var s2 = "hello";
// s1 == s2 → true (same reference, both from pool)

// s3 is a NEW object on the heap (not from pool)
var s3 = new String("hello");
// s1 == s3 → false (different references)

// s4 is explicitly interned into the pool
var s4 = s3.intern();
// s1 == s4 → true (s4 now points to the pooled instance)
```

- **Java 7 以降:** String Pool はヒープ内に配置されている（Java 6 以前は PermGen に配置されていた）
- **`String.intern()`:** 明示的に文字列をプールに追加するメソッド。ただし、乱用するとプールが肥大化するため注意
- **サイズ制御:** `-XX:StringTableSize`（ハッシュテーブルのバケット数、デフォルトは 65536）

### 5.7 メモリ領域の全体対比表

| 領域 | 所有者 | 格納内容 | サイズ制御フラグ | 開発者の意識度 |
|------|--------|---------|-----------------|--------------|
| **ヒープ (Heap)** | 全スレッド共有 | オブジェクトインスタンス、配列 | `-Xmx`, `-Xms` | **高** -- 最も重要 |
| **JVM スタック (Stack)** | スレッドごと | ローカル変数、参照、戻りアドレス | `-Xss` | **高** -- スレッド数に注意 |
| **PC レジスタ** | スレッドごと | 現在のバイトコード位置 | なし（固定、極小） | **低** -- 意識不要 |
| **ネイティブメソッドスタック** | スレッドごと | JNI 呼び出しのフレーム | `-Xss`（HotSpot では統合） | **低** -- JNI 使用時のみ |
| **Metaspace** | 全スレッド共有 | クラスメタデータ、定数プール | `-XX:MaxMetaspaceSize` | **中** -- Spring では注意 |
| **コードキャッシュ** | 全スレッド共有 | JIT コンパイル済み機械語 | `-XX:ReservedCodeCacheSize` | **中** -- 大規模アプリで注意 |
| **ダイレクトメモリ** | 確保したスレッド | NIO バッファ等 | `-XX:MaxDirectMemorySize` | **中** -- NIO/Netty 使用時 |
| **文字列プール** | 全スレッド共有（ヒープ内） | 文字列リテラル | `-XX:StringTableSize` | **低** -- 通常は意識不要 |

## 7. まとめ

| 問い | 答え |
|------|------|
| スタックとは何か？ | スレッド専有の LIFO メモリ。メソッド呼び出しのたびにフレームが積まれ、終了で自動解放。 |
| ヒープとは何か？ | 全スレッド共有のメモリ。`new` で確保されたオブジェクト本体が置かれる。 |
| なぜスタックは GC 不要？ | LIFO 構造により、スタックポインタを戻すだけで解放が完了するため。 |
| なぜヒープは GC が必要？ | 生成・消滅が不規則で、参照グラフが複雑。到達可能性を解析しなければ不要オブジェクトを判定できない。 |
| Spring 開発者は何に注意？ | スレッド数 x `-Xss` のスタック総量、シングルトン Bean の長寿命性、static Collection やリスナーのリーク、`-Xmx` の適切な設定。 |

### Zulu との差分
Azul Zulu は OpenJDK のディストリビューションであり、ヒープ・スタックの基本的なメモリモデルは OpenJDK mainline と同一である。ただし、Azul の商用製品 (Zing/Prime) では **C4 GC** という独自の GC を搭載しており、ヒープのコンパクションを並行処理で行うため、GC の停止時間特性が大きく異なる。Zulu (OSS 版) を使う分には、HotSpot の G1GC/ZGC と同等の挙動となる。
