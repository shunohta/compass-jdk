## 概要

本ドキュメントは、ZGC (Z Garbage Collector) の内部構造を詳細に解説するものである。[gc-deep-dive.md](gc-deep-dive.md) で扱った GC の基礎理論（世代別仮説、Mark-Sweep-Compact、GC Roots 等）を前提知識とし、ZGC に特化した設計思想・実装メカニズム・チューニング手法を掘り下げる。

**対象層:** Java 層 / HotSpot C++ 層（ZGC サブシステム）

**対象ランタイム:** Azul Zulu 25 (OpenJDK 25 ベース) -- Generational ZGC がデフォルト

**前提知識:** GC の基礎（gc-deep-dive.md の第1〜6章）

## 第1章: ZGC の設計思想と歴史

### 1.1 ZGC が解決する問題

従来の GC（G1GC を含む）には、ヒープサイズが大きくなるにつれて **Stop-the-World (STW) の停止時間が増加する** という本質的な問題があった。

```
┌─────────────────────────────────────────────────────┐
│             STW 停止時間とヒープサイズの関係           │
│                                                     │
│  停止時間                                            │
│  (ms)                                               │
│   │                                                 │
│  500│                                    ╱ G1GC     │
│     │                                 ╱             │
│  200│                              ╱                │
│     │                           ╱                   │
│  100│                        ╱                      │
│     │                     ╱                         │
│   50│                  ╱                            │
│     │               ╱                               │
│   10│............╱..................................│
│     │         ╱   ──────────────────────── ZGC      │
│    1│──────╱────────────────────────────────────    │
│     │                                               │
│     └───┬───┬───┬───┬───┬───┬───┬── ヒープサイズ    │
│        1G  4G  8G  16G 32G 64G 128G                 │
└─────────────────────────────────────────────────────┘
```

G1GC は「停止時間目標（デフォルト 200ms）」を設定できるが、これは **目標** であって **保証** ではない。ヒープが大きくなると、マーキングやコンパクションの STW フェーズで目標を超える停止が発生する。

ZGC はこの問題に対して、**「ヒープサイズに関係なく停止時間を 1ms 以下に抑える」** という野心的な目標を掲げて設計された。

### 1.2 設計目標

ZGC の設計目標は明確で、すべてが **低レイテンシ** に集約される。

| 設計目標 | 詳細 |
|---|---|
| **停止時間 < 1ms** | ヒープサイズやライブセットサイズに比例しない |
| **数百 MB 〜 数 TB のヒープに対応** | 小規模から超大規模まで |
| **スループットの大幅な低下を避ける** | G1GC 比で 15% 以上の低下は許容しない |
| **将来の GC 最適化の土台** | カラーポインタ等の拡張可能なアーキテクチャ |

ここで重要な点は、ZGC が **「スループット最大化」ではなく「レイテンシ最小化」を最優先する GC** だということである。アプリケーションスレッドと並行して GC 処理を行うため、CPU リソースの一部を GC に割く必要があり、純粋なスループットでは G1GC に劣る場面もある。

### 1.3 歴史: 実験から世代別への進化

ZGC は Oracle の Per Liden を中心に開発され、段階的に進化してきた。

### 1.4 Java 25 時点の位置づけ

Java 25 では、ZGC は以下の状態にある。

- **`-XX:+UseZGC`** で有効化（デフォルト GC は引き続き G1GC）
- **世代別モードが唯一のモード**（非世代別モードは Java 24 で削除済み: JEP 490）
- `-XX:+ZGenerational` / `-XX:-ZGenerational` フラグは Java 24 で削除済み（指定するとエラー）
- すべての主要プラットフォーム（Linux, macOS, Windows）で利用可能
- aarch64, x86_64 の両アーキテクチャに対応

## 第2章: カラーポインタ (Colored Pointers)

### 2.1 従来の GC とポインタの違い

従来の GC（G1GC や CMS 等）は、オブジェクトの GC メタデータを **オブジェクトヘッダ（Mark Word）** に格納していた。ZGC は革新的なアプローチとして、**オブジェクト参照（ポインタ）自体にメタデータを埋め込む**。これがカラーポインタ (Colored Pointers) である。

```
┌───────────────────────────────────────────────────────────┐
│                 メタデータの格納場所の違い                   │
├───────────────────────────────────────────────────────────┤
│                                                           │
│  ◆ G1GC: オブジェクトヘッダに格納                          │
│                                                           │
│    参照 ──→ [Mark Word (GC情報) | Klass Pointer | Fields] │
│              ^^^^^^^^^^^^^^^^                             │
│              ここにGC状態を格納                             │
│                                                           │
│  ◆ ZGC: ポインタ自体に格納                                 │
│                                                           │
│    参照 [メタデータ | アドレス] ──→ [Object Data]          │
│          ^^^^^^^^^^                                       │
│          ポインタ内にGC状態を格納                           │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

この設計の最大の利点は、**オブジェクトにアクセスせずとも参照を見るだけで GC 状態が分かる** ことにある。これにより、GC がオブジェクトをリロケート（移動）している最中でも、アプリケーションスレッドがポインタのメタデータビットを見るだけで適切な処理を判断できる。

### 2.2 64ビットポインタのメタデータビット

ZGC は **64ビットプラットフォーム専用** である。これは、カラーポインタが 64ビットアドレス空間の上位ビットをメタデータとして利用するためである。

```
┌──────────────────────────────────────────────────────────────┐
│           ZGC のカラーポインタ構造（64ビット）                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  63        47  46  45  44  43  42  41        0               │
│  ┌─────────┬───┬───┬───┬───┬───┬───┬─────────┐              │
│  │ 未使用   │ F │ R │M1 │M0 │ L │ X │ アドレス │              │
│  └─────────┴───┴───┴───┴───┴───┴───┴─────────┘              │
│                                                              │
│  F  : Finalizable   -- ファイナライズ可能                     │
│  R  : Remapped      -- リマップ済み（最新のアドレスを指す）     │
│  M1 : Marked1       -- マーキングフェーズ1で到達可能           │
│  M0 : Marked0       -- マーキングフェーズ0で到達可能           │
│  L  : Load Good     -- ロードバリアが不要（良好な参照）        │
│  X  : Store Good    -- ストアバリアが不要                     │
│                                                              │
│  アドレス: 下位 42 ビット → 最大 4TB のアドレス空間            │
│                                                              │
│  ※ 実際のビット配置はプラットフォームと世代で異なる。           │
│    上記は概念的な図であり、正確な配置は zAddress.hpp を参照。   │
└──────────────────────────────────────────────────────────────┘
```

**なぜ 64ビット限定なのか？** 32ビットプラットフォームではアドレス空間が 4GB しかなく、メタデータビットに割く余裕がない。64ビットプラットフォームでは理論上 16EB (エクサバイト) のアドレス空間があり、現在の物理メモリ制約（数 TB）を考慮すると、上位ビットを安全にメタデータに流用できる。

### 2.3 マルチマッピング (Multi-Mapping)

カラーポインタの仕組みを支えるのが **マルチマッピング** である。ZGC は同一の物理メモリページを、異なるメタデータビットパターンに対応する **複数の仮想アドレス** にマップする。

```
┌───────────────────────────────────────────────────────────┐
│                  マルチマッピングの仕組み                    │
├───────────────────────────────────────────────────────────┤
│                                                           │
│  仮想アドレス空間                    物理メモリ             │
│  ─────────────────                  ──────────            │
│                                                           │
│  [Marked0  | 0x1000]  ──┐                                 │
│                          ├───→  物理ページ P              │
│  [Marked1  | 0x1000]  ──┤       (実際のオブジェクトデータ)  │
│                          │                                │
│  [Remapped | 0x1000]  ──┘                                 │
│                                                           │
│  3つの異なる仮想アドレスが、同一の物理ページを指す。          │
│  メタデータビットが違うだけで、データは同じものが見える。      │
│                                                           │
│  ※ mmap の MAP_FIXED + ファイルディスクリプタ共有で実現      │
│    (Linux: /dev/shm 上の tmpfs ファイルを多重マップ)        │
└───────────────────────────────────────────────────────────┘
```

この設計により、GC がメタデータビットを変更しても（例: Marked0 → Remapped）、同じ物理メモリにアクセスできる。**ポインタの「色」を変えるだけで GC 状態を更新でき、オブジェクト自体をコピーする必要がない。**

### 2.4 Marked0 / Marked1 / Remapped ビットの遷移

GC サイクルが進むにつれて、ポインタの「色」は以下のように遷移する。

```
┌──────────────────────────────────────────────────────────────┐
│              カラーポインタの状態遷移                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ◆ GC サイクル N (偶数回目: Marked0 を使用)                   │
│                                                              │
│    Remapped ──→ Marked0    (マーキングフェーズで到達確認)      │
│    Marked0  ──→ Remapped   (リロケーション後に更新)           │
│                                                              │
│  ◆ GC サイクル N+1 (奇数回目: Marked1 を使用)                 │
│                                                              │
│    Remapped ──→ Marked1    (マーキングフェーズで到達確認)      │
│    Marked1  ──→ Remapped   (リロケーション後に更新)           │
│                                                              │
│                                                              │
│  なぜ Marked0 と Marked1 の2つがあるのか？                    │
│  ──────────────────────────────────────────                   │
│  前回の GC サイクルのマーキング情報と、今回のサイクルの         │
│  マーキング情報を区別するため。GC が並行で動作するため、        │
│  前回のサイクルの残り処理と今回のサイクルのマーキングが         │
│  重なる可能性がある。2つのマークビットを交互に使うことで、      │
│  古い情報と新しい情報を安全に区別できる。                      │
│                                                              │
│  遷移の時系列:                                                │
│                                                              │
│  サイクル1:  Remapped → Marked0 → Remapped                   │
│  サイクル2:  Remapped → Marked1 → Remapped                   │
│  サイクル3:  Remapped → Marked0 → Remapped                   │
│  サイクル4:  Remapped → Marked1 → Remapped  (以降繰り返し)    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.5 HotSpot C++ 層の実装

カラーポインタの実装は以下のファイルに集約されている。

| ファイル | 役割 |
|---|---|
| `src/hotspot/share/gc/z/zAddress.hpp` | カラーポインタのビットマスク定義 |
| `src/hotspot/share/gc/z/zAddress.inline.hpp` | ビット操作のインライン関数群 |
| `src/hotspot/share/gc/z/zPointer.hpp` | 型安全なカラーポインタラッパー |

`zAddress.hpp` では `ZPointerMarked0`, `ZPointerMarked1`, `ZPointerRemapped` 等のビットマスク定数が定義されており、ポインタに対するビット演算でメタデータを読み書きする。

## 第3章: ロードバリア (Load Barrier)

### 3.1 Write Barrier ではなく Load Barrier を使う理由

G1GC は **Write Barrier**（書き込みバリア）を使用する。これは参照の **書き込み時** にフックする仕組みで、世代間参照の追跡（Remembered Set の更新）に使われる。

ZGC は代わりに **Load Barrier**（ロードバリア）を使用する。これは参照の **読み込み時** にフックする。

```
┌──────────────────────────────────────────────────────────────┐
│              Write Barrier vs Load Barrier                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ◆ G1GC: Write Barrier (書き込み時にフック)                   │
│                                                              │
│    obj.field = newRef;                                        │
│                 │                                            │
│                 ▼                                            │
│    [Write Barrier] → RSet 更新 / SATB 記録                   │
│                                                              │
│    特徴: 書き込みが少ない場合にオーバーヘッドが小さい           │
│    問題: 古い参照を読んでも気づけない                          │
│                                                              │
│  ──────────────────────────────────────────────────          │
│                                                              │
│  ◆ ZGC: Load Barrier (読み込み時にフック)                     │
│                                                              │
│    Object ref = obj.field;                                    │
│                      │                                       │
│                      ▼                                       │
│    [Load Barrier] → ポインタの色を確認                        │
│         │               │                                    │
│         │  色が正常      │  色が不正（古い）                   │
│         ▼               ▼                                    │
│       そのまま使用    参照を修復 (Self-Healing)                │
│                      → 正しいアドレスに更新                   │
│                      → 元のフィールドも修正                   │
│                                                              │
│    特徴: 読み込みのたびにチェックが入る                        │
│    利点: 古い参照を即座に検出・修復できる                      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**なぜ Load Barrier なのか？** ZGC の最大の特徴は「オブジェクトのリロケーション（移動）を GC とアプリケーションが並行して行う」ことにある。オブジェクトが移動された場合、古いアドレスを指す参照がアプリケーション内に残る可能性がある。Load Barrier により、参照を **使う瞬間** にそれが最新かどうかをチェックし、古ければその場で修正する。これにより、**STW でまとめて参照を書き換える必要がなくなる。**

### 3.2 Load Barrier の動作メカニズム

Load Barrier の判定は極めて高速で、ポインタのメタデータビットを確認するだけで済む。

```
┌──────────────────────────────────────────────────────────────┐
│                 Load Barrier の処理フロー                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│    Object ref = obj.field;   // ヒープからの参照読み込み       │
│         │                                                    │
│         ▼                                                    │
│    ┌─────────────────────────────┐                           │
│    │ メタデータビットの確認        │                           │
│    │ (Load Good ビットをテスト)    │                           │
│    └─────────┬───────────────────┘                           │
│              │                                               │
│       ┌──────┴──────┐                                        │
│       │              │                                       │
│    Good ✓         Bad ✗                                      │
│       │              │                                       │
│       ▼              ▼                                       │
│   そのまま使用   スローパス (Slow Path) に入る                 │
│   (ファストパス)        │                                     │
│                        ▼                                     │
│                 ┌──────────────────────┐                     │
│                 │ 1. 転送テーブル参照    │                     │
│                 │    → 移動先アドレス取得 │                    │
│                 │ 2. 参照を修正          │                     │
│                 │ 3. 元のフィールドも更新 │                     │
│                 │    (Self-Healing)      │                     │
│                 └──────────────────────┘                     │
│                        │                                     │
│                        ▼                                     │
│                  修正済み参照を使用                            │
│                                                              │
│  ※ ファストパスは 1回の AND 演算 + 条件分岐のみ。              │
│    ほとんどの場合ファストパスを通るため、                       │
│    実質的なオーバーヘッドは数ナノ秒。                          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

JIT コンパイラ（C2）は、ヒープからの参照読み込みごとに Load Barrier コードを挿入する。ただし、連続して同じオブジェクトの複数フィールドを読む場合など、JIT の最適化によりバリアチェックが省略されることもある。

### 3.3 Self-Healing（自己修復）参照

ZGC の Load Barrier における最も巧妙な仕組みが **Self-Healing（自己修復）** である。

```
┌──────────────────────────────────────────────────────────────┐
│                  Self-Healing の動作                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ◆ Step 1: GC がオブジェクト A を新しい場所に移動             │
│                                                              │
│    obj.field ──→ [A (旧アドレス 0x1000)]                     │
│                          │                                   │
│                          │ GC がリロケーション                │
│                          ▼                                   │
│                  [A (新アドレス 0x5000)]                      │
│                                                              │
│    転送テーブル: 0x1000 → 0x5000                              │
│                                                              │
│  ◆ Step 2: アプリが古い参照を読む → Load Barrier 発動         │
│                                                              │
│    Object ref = obj.field;   // 0x1000 を読む                │
│         │                                                    │
│         ▼ Load Barrier                                       │
│    「色が古い！ → 転送テーブルから 0x5000 を取得」             │
│         │                                                    │
│         ▼ Self-Healing                                       │
│    obj.field = 0x5000;   // 元のフィールドを修正！             │
│    ref = 0x5000;         // 今回の読み込み結果も修正           │
│                                                              │
│  ◆ Step 3: 次回以降、同じフィールドを読むとファストパス通過    │
│                                                              │
│    Object ref = obj.field;   // 0x5000 (修正済み)            │
│         │                                                    │
│         ▼ Load Barrier                                       │
│    「色が正常 → そのまま通過」 ← ファストパス!                │
│                                                              │
│  ポイント: 一度スローパスを通ると、以降は同じ参照に対して       │
│  ファストパスだけで済む。参照が「自己修復」するため、           │
│  スローパスの頻度は時間とともに急速に減少する。                │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

Self-Healing は ZGC の並行リロケーションを成立させる鍵である。GC がオブジェクトを移動しても、アプリケーションが参照を読むたびに自動的に最新のアドレスに修正されるため、**STW での参照一括書き換えが不要** になる。

### 3.4 G1GC の Write Barrier との比較

| 比較項目 | G1GC Write Barrier | ZGC Load Barrier |
|---|---|---|
| **フックタイミング** | 参照の書き込み時 | 参照の読み込み時 |
| **主な目的** | RSet 更新、SATB バッファ記録 | ポインタの色チェック、Self-Healing |
| **オーバーヘッドの発生箇所** | 書き込みが多いワークロード | 読み込みが多いワークロード |
| **ファストパスのコスト** | 条件分岐 + カードテーブル更新 | AND 演算 + 条件分岐 |
| **並行リロケーション** | 不可（STW で実施） | 可能（Load Barrier で Self-Healing） |
| **JIT 最適化** | バリア省略は限定的 | 連続アクセス時にバリア省略可能 |

### 3.5 HotSpot C++ 層の実装

| ファイル | 役割 |
|---|---|
| `src/hotspot/share/gc/z/zBarrier.hpp` | Load Barrier のインターフェース定義 |
| `src/hotspot/share/gc/z/zBarrier.inline.hpp` | ファストパス / スローパスの実装 |
| `src/hotspot/share/gc/z/zBarrierSet.hpp` | BarrierSet フレームワークとの接続 |
| `src/hotspot/cpu/x86/gc/z/zBarrierSetAssembler_x86.cpp` | x86 向け JIT バリアコード生成 |
| `src/hotspot/cpu/aarch64/gc/z/zBarrierSetAssembler_aarch64.cpp` | AArch64 向け JIT バリアコード生成 |

バリアの実装は、C2 JIT コンパイラとの統合が重要である。`c2_MacroAssembler` を通じて、Load Barrier がマシンコードレベルで効率的に挿入される。

## 第4章: ZGC のフェーズ

### 4.1 全体像: 3つの並行フェーズ

ZGC の GC サイクルは、**ほぼすべてがアプリケーションスレッドと並行して実行される**。STW フェーズは存在するが、その時間は極めて短い（通常 1ms 以下）。

```
┌──────────────────────────────────────────────────────────────┐
│                ZGC サイクルの全体像                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  時間 ──────────────────────────────────────────→            │
│                                                              │
│  アプリ  ███████░███████████████░███████████████░███████████  │
│  スレッド       │               │               │            │
│                 │               │               │            │
│  GC      ░░░░░░░│▓▓▓▓▓▓▓▓▓▓▓▓▓│▓▓▓▓▓▓▓▓▓▓▓▓▓│▓▓▓▓▓▓▓░░  │
│  スレッド       │               │               │            │
│                 ▼               ▼               ▼            │
│          Pause Mark     Pause Mark      Pause Relocate       │
│           Start           End             Start              │
│          (<1ms)          (<1ms)           (<1ms)             │
│                                                              │
│          │← Concurrent →│← Concurrent  →│← Concurrent →│   │
│             Mark           Process Refs    Relocate          │
│                            + Relocate                        │
│                                                              │
│  ███ = アプリケーション実行                                   │
│  ▓▓▓ = GC 並行処理 (アプリも同時に動いている)                 │
│  ░░░ = STW ポーズ (アプリ停止、ごく短い)                      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 Concurrent Mark（並行マーキング）

並行マーキングフェーズでは、GC Roots から到達可能な全オブジェクトを特定する。

```
┌──────────────────────────────────────────────────────────────┐
│              Concurrent Mark の詳細                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  (1) Pause Mark Start  [STW < 1ms]                           │
│      ────────────────────────────────                        │
│      ・GC Roots のスキャン開始点を設定                         │
│      ・TLAB をフラッシュ                                      │
│      ・マーキングビットを切り替え (Marked0 ↔ Marked1)         │
│                                                              │
│  (2) Concurrent Mark  [アプリと並行]                          │
│      ────────────────────────────────                        │
│      ・GC Roots からオブジェクトグラフをトラバース             │
│      ・到達可能なオブジェクトのポインタに Mark ビットを設定     │
│      ・Load Barrier がアプリの参照読み込みも同時にマーク       │
│        (SATB: Snapshot-At-The-Beginning に類似した仕組み)     │
│                                                              │
│      GC Roots                                                │
│        │                                                     │
│        ▼                                                     │
│      [A:Marked] ──→ [B:Marked] ──→ [C:Marked]               │
│        │                                                     │
│        └──→ [D:Marked]                                       │
│                                                              │
│      [E:Unmarked]  ← 到達不能 → 回収対象                     │
│      [F:Unmarked]  ← 到達不能 → 回収対象                     │
│                                                              │
│  (3) Pause Mark End  [STW < 1ms]                             │
│      ────────────────────────────────                        │
│      ・マーキング完了の同期                                   │
│      ・弱参照 (WeakReference) の処理をスケジュール             │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 4.3 Concurrent Relocate（並行リロケーション）

並行リロケーションは ZGC の最も革新的な部分である。**オブジェクトの物理的な移動をアプリケーションと並行して行う。**

```
┌──────────────────────────────────────────────────────────────┐
│              Concurrent Relocate の詳細                       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  (1) Relocation Set の選定                                    │
│      ──────────────────────                                  │
│      ・ゴミの割合が高い ZPage を「リロケーション対象」に選定    │
│      ・コスト対効果で対象を決定（ゴミが少ないページは残す）     │
│                                                              │
│     Page A (ゴミ率 80%) ← 対象!                               │
│     ┌─────────────────────────────────────┐                  │
│     │ [Live] [Dead] [Dead] [Dead] [Live]  │                  │
│     └─────────────────────────────────────┘                  │
│                                                              │
│     Page B (ゴミ率 10%) ← 対象外                              │
│     ┌─────────────────────────────────────┐                  │
│     │ [Live] [Live] [Live] [Live] [Dead]  │                  │
│     └─────────────────────────────────────┘                  │
│                                                              │
│  (2) Pause Relocate Start  [STW < 1ms]                       │
│      ────────────────────────────────                        │
│      ・GC Roots が指す参照で、リロケーション対象のものを移動    │
│      ・GC Roots の数は限られるため、ごく短時間で完了            │
│                                                              │
│  (3) Concurrent Relocate  [アプリと並行]                      │
│      ────────────────────────────────                        │
│      ・リロケーション対象のページから生存オブジェクトを移動      │
│      ・転送テーブル (Forwarding Table) に旧→新のマッピングを記録│
│      ・アプリが移動中のオブジェクトにアクセスすると             │
│        Load Barrier が Self-Healing を実行                    │
│                                                              │
│     移動の様子:                                               │
│                                                              │
│     旧 Page                         新 Page                  │
│     ┌──────────────────┐           ┌──────────────────┐     │
│     │ [Obj1]  [    ]   │    ──→    │ [Obj1'] [Obj2']  │     │
│     │ [    ]  [Obj2]   │           │                  │     │
│     └──────────────────┘           └──────────────────┘     │
│              │                              ▲                │
│              └── 転送テーブル ──→ ──────────┘                │
│                  Obj1 @ 旧 → Obj1' @ 新                      │
│                  Obj2 @ 旧 → Obj2' @ 新                      │
│                                                              │
│  (4) ページ解放                                               │
│      ──────────                                              │
│      ・全生存オブジェクトの移動完了後、旧ページを解放           │
│      ・物理メモリを OS に返却可能                              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 4.4 Concurrent Remap（並行再マッピング）

リロケーション後、ヒープ内に残っている旧アドレスへの参照を新アドレスに更新する必要がある。これが Concurrent Remap である。

```
┌──────────────────────────────────────────────────────────────┐
│              Concurrent Remap の詳細                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ポイント: Remap は次の GC サイクルの Mark と統合される         │
│  ──────────────────────────────────────────────────          │
│                                                              │
│  サイクル N:                                                  │
│    Mark → Relocate → (Remap は即座には完了しない)             │
│                                                              │
│  サイクル N+1:                                                │
│    Mark (この過程で前サイクルの Remap も同時に実行)             │
│     │                                                        │
│     ├─ オブジェクトグラフのトラバース中に                      │
│     │  旧アドレスの参照を発見                                  │
│     │     │                                                  │
│     │     ▼                                                  │
│     │  転送テーブルを参照して新アドレスに更新                   │
│     │  (Mark と Remap を同時に処理)                            │
│     │                                                        │
│     └─ 結果: Mark 完了時点で Remap も完了                     │
│                                                              │
│  この統合により、独立した Remap フェーズが不要になり、          │
│  GC のオーバーヘッドが大幅に削減される。                       │
│                                                              │
│  また、アプリケーションが Remap 完了前に旧参照を読んでも、     │
│  Load Barrier の Self-Healing により即座に修正される。         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 4.5 STW フェーズ（マイクロポーズ）

ZGC の STW フェーズは3つあり、いずれも **O(1)** の処理時間で完了する。つまり、ヒープサイズやオブジェクト数に比例しない。

| STW フェーズ | 処理内容 | 所要時間 |
|---|---|---|
| **Pause Mark Start** | マーキング開始の同期、GC Roots の初期スキャン準備 | 通常 < 200μs |
| **Pause Mark End** | マーキング完了の同期、弱参照処理のスケジュール | 通常 < 200μs |
| **Pause Relocate Start** | GC Roots の参照をリロケーション | 通常 < 200μs |

**なぜ O(1) なのか？**

- **Pause Mark Start/End**: 各スレッドのローカル状態を同期するだけで、ヒープ全体のスキャンは行わない
- **Pause Relocate Start**: GC Roots（スレッドスタック上の参照等）のみを処理。ヒープ内の参照は並行フェーズで処理する

JEP 376 (Java 16) 以降、スレッドスタックの処理すら並行化されているため、STW での処理量はさらに削減されている。

## 第5章: Generational ZGC (JEP 439)

### 5.1 なぜ世代別が必要になったか

初期の ZGC（非世代別）はヒープ全体を単一世代として扱っていた。これはシンプルで実装が容易だったが、**世代別仮説を活用できない** という根本的な非効率があった。

```
┌──────────────────────────────────────────────────────────────┐
│          非世代別 ZGC の問題点                                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ◆ 問題1: 短命オブジェクトにも完全な GC サイクルが必要         │
│                                                              │
│  非世代別 ZGC:                                                │
│  ┌────────────────────────────────────────┐                  │
│  │ ヒープ全体                              │                  │
│  │ [短命][短命][長命][短命][短命][長命]...  │                  │
│  │  ↑                                     │                  │
│  │  この1個を回収するにも                   │                  │
│  │  ヒープ全体を Mark する必要がある         │                  │
│  └────────────────────────────────────────┘                  │
│                                                              │
│  ◆ 問題2: GC の頻度を上げにくい                               │
│                                                              │
│  大きなヒープ全体の Mark は時間がかかるため、                   │
│  GC サイクルの頻度を上げると CPU を圧迫する。                  │
│  結果的に短命オブジェクトがメモリに滞留し、                    │
│  ヒープ使用率が高止まりする。                                  │
│                                                              │
│  ◆ 問題3: 割り当てレートが高いワークロードで不利               │
│                                                              │
│  短命オブジェクトを大量に生成するワークロードでは、             │
│  GC が追いつかず Allocation Stall（割り当て停止）が発生する    │
│  可能性がある。                                               │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

Generational ZGC はこれらの問題を解決するため、**若い世代 (Young) と古い世代 (Old) を分離して独立した GC サイクルを実行する** 仕組みを導入した。

### 5.2 Young Generation と Old Generation

```
┌──────────────────────────────────────────────────────────────┐
│          Generational ZGC のヒープレイアウト                   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌───────────────────────────────────────────────────┐       │
│  │                    ヒープ全体                      │       │
│  │                                                   │       │
│  │  ┌─────────────────────┐ ┌─────────────────────┐ │       │
│  │  │   Young Generation   │ │   Old Generation    │ │       │
│  │  │                     │ │                     │ │       │
│  │  │  [ZPage] [ZPage]   │ │  [ZPage] [ZPage]   │ │       │
│  │  │  [ZPage] [ZPage]   │ │  [ZPage]           │ │       │
│  │  │  [ZPage]           │ │  [ZPage] [ZPage]   │ │       │
│  │  │                     │ │                     │ │       │
│  │  │  ・新規割り当て先    │ │  ・昇格オブジェクト  │ │       │
│  │  │  ・高頻度で GC      │ │  ・低頻度で GC      │ │       │
│  │  │  ・短いサイクル      │ │  ・長いサイクル      │ │       │
│  │  └─────────────────────┘ └─────────────────────┘ │       │
│  │                                                   │       │
│  │  Young と Old の境界は動的に調整される               │       │
│  │  (G1GC のリージョンと同様の柔軟性)                   │       │
│  └───────────────────────────────────────────────────┘       │
│                                                              │
│  ◆ Young GC サイクル:                                        │
│    ・頻繁に実行（短命オブジェクトを素早く回収）                │
│    ・Young ZPage のみをスキャン → 高速                       │
│    ・生存オブジェクトは年齢に応じて Old に昇格                 │
│                                                              │
│  ◆ Old GC サイクル:                                          │
│    ・比較的低頻度で実行                                       │
│    ・Old ZPage をスキャン                                    │
│    ・Young GC と並行して実行可能                              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

Generational ZGC では、**Young GC と Old GC が独立したサイクルで動作** する。これにより:

- **短命オブジェクト**: Young 世代の高頻度 GC で即座に回収
- **長命オブジェクト**: Old 世代の低頻度 GC で効率的に管理
- **全体のスループット**: ヒープ全体をスキャンする頻度が大幅に減少

### 5.3 世代間バリア

Generational ZGC では、Load Barrier に加えて **Store Barrier**（書き込みバリア）も導入されている。これは Old 世代のオブジェクトから Young 世代のオブジェクトへの参照（世代間参照）を追跡するためである。

```
┌──────────────────────────────────────────────────────────────┐
│          Generational ZGC のバリア構成                        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ◆ Load Barrier (従来通り)                                   │
│    ・参照の読み込み時に色チェック                              │
│    ・Self-Healing                                            │
│                                                              │
│  ◆ Store Barrier (Generational で追加)                       │
│    ・参照の書き込み時にフック                                  │
│    ・Old → Young の参照を Remember Set に記録                 │
│                                                              │
│    Old Generation          Young Generation                  │
│    ┌──────────────┐       ┌──────────────┐                  │
│    │  OldObj       │       │  YoungObj    │                  │
│    │  .field ──────┼──→────┼─→ [data]    │                  │
│    │               │  ↑    │              │                  │
│    └──────────────┘  │    └──────────────┘                  │
│                       │                                      │
│              Store Barrier が検知し                           │
│              Remember Set に記録                              │
│                                                              │
│    → Young GC 時、Remember Set の参照も GC Roots として扱う   │
│    → Old 全体をスキャンせずに Young のみを安全に回収できる     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

非世代別 ZGC では Load Barrier のみだったが、Generational ZGC では Store Barrier も加わるため、バリアの構成がやや複雑になっている。ただし、JIT コンパイラによる最適化により、実行時オーバーヘッドは最小限に抑えられている。

### 5.4 Non-generational ZGC との違い

| 比較項目 | Non-generational ZGC | Generational ZGC |
|---|---|---|
| **ヒープの分割** | 単一世代 | Young + Old |
| **GC サイクル** | ヒープ全体を1サイクルで処理 | Young / Old で独立したサイクル |
| **バリア** | Load Barrier のみ | Load Barrier + Store Barrier |
| **短命オブジェクトの回収** | サイクル全体が完了するまで待つ | Young GC で即座に回収 |
| **割り当てレート耐性** | 低め（全体 Mark が必要） | 高い（Young のみを高速 Mark） |
| **メモリ効率** | 短命オブジェクトが滞留しがち | 世代別仮説の恩恵で効率的 |
| **Java バージョン** | Java 15〜23（Java 23 で非推奨、Java 24 で削除: JEP 490） | Java 21〜（Java 24 以降は唯一のモード） |
| **有効化フラグ** | `-XX:+UseZGC` (旧) | `-XX:+UseZGC` (Java 25) |

### 5.5 Java 25 でのデフォルト動作

Java 25 (Azul Zulu 25) では:

- `-XX:+UseZGC` を指定すると、**自動的に Generational ZGC** が有効になる
- 非世代別モードは **Java 24 で削除済み**（JEP 490。`-XX:-ZGenerational` フラグも削除）
- デフォルト GC は引き続き **G1GC** である（`-XX:+UseZGC` の明示指定が必要）

```bash
# Java 25 での ZGC 有効化（世代別が自動的に有効）
java -XX:+UseZGC -Xmx4g MyApp.java

# デフォルトの GC を確認
java -XX:+PrintFlagsFinal -version 2>&1 | grep UseZGC
# bool UseZGC = false
```

## 第7章: ZGC のパフォーマンス特性

### 7.1 最適なワークロード

ZGC が最も威力を発揮するのは以下のようなワークロードである。

| ワークロード | ZGC が有利な理由 |
|---|---|
| **レイテンシ重視の Web API** | p99 レイテンシを数ms 以下に維持できる |
| **金融系リアルタイム処理** | GC ポーズによる取引遅延を排除 |
| **大容量ヒープ（16GB 超）** | ヒープサイズに比例しない停止時間 |
| **インタラクティブ UI** | ユーザ体験を損なう GC ポーズがない |
| **ストリーム処理** | 一定のレイテンシで連続処理が可能 |

### 7.2 G1GC との比較

### 7.4 不向きなケース

ZGC が **不向き** なケースも理解しておくことが重要である。

| ケース | 理由 |
|---|---|
| **ヒープが非常に小さい (< 256MB)** | ZGC のメタデータオーバーヘッドが相対的に大きい |
| **CPU コア数が少ない (1-2コア)** | 並行 GC スレッドがアプリスレッドと CPU を奪い合う |
| **スループット最優先のバッチ処理** | 並行 GC のオーバーヘッドにより、G1GC より遅くなる可能性 |
| **32ビット JVM** | カラーポインタが 64ビットポインタに依存しているため、32ビットでは使用不可 |
| **組み込み・リソース制約環境** | メモリと CPU の両方にオーバーヘッドがある |

## 第8章: ZGC のチューニング

### 8.1 主要パラメータ一覧

ZGC は **「ほとんどチューニング不要」** を設計思想としている。必要なパラメータは限られている。

| パラメータ | デフォルト | 説明 |
|---|---|---|
| `-XX:+UseZGC` | false | ZGC を有効化 |
| `-Xmx` | -- | 最大ヒープサイズ（必須） |
| `-Xms` | -- | 初期ヒープサイズ |
| `-XX:SoftMaxHeapSize` | `-Xmx` と同じ | ソフトなヒープ上限（後述） |
| `-XX:ZCollectionInterval` | 0 (無効) | GC の最小実行間隔（秒） |
| `-XX:ZAllocationSpikeTolerance` | 2.0 | 割り当てスパイクへの耐性係数 |
| `-XX:ZUncommitDelay` | 300 | 未使用メモリの OS 返却までの遅延（秒） |
| `-XX:ZFragmentationLimit` | 25 | フラグメンテーション許容率（%） |
| `-XX:ConcGCThreads` | 自動 | 並行 GC スレッド数 |
| `-XX:ParallelGCThreads` | 自動 | STW フェーズのスレッド数 |

### 8.2 基本的な有効化

```bash
# 最もシンプルな ZGC 有効化
java -XX:+UseZGC -Xmx4g MyApp.java

# ヒープサイズを明示的に設定
java -XX:+UseZGC -Xms2g -Xmx8g MyApp.java

# GC ログ付き
java -XX:+UseZGC -Xmx4g -Xlog:gc*:file=gc.log:time MyApp.java
```

ZGC のチューニングで最も重要なのは **適切なヒープサイズの設定** である。ZGC は並行で動作するため、GC 中にもオブジェクトが割り当てられる。ヒープが小さすぎると、GC が完了する前にメモリが枯渇する恐れがある。

**目安**: ライブセット（常時生存しているオブジェクトの総量）の **3倍以上** のヒープを確保する。

### 8.3 SoftMaxHeapSize の活用

`-XX:SoftMaxHeapSize` は ZGC 特有の非常に有用なパラメータである。

```
│                                                              │
│  ・通常時: SoftMaxHeapSize (4GB) 以下に収まるよう GC が努力    │
│  ・スパイク時: -Xmx (8GB) まで一時的に拡大を許容               │
│  ・スパイク後: SoftMaxHeapSize まで戻るよう GC がメモリを回収   │
│                                                              │
│  用途:                                                       │
│  ・コンテナ環境でメモリ使用量を柔軟に制御                      │
│  ・通常時はメモリを節約し、ピーク時には OOM を回避              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

```bash
# SoftMaxHeapSize の使用例
java -XX:+UseZGC -Xmx8g -XX:SoftMaxHeapSize=4g MyApp.java
```

**ログの読み方のポイント:**

| ログ項目 | 意味 | 注目すべき点 |
|---|---|---|
| `Pause Mark Start` | STW ポーズ時間 | **1ms を超えていないか** |
| `Pause Mark End` | STW ポーズ時間 | **1ms を超えていないか** |
| `Pause Relocate Start` | STW ポーズ時間 | **1ms を超えていないか** |
| `Concurrent Mark` | 並行マーキング時間 | 長すぎる場合は GC スレッド不足 |
| `Concurrent Relocate` | 並行リロケーション時間 | ゴミが多いほど長くなる |

| `Free` / `Used` | ヒープ使用状況 | Free が少なすぎないか |
| `Garbage Collection (Proactive)` | GC のトリガー理由 | Allocation Stall があれば問題 |

**GC トリガー理由:**

| トリガー | 意味 |
|---|---|
| `Proactive` | ZGC が予防的に実行（正常） |
| `Warmup` | JVM 起動直後の初期 GC |
| `Allocation Rate` | 割り当てレートに基づくトリガー |
| `Allocation Stall` | **メモリ不足で割り当てが停止**（要対処） |
| `Timer` | `-XX:ZCollectionInterval` による定期実行 |
| `System.gc()` | `System.gc()` 呼び出し |

**`Allocation Stall` は最も注意すべきログ** である。これが頻発する場合、ヒープサイズが不足しているか、GC サイクルが割り当てレートに追いついていない。対策:
1. ヒープサイズ (`-Xmx`) を増やす
2. `-XX:ConcGCThreads` を増やす（CPU に余裕がある場合）
3. 割り当てレートを下げる（アプリ側の最適化）

## 第10章: G1GC との比較まとめ

最後に、G1GC と ZGC の包括的な比較表を示す。

| 比較項目 | G1GC | ZGC (Generational) |
|---|---|---|
| **導入バージョン** | Java 7 (実験), Java 9 (デフォルト) | Java 11 (実験), Java 15 (正式), Java 21 (世代別) |
| **デフォルト GC** | Yes (Java 9+) | No |
| **設計目標** | 停止時間と高スループットのバランス | 超低レイテンシ (< 1ms) |
| **最大停止時間** | 10ms 〜 数秒 | < 1ms |
| **スループット** | 高い (~95%) | やや低い (~85-95%) |
| **ヒープ構造** | リージョン (固定サイズ, 1-32MB) | ZPage (3サイズクラス) |
| **世代別** | あり (Eden/Survivor/Old) | あり (Young/Old) |
| **コンパクション** | STW (Evacuation) | 並行 (Concurrent Relocate) |
| **バリア方式** | Write Barrier (SATB + RSet) | Load Barrier + Store Barrier |
| **ポインタ** | 通常の 64bit ポインタ | カラーポインタ (メタデータ埋め込み) |
| **32bit JVM** | 対応 | 非対応 |
| **メモリオーバーヘッド** | ~10-15% | ~15-20% |
| **NUMA 対応** | あり | あり（より積極的） |
| **推奨最小ヒープ** | 数十 MB | 数百 MB |
| **推奨最大ヒープ** | ~64GB | 数 TB |
| **推奨 CPU コア** | 2+ | 4+ |
| **Compressed Oops** | 対応 | 非対応（カラーポインタと非互換） |
| **OS メモリ返却** | あり (-XX:G1PeriodicGCInterval) | あり (ZUncommitDelay, デフォルト有効) |
| **チューニング難易度** | 中（パラメータが多い） | 低（ほぼデフォルトで動作） |
| **主要 JVM フラグ** | `-XX:+UseG1GC` (デフォルト) | `-XX:+UseZGC` |

**選定の指針:**


### Zulu との差分

Azul Zulu (OSS 版) は OpenJDK mainline のビルドであり、ZGC の実装は HotSpot そのものである。Zulu 25 で `-XX:+UseZGC` を指定した場合、OpenJDK 25 と完全に同一の Generational ZGC が動作する。

ただし、Azul の商用製品 **Azul Platform Prime（旧 Zing）** の **C4 GC** は ZGC とは全く異なる独自実装であり、以下の点で異なる:

| 比較項目 | ZGC | Azul C4 GC |
|---|---|---|
| **リードバリア** | カラーポインタ + Load Barrier | Loaded Value Barrier (LVB) |
| **仮想メモリ技法** | マルチマッピング | Virtual Memory-based compaction |
| **世代別** | あり (Java 25) | あり（当初から） |
| **最大停止時間** | < 1ms | < 1ms |
| **ライセンス** | OSS (GPLv2) | 商用（有償） |
| **JVM** | HotSpot | Zing (Falcon JIT) |
